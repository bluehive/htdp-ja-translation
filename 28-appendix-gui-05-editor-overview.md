## 5 エディタ

エディタ機構ツールボックスは、次の二種類のよくあるアプリケーションのための基盤を提供します。

- **洗練されたテキスト・エディタを必要とするプログラム** — 単純なテキストフィールド・コントロールでは、テキスト中心のアプリケーションには不十分です。多くのプログラムは、複数のフォントや非テキスト項目を扱えるエディタを必要とします。
- **ドラッグ可能なオブジェクトを持つキャンバスを必要とするプログラム** — 描画ツールボックスは、線や箱を描くための汎用描画面を提供しますが、多くのアプリケーションは、ユーザが個々のオブジェクトをドラッグしたりリサイズしたりできる対話的キャンバスを必要とします。

どちらの種類のアプリケーションも、テキスト、画像、プログラマ定義の項目、さらには埋め込みエディタまで扱える拡張可能なエディタを必要とします。両者の違いは項目のレイアウトにあります。したがってエディタ機構ツールボックスは、二つのクラス経由で二種類のエディタを提供します。

- `text%` — テキスト・エディタでは、項目は段落フローの中で自動的に位置決めされます。
- `pasteboard%` — ペーストボード・エディタでは、項目は明示的に位置決めされ、ドラッグ可能です。

このエディタ・アーキテクチャは、切り取りと貼り付け、拡張可能なファイル形式、層状のテキスト・スタイルを含む、エディタに関する現実世界の課題の全範囲に対応しつつ、高い拡張性を支えます。その結果としてシステムはかなり複雑になり、エディタ・クラスを効果的に使うには、エディタ機構ツールボックスの構造と用語をしっかり理解する必要があります。それでも、上の記述の一方（または両方）に当てはまるアプリケーションは十分多いので、ツールボックスの深さと複雑さ、およびそれを使うために必要な学習投資を正当化できます。

短い例が、エディタの動きを示します。まず、エディタには内容を表示するための `editor-canvas%` が必要です。次にテキスト・エディタを作り、キャンバスにインストールします。

```racket
(define f (new frame% [label "Simple Edit"]
                      [width 200]
                      [height 200]))
(define c (new editor-canvas% [parent f]))
(define t (new text%))
(send c set-editor t)
(send f show #t)
```

この時点でエディタは完全に機能します。ユーザはエディタにテキストを入力できますが、切り取りと貼り付けや取り消し操作は使えません。メニューバー経由で、エディタに対する標準操作をすべてサポートできます。

```racket
(define mb (new menu-bar% [parent f]))
(define m-edit (new menu% [label "Edit"] [parent mb]))
(define m-font (new menu% [label "Font"] [parent mb]))
(append-editor-operation-menu-items m-edit #f)
(append-editor-font-menu-items m-font)
(send t set-max-undo-history 100)
```

これで標準の切り取りと貼り付け操作と取り消しが動き、ユーザはフォント・スタイルまで設定できます。エディタは取り消し履歴スタックなしで作られるため、`set-max-undo-history` で非ゼロのスタックを設定し、取り消し操作を記録できるようにします。ユーザは Edit メニューから Insert Text を選んで埋め込みエディタを挿入することもできます。メニュー項目を選ぶと、エディタ内にボックスが現れ、キャレットがその中にあります。キャレットがボックス内にある状態で打つと、テキストが増えるにつれてボックスが伸び、フォント操作はキャレットがアクティブな場所に適用されます。ボックスの外側のテキストは、ボックスのサイズが変わるにつれて再配置されます。ボックス自体をコピーして貼り付けられることにも注意してください。

エディタの内容は**スニップ（snip）** から成ります。埋め込みエディタは、埋め込み側のエディタから見ると単一のスニップです。即時テキストを符号化するために、スニップは単一の文字でもよいですが、より多くの場合、スニップは同じ行上の隣接する文字の列です。`find-snip` メソッドはテキスト・エディタからスニップを取り出します。

```racket
(send t find-snip 0 'after)
```

上の式はエディタ内の最初のスニップを返します。それは文字列スニップ（即時テキスト用）か、エディタ・スニップ（埋め込みエディタ用）かもしれません。

エディタは特定の表示に永続的に結び付いていません。キャンバスからテキスト・エディタを取り出し、代わりにペーストボード・エディタをキャンバスに置けます。

```racket
(define pb (new pasteboard%))
(send c set-editor pb)
```

ペーストボード・エディタをインストールすると、ユーザはもはやエディタに文字を直接入力できません（ペーストボードは直接入力テキストをサポートしないため）。ただし、ユーザは他の場所からテキストを切り取ってペーストボードに貼り付けたり、Edit メニューの Insert メニュー項目の一つを選んだりできます。スニップは（テキスト・エディタと異なり）ペーストボード・エディタでははっきり識別できます。各スニップが個別にドラッグ可能だからです。

キャンバスから最近取り除いた古いテキスト・エディタを、エディタ・スニップを明示的に作ることでペーストボード内の埋め込みエディタとして挿入できます。

```racket
(define s (make-object editor-snip% t)); t is the old text editor
(send pb insert s)
```

個々のスニップを同時に異なるエディタへ挿入したり、同じエディタに複数回挿入したりはできません。

```racket
(send pb insert s); no effect
```

ただし、スニップの深いコピーを作り、そのコピーをペーストボードに挿入することはできます。

```racket
(send pb insert (send s copy))
```

エディタ・クラスを使うアプリケーションは通常、`text%` と `pasteboard%` クラスの新しい版を派生します。たとえば、末尾への挿入だけを許し削除を決して許さない追記専用エディタを実装するには、`text%` から新しいクラスを派生し、`can-insert?` と `can-delete?` メソッドをオーバーライドします。

```racket
(define append-only-text%
  (class text%
    (inherit last-position)
    (define/augment (can-insert? s l) (= s (last-position)))
    (define/augment (can-delete? s l) #f)
    (super-new)))
```

### 5.1 エディタの構造と用語

エディタ機構ツールボックスは、エディタ・アセンブリを三つの機能部分に分解することで、拡張可能で入れ子可能なエディタを支えます。

- **エディタ自体** は、テキストまたはペーストボードの状態を格納し、ほとんどのイベントと編集操作を扱います。`editor<%>` インタフェースがコア・エディタ機能を定義しますが、エディタは `text%` または `pasteboard%` のインスタンスとして作られます。
- **スニップ** は、エディタ内の情報の断片です。各スニップは、文字列、絵、または対話的オブジェクト（埋め込みエディタなど）を含められます。テキスト・エディタでは、スニップは一行に収まるよう制約され、一般に単一の型のデータを含みます。`snip%` クラスは基本スニップを実装し、新しいスニップを実装するための基底となります。「新しいスニップの実装」を参照してください。他のスニップ・クラスには、テキスト管理用の `string-snip%`、絵管理用の `image-snip%`、埋め込みエディタ管理用の `editor-snip%` があります。
- **ディスプレイ（display）** は、画面上にエディタを提示します。ディスプレイは、ユーザがエディタをスクロールしたりエディタを切り替えたりできるようにします。ほとんどのディスプレイは `editor-canvas%` クラスのインスタンスですが、`editor-snip%` クラスも埋め込みエディタ用のディスプレイとして働きます。

これら三つの部分は、単純なワープロで説明できます。エディタはテキスト文書に対応します。エディタ・オブジェクトは、テキスト編集のためのキーボードとマウスのコマンドを受け取ります。テキスト自体はスニップのあいだに分散されます。各文字が別々のスニップでもよく、一行上の複数文字を一つのスニップにまとめてもかまいません。ディスプレイは、おおむねテキストが表示されるウィンドウに対応します。エディタがテキストをウィンドウへ表示するときの配置を管理する一方、ディスプレイはどのウィンドウへ描くか、エディタのどの部分を表示するかを決めます。

エディタ内の選択可能な各実体は**項目（item）** です。ペーストボードでは、すべての選択とドラッグ操作がスニップに対して働くため、スニップと項目のあいだには一対一の対応があります。テキスト・エディタでは、一つのスニップが一つ以上の連続する項目を含み、すべての項目はどれかのスニップに属します。たとえば単純なテキスト・エディタでは、各文字が項目ですが、隣接する複数の文字が単一のスニップにまとめられることがあります。スニップ内の項目数は、スニップの**カウント（count）** です。

テキスト・エディタで挿入点が現れ得る各場所は**位置（position）** です。*n* 個の項目を持つテキスト・エディタは *n*+1 個の位置を持ちます。各項目の前に一つ、最後の項目のあとに一つです。

ペーストボード内のスニップの順序は、各スニップの描画平面を決めます。二つのスニップがペーストボード内で重なるとき、順序で前にあるスニップがもう一方のスニップの前にあります（すなわち前者は後者のあとに描かれ、前者のスニップが後者のスニップの一部を覆うことがあります）。

エディタがディスプレイへ描かれるとき、各スニップと位置は**場所（location）** を持ちます。位置またはスニップの場所は、エディタの左上隅からの相対座標で指定されます。エディタ内の場所は、エディタが表示されているときだけ意味を持ちます。

#### 5.1.1 文字と書記素（Graphemes）

テキストを持つエディタでは、項目は Racket の**文字**に対応します。ただし、ユーザが一つの文字として知覚するものの中には、複数の Racket 文字から成るものがあります。たとえば海賊旗の絵文字（四文字の符号化を使う）や、「e」とアクセント修飾子の組み合わせ（単一文字表現もありますが、それら二つの文字として表されることもあります）です。**書記素（grapheme）** は、Unicode の書記素クラスタ仕様が定義する、ユーザが知覚する文字への近似です。Racket は `string-grapheme-count` や `char-grapheme-step` のような関数を通じて書記素をサポートします。

テキスト・エディタで書記素を扱うには、特別な注意が必要です。`text%` の `grapheme-position` や `position-grapheme` のようなメソッドは項目と書記素のインデックスを変換しますが、ほとんどの操作は項目位置に基づき、一般に書記素クラスタ列を保存するようには制約されていません。単一のスニップ内にある書記素クラスタは単一の書記素として描画されますが、スニップ境界をまたぐ書記素クラスタは二つの部分的な書記素として描画されます。`text%` の `insert` メソッドは、既存の内容と挿入内容にまたがる書記素列を検出し、列をまとめて保つための省略可能な引数を取ります。

既定で書記素に敏感な `text%` メソッドは少数です。`delete`、文字の `insert`、および `move-position` です。

#### 5.1.2 管理者（Administrators）

ディスプレイ–エディタおよびエディタ–スニップの接続を管理する、二つの追加の管理層があります。エディタはディスプレイと直接通信することはなく、常に **エディタ管理者（editor administrator）**——`editor-admin%` クラスのインスタンス——と通信し、それが情報をディスプレイへ中継します。同様に、スニップは **スニップ管理者（snip administrator）**——`snip-admin%` クラスのインスタンス——と通信します。

管理層があることで、エディタ・アセンブリのすべての部分が複数部分の機能を持つことを強制せずに、エディタ階層を柔軟にできます。たとえばテキスト・エディタは、別のエディタ内の単一の項目になれます。管理者がなければ、`text%` クラスは（包含側エディタのための）ディスプレイの機能と（埋め込み側エディタのための）スニップの機能をすべて含まなければなりません。管理者を使うことで、エディタ・クラスはディスプレイとスニップの機能を直接実装せずに、包含側と埋め込み側の両方のエディタとして働けます。

スニップは、単一の管理者経由で高々一つのエディタに属します。エディタも、一度に一つの管理者しか持ちません。ただし、エディタを標準ディスプレイ（すなわちエディタ・キャンバス）に接続する管理者は、他のそのような管理者と協調できます。特に、`editor-canvas%` の管理者（それぞれが自身の管理者を持つ）は他の `editor-canvas%` の管理者と協調でき、エディタを同時に複数の `editor-canvas%` ウィンドウに表示できます。

エディタが複数のキャンバスに表示されているとき、キャンバスの管理者の一つがエディタの主管理者として使われます。他のキャンバスのユーザ・イベントと更新イベントを扱うため、エディタの管理者は一時的に変更され、その後エディタの `set-admin` メソッドを通じて復元されます。したがって、エディタの `get-admin` メソッドの戻り値は、呼び出しの文脈に依存します。

#### 5.1.3 スタイル

**スタイル**——`style<%>` インタフェースのインスタンス——は、すべてのスニップ・クラスに共通する高水準の表示情報をパラメータ化します。これには、項目を描くためのフォント、色、揃えが含まれます。各スニップには単一のスタイルが付けられます。

スタイルは階層的です。各スタイルは別のスタイルに基づいて定義されます。`"Basic"` という名前の単一の**ルート・スタイル**があり、エディタ内の他のすべてのスタイルはそこから派生します。基底スタイルと、それから派生した各スタイルとの違いは、**スタイル・デルタ（style delta）**（あるいは単に**デルタ**）に符号化されます。デルタは次のような変更を符号化します。

- フォント・ファミリを *X* に変更する
- ポイント・サイズに *Y* を加えてフォントを大きくする
- フォントの太字を切り替える
- すべてをスタイル記述 *Z* に一致させるように変更する

スタイル・オブジェクトが単独で作られることはなく、常に **スタイル・リスト**——`style-list%` クラスのインスタンス——を通じて作られます。スタイル・リストはスタイルを管理し、特定のスタイルを探す外部要求に応え、スタイル間の階層関係を管理します。グローバルなスタイル・リスト `the-style-list` が利用可能ですが、別々のスタイル階層を管理するために新しいスタイル・リストを作ることもできます。たとえば、各エディタは通常、自身のスタイル・リストを持ちます。

新しい各スタイルは、次の二通りのどちらかで定義されます。

- **派生スタイル（derived style）** は、基底スタイルとデルタに基づいて定義されます。すべてのスタイル（ルート・スタイルを除く）は、基底スタイルにまったく依存しなくても（すなわち、デルタが既存スタイルへの拡張ではなく固定スタイルを記述していても）基底スタイルを持ちます。（これは Microsoft Word などのワープロに見られる、通常の種類のスタイル継承です。）
- **結合スタイル（join style）** は、二つの他のスタイル——**基底スタイル**と**シフト・スタイル**——に基づいて定義されます。結合スタイルの意味は、シフト・スタイルを再解釈することで決まります。再解釈では、基底スタイルがシフト・スタイルのルート・スタイルとして使われます。（これは FrameMaker の段落スタイルと文字スタイルのような多水準スタイルに類似します。この類推では、段落スタイルが基底スタイル、文字スタイルがシフト・スタイルです。ただし FrameMaker はこれら二水準だけを許します。結合スタイルは任意の数の水準を支えます。）

通常、テキストがテキスト・エディタに挿入されるとき、それは直前のスニップのスタイルを継承します。空のエディタにテキストが挿入される場合、テキストは通常 `"Standard"` というスタイルが割り当てられます。既定では、`"Standard"` スタイルはルート・スタイルから変更されていません。既定のスタイル名は、`default-style-name` をオーバーライドすることで変更できます。

上の例外は、（選択が範囲ではなく位置であるとき）現在の選択位置で `text%` の `change-style` が呼ばれた場合です。その場合、スタイルが記憶され、次のエディタ変更操作がテキスト挿入なら、挿入されたテキストはその記憶されたスタイルを得ます。

挿入テキストのスタイルの詳細は、`text%` の `get-styles-sticky` を参照してください。

### 5.2 ファイル形式

エディタの内容をファイルに保存できるようにするため、エディタ・クラスは **WXME** と呼ばれる特別なファイル形式を実装します。（この形式は、アプリケーション間またはイベントスペース間での切り取りと貼り付けでも使われます。）ファイル形式は文書化されておらず、`WXME01‹digit›‹digit› ##` で始まることだけが決まっています。それ以外は、`load-file` と `save-file` メソッドが内部的に形式を定義します。ファイル形式はテキスト・エディタとペーストボード・エディタで同じです。ペーストボードが内容をファイルに保存するとき、スニップを前から後ろへ保存し、追加の場所情報も含めます。`wxme` ライブラリは、WXME ファイルを操作するためのユーティリティを提供します。

エディタ・データは `editor-stream-in%` と `editor-stream-out%` オブジェクトを使って読み書きされます。エディタ情報は、一度に一つのストリームからのみ読み書きできます。一つ以上のエディタをストリームへ書くには、まず関数 `write-editor-global-header` を呼んで初期化データを出力ストリームへ書きます。すべてのエディタをストリームへ書いたら、`write-editor-global-footer` を呼びます。同様に、ストリームからエディタを読むことは `read-editor-global-header` で初期化され、`read-editor-global-footer` で終了します。省略可能に、Racket の版をまたぐストリームを支えるには、ヘッダ操作の前に `write-editor-version` と `read-editor-version` を使います。

エディタ・ファイルのデータ形式は、別のファイル内に埋め込め、新しい種類のデータで拡張できます。エディタ・ファイル形式は二通りに拡張できます。スニップ固有または内容固有のデータによる拡張と、エディタ固有のグローバル・データによる拡張です。これらを本節の残りで述べます。

#### 5.2.1 スニップの符号化

スニップの一般化された概念により、新しいスニップ型を定義し、どのエディタ・クラスでも直ちに使えます。また、二つのアプリケーションが同じ種類のスニップをサポートしていれば、スニップ・データをそれらのあいだで容易に切り取り・貼り付けでき、同じデータ・ファイルを各プログラムが読めます。この相互運用性は、スニップ・システムに組み込まれた一貫した符号化機構によるものです。

スニップの優雅で拡張可能な符号化には、次の二つの問題に対処する必要があります。

- スニップの符号化関数はスニップ自体に関連付けられます。符号化された表現（例：ファイル内のバイト）からメモリ・オブジェクトへスニップを変換するには、各種類のスニップに対する復号関数を提供しなければなりません。さらに、高水準の復号過程が利用できる、そのような復号器のリストが必要です。この復号対応付けは、すべてのスニップに **スニップクラス・オブジェクト** を関連付けることで定義されます。スニップクラスは `snip-class%` クラスのインスタンスです。
- 一部のエディタは、スニップについて追加情報を格納する必要があり得ます。この情報は、スニップ自体が格納する型固有の情報とは直交します。たとえばペーストボードはスニップの場所を覚える必要がありますが、テキスト・エディタにはこの情報は必要ありません。データがあるペーストボードから別のペーストボードへ切り取られ貼り付けられるなら、相対的な場所についての情報を維持する必要がありますが、この情報はエディタへの貼り付けを妨げてはなりません。追加データは、**エディタ・データ・オブジェクト**——`editor-data%` クラスのインスタンス——を通じてスニップに関連付けられます。復号には、各エディタ・データ・オブジェクトが **エディタ・データ・クラス**——`editor-data-class%` クラスのインスタンス——を持つことが必要です。

スニップクラス、スニップ・データ、スニップ・データ・クラスは、スニップの符号化と復号に関する問題を解決します。ファイル保存や切り取りと貼り付けを必要としないアプリケーションでは、これらの問題を安全に無視できます。

##### 5.2.1.1 スニップクラス

各スニップはスニップクラスに関連付けられます。この「クラス」はプログラマの言語におけるクラス記述ではなく、符号化されたスニップ仕様から適切な型の新しいスニップを作る方法を提供するオブジェクトです。

スニップクラス・オブジェクトは、イベントスペース固有の**スニップクラス・リスト**——`get-the-snip-class-list` が返すもの——に追加できます。スニップが符号化されるとき、スニップのクラス名が符号化に関連付けられます。スニップを復号する必要があるとき、スニップクラス・リストが名前で検索されてスニップのクラスを見つけます。その後スニップクラスが、符号化から新しいスニップを作れる復号関数を提供します。

スニップクラスの名前が `"((lib ...) (lib ...))"` の形なら、スニップクラス実装をオンデマンドで読み込めます。名前は `read` を使って構文解析され、結果が `((lib string ...) (lib string ...))` の形なら、最初の要素が `'snip-class` とともに `dynamic-require` に使われます。`dynamic-require` の結果が `snip-class%` オブジェクトなら、それは現在のイベントスペースのスニップクラス・リストに挿入され、読み込みまたは保存は新しいクラスを使って続きます。

`"((lib ...) (lib ...))"` の二番目の `lib` 形式は、スニップのテキスト専用版のためのリーダを供給します。そのようなスニップクラスの働きの詳細は「Snip Class Mapping」を（そして一般に `wxme` ライブラリを）参照してください。

スニップクラスの名前は単に `"(lib ...)"` でもよく、二 `lib` 形式の前半と同様に使われます。ただし、この形式はテキスト専用の `wxme` リーダのための情報を提供しません。

例としてのスニップ実装と、それに関連付けられたスニップクラス実装については、「新しいスニップの実装」を参照してください。

##### 5.2.1.2 エディタデータ

スニップがエディタに属しているあいだ、エディタはスニップについての追加情報を特別な方法で格納することがあります。スニップを符号化するとき、この追加情報も符号化できるよう、**エディタ・データ・オブジェクト**に入れる必要があります。テキスト・エディタでは、追加情報をスニップだけでなく項目の範囲にも関連付けられます。

スニップが復号されるにはスニップクラスに関連付けられなければならないのと同様（「スニップクラス」を参照）、エディタ・データ・オブジェクトも復号のためにエディタ・データ・クラスを必要とします。すべてのエディタ・データ・クラス・オブジェクトは、イベントスペース固有のエディタ・データ・クラス・リスト——`get-the-editor-data-class-list` が返すもの——に追加できます。あるいは、スニップクラスと同様（「スニップクラス」を参照）、エディタ・データ・クラス名もオンデマンド読み込みを可能にする `"((lib ...) (lib ...))"` の形を使えます。対応するモジュールは、`'editor-data-class` という名前の `editor-data-class%` オブジェクトをエクスポートすべきです。

エディタ内のスニップまたは領域についての情報を格納・読み込むには：

- `editor-data%` と `editor-data-class%` から新しいクラスを派生する。
- `text%` または `pasteboard%` クラスから新しいクラスを派生し、`get-snip-data` と `set-snip-data` メソッド、および／または `get-region-data` と `set-region-data` メソッドをオーバーライドする。

  注：`get-region-data` と `set-region-data` メソッドは、切り取りと貼り付けの符号化では呼ばれますが、ファイル保存の符号化では呼ばれません。ファイル形式の拡張については「グローバルデータ：ヘッダとフッタ」を参照してください。

#### 5.2.2 グローバルデータ：ヘッダとフッタ

エディタ・ファイル形式は、特別なヘッダおよびフッタ節に追加のグローバル・データを足す手段を提供します。特別なヘッダおよび／またはフッタ・レコードを保存・読み込むには：

- 各ヘッダ／フッタ・レコードの名前を選びます。この名前は、使用中の他のどのヘッダ／フッタ・レコード名とも衝突してはならず、他の誰もこれらの名前を使うべきではありません。「wx」で始まるすべての名前は内部使用のために予約されています。追加のヘッダとフッタのレコードに一意な名前を付けることで、レコードをサポートしないインストールでもファイルを安全に読み込めます。
- `text%` または `pasteboard%` クラスから新しいクラスを派生し、`write-headers-to-file`、`write-footers-to-file`、`read-header-from-file` および／または `read-footer-from-file` メソッドをオーバーライドします。

エディタが保存されるとき、`write-headers-to-file` と `write-footers-to-file` メソッドが呼び出されます。このとき、派生した `text%` または `pasteboard%` オブジェクトはレコードを保存する機会を得ます。ヘッダ／フッタ・レコードを書くには、まず `begin-write-header-footer-to-file` メソッドを呼び出し、その時点でレコード名が提供されます。レコードを書き終えたら、`end-write-header-footer-to-file` を呼びます。

エディタが読み込まれ、ヘッダ／フッタ・レコードに出会うと、レコード名を引数として `read-header-from-file` または `read-footer-from-file` メソッドが呼び出されます。名前が既知のレコード型に一致すれば、データを読み込めます。

`write-headers-to-file` と `read-header-from-file` も参照してください。

### 5.3 行末の曖昧さ

エディタは改行項目がなくても強制的に改行できるため、位置だけではキャレットの場所を常に指定できるわけではありません。ソフト改行された（すなわち改行がない）行の最後の位置を考えてください。行の最後の項目と次の行の最初の項目のあいだに項目はないため、二つの場所（一つは行末、一つは行頭）が同じ位置に対応します。

このため、位置設定および位置取得のメソッドには、しばしば余分な引数があります。位置設定メソッドの場合、その引数は（場所が二重に定義されているとき）キャレットをページの左側と右側のどちらに描くかを指定します。`#t` は右側に描くことを意味します。同様に、場所から位置を計算するメソッドは、余分なボックス化された真偽値を取ります。そのボックスは、位置があいまいで右側の場所から来た場合は `#t` で、そうでなければ `#f` で埋められます。

### 5.4 新しいスニップの実装

エディタ内で新しい種類の内容を支えるには、`snip%` のサブクラスを派生して新しい種類のスニップを実装します。新しいスニップ実装を派生するとき、有用なスニップを作るには、次の `snip%` のメソッドをオーバーライドしなければなりません。

- `get-extent`
- `draw`
- `copy`
- ユーザがスニップをリサイズできるなら `resize`
- スニップが複数の項目を含められるなら `partial-offset`
- スニップが複数の項目を含められるなら `split`
- スニップが `get-extent` の結果をキャッシュするなら `size-cache-invalid`
- `get-text`（必須ではない）
- スニップが複数のスクロール位置を含められるなら `find-scroll-step`、`get-num-scroll-steps`、`get-scroll-step-offset`
- スニップの内部状態をユーザが変更できるなら `set-unmodified` を実装し、状態が最初に変わるときにスニップの管理者の `modified` を呼ぶ

スニップが複数の項目を含められるなら、スニップのカウントも維持しなければなりません。

保存または切り取りと貼り付けが可能なスニップのクラスを定義するには（「スニップクラス」も参照）：

- `snip-class%` のインスタンスを作り、`read` メソッドを実装する。`snip-class%` インスタンスをモジュールから `snip-class` としてエクスポートし、「スニップクラス」で述べる `"(lib ...)"` の形のクラス名を使う。
- スニップ・クラスの各インスタンスに対し、`set-snipclass` でスニップのクラス・オブジェクトを設定する。
- `copy` メソッドをオーバーライドする。
- `write` メソッドをオーバーライドする。

新しい `snip-class%` クラスを派生するとき：

- `set-classname` でクラス名を設定する。
- `set-version` で版を設定する。
- `get-the-snip-class-list` が返すリストへ、`add` メソッドを使ってクラスをインストールする。同じ名前が同じクラス・リストへ複数回挿入されても、最初の挿入以外は無視されることに注意してください。

`open-input-text-editor` で特別に読まれるスニップのクラスを定義するには：

- あなたの `snip%` クラスに `readable-snip<%>` を実装させる。
- `read-special` メソッドを実装する。

例として、次のモジュールは円を描くスニップを実装します。スニップをクリックすると円が大きくなります。プログラム／イベントスペース間でスニップのインスタンスをコピーできるようにするには、モジュールを `"circle-snip"` パッケージとしてインストールされた `"circle-snip"` ディレクトリ内の `"main.rkt"` にすべきです。スニップにはそれに続く `wxme` リーダ実装もあり、それを `"circle-snip"` ディレクトリ内のファイル `"wxme-circle-snip.rkt"` としてインストールしなければなりません。

```racket
  #lang racket/base
  (require racket/class
           racket/snip
           racket/format)

  (provide circle-snip%
           (rename-out [circle-snip-class snip-class]))

  (define circle-snip%
    (class snip%
      (inherit set-snipclass
               get-flags set-flags
               get-admin)
      (init-field [size 20.0])

      (super-new)
      (set-snipclass circle-snip-class)
      (send (get-the-snip-class-list) add circle-snip-class)
      (set-flags (cons 'handles-events (get-flags)))

      (define/override (get-extent dc x y
                                   [w #f]
                                   [h #f]
                                   [descent #f]
                                   [space #f]
                                   [lspace #f]
                                   [rspace #f])
        (define (maybe-set-box! b v) (when b (set-box! b v)))
        (maybe-set-box! w (+ 2.0 size))
        (maybe-set-box! h (+ 2.0 size))
        (maybe-set-box! descent 1.0)
        (maybe-set-box! space 1.0)
        (maybe-set-box! lspace 1.0)
        (maybe-set-box! rspace 1.0))

      (define/override (draw dc x y left top right bottom dx dy draw-caret)
        (send dc draw-ellipse (+ x 1.0) (+ y 1.0) size size))

      (define/override (copy)
        (new circle-snip% [size size]))

      (define/override (write f)
        (send f put size))

      (define/override (on-event dc x y editorx editory e)
        (when (send e button-down?)
          (set! size (+ 1.0 size))
          (define admin (get-admin))
          (when admin
            (send admin resized this #t))))))

  (define circle-snip-class%
    (class snip-class%
      (inherit set-classname)

      (super-new)
      (set-classname (~s '((lib "main.rkt" "circle-snip")
                           (lib "wxme-circle-snip.rkt" "circle-snip"))))

      (define/override (read f)
        (define size-b (box 0.0))
        (send f get size-b)
        (new circle-snip% [size (unbox size-b)]))))

  (define circle-snip-class (new circle-snip-class%))
```

これが `"wxme-circle-snip.rkt"` ファイルです。

  (require racket/class
           racket/format
           wxme

  (provide reader)

  (define circle-reader%

### 5.5 平坦化されたテキスト

プレーン・テキスト・エディタでは、位置と文字のあいだに単純な対応があります。`editor<%>` オブジェクトでは、多くの場合これは成り立ちませんが、それでもエディタの「テキストを得る」ことが有用なときがあります。

エディタからテキストを取り出す形は次の二通りです。

- **単純テキスト** — 項目ごとに一つの文字がある形。文字である項目は自分自身に写像され、他のすべての項目はピリオドに写像されます。改行は改行文字（ASCII 10）で表されます。
- **平坦化されたテキスト** — 各項目が任意の文字列に写像される形。文字である項目は依然として自分自身に写像されますが、より複雑な項目は、項目のスニップが決める有用な文字列で表せます。改行はプラットフォーム固有の文字列（Unix と Mac OS ではラインフィード、Windows ではラインフィード–キャリッジリターン）に写像されます。エディタの項目が文字の線形列に還元されているため、この形は「平坦化された」と呼ばれます。

### 5.6 キャレットの所有権

フレーム内では、キーボード・フォーカスを含められるオブジェクトは一つだけです。フレームが複数のディスプレイ内に複数のエディタを含むとき、また単一のエディタが他のエディタを項目として含むときにも、この性質を維持しなければなりません。

エディタがキーボード・フォーカスを持つとき、通常は現在の選択、または挿入点を示す線を表示します。その線は**キャレット**と呼ばれます。

エディタが他のエディタを含むとき、含まれたサブエディタ間のキャレット所有権を追跡します。キャレットが主エディタから取り上げられると、適切なサブエディタからキャレット所有権を取り消します。

エディタまたはスニップが描かれるとき、描画メソッドへの引数が、キャレットをデータとともに描くべきか、選択がデータにまたがるかを指定します。この引数は次のいずれかです。

- `'no-caret` — キャレットをまったく描くべきではありません。
- `'show-inactive-caret` — キャレットを非アクティブとして描くべきです。項目は局所的な現在の選択として識別され得ますが、キーボード・フォーカスは別の場所にあります。
- `'show-caret` — キーボード・フォーカスの所有を示すためにキャレットを描くべきです。
- `(cons start end)` — キャレットは囲む領域が所有し、その選択が現在のエディタまたはスニップにまたがります。スニップの場合、選択はスニップ内の `start` から `end` 位置までの要素にまたがります。

`'show-inactive-caret` 表示モードは、フォーカスを持たないテキスト・エディタで選択範囲を示すのに有用です。この `'show-inactive-caret` モードは `'no-caret` モードとは異なります。エディタが埋め込まれているとき、局所的にアクティブなエディタだけが選択を表示します。

### 5.7 切り取りと貼り付けのタイムスタンプ

クリップボードを使う `editor<%>` のメソッド——`copy`、`cut`、`paste`、`do-edit-operation` を含む——は、タイムスタンプ引数を消費します。このタイムスタンプは、一般にクリップボード操作を引き起こした `mouse-event%` または `key-event%` オブジェクトから取り出されます。Unix は、クリップボード・クライアント間でクリップボード操作を同期するためにタイムスタンプを使います。

`event%` のすべてのインスタンスはタイムスタンプを含み、`get-time-stamp` を使って得られます。

タイムスタンプが 0 の場合、現在時刻に既定されます。タイムスタンプとして 0 を使うことはほとんど常に問題なく動きますが、Unix では行儀が悪いと見なされます。

### 5.8 クリックバック

`text%` エディタの**クリックバック**は、ハイパーテキストのような単純な対話的オブジェクトの作成を助けます。クリックバックは、エディタ内の項目の範囲にコールバック関数を関連付けることで定義されます。ユーザがその範囲の項目をクリックすると、コールバック関数が呼び出されます。たとえばハイパーテキストのクリックバックは、エディタ内の選択範囲を変えるコールバック関数に範囲を関連付けます。

既定では、コールバック関数はユーザがマウスボタンを離したときに呼び出されます。`set-clickback` メソッドは、代わりにボタン押し下げ時にコールバック関数を呼び出す省略可能な引数を受け取ります。この振る舞いは、たとえばポップアップ・メニューを作るクリックバックに有用です。

クリックバックには任意の手続きが関連付けられるため、ファイルが保存されるときにクリックバック情報を保存しようとする試みはないことに注意してください。

### 5.9 内部エディタロック

`editor<%>` のインスタンスは、三水準の内部ロックを持ちます。

- **書き込みロック** — エディタが内部的に書き込み用にロックされているとき、エディタの抽象的な内容は変更できません（例：挿入の試みは黙って失敗します）。ただし、テキスト・エディタ内のスニップは依然として分割・結合でき、テキスト・エディタは行の流れに影響する仕方で変更できます。`locked-for-write?` メソッドは、エディタが現在書き込み用にロックされているかどうかを報告します。
- **フロー・ロック** — テキスト・エディタが内部的にリフロー用にロックされているとき、それは書き込み用にもロックされ、エディタのスニップ内容は変更できず、スニップの場所がまだ分かっていなければ計算できず（`editor<%>` の `locations-computed?` を参照）、エディタをディスプレイへ描くこともできません。フロー・ロック中に未計算の場所情報を要求すると、未定義の結果になります。`locked-for-flow?` メソッドは、エディタが現在フロー用にロックされているかどうかを報告します。
- **読み取りロック** — エディタが内部的に読み取り用にロックされているとき、エディタに対して操作を実行できません（例：現在の選択位置の要求は未定義の値を返します）。この極端な状態は、スニップの管理者の設定、スニップの分割、またはスニップの結合のためのスニップへのコールバック中にのみ使われます。`locked-for-read?` メソッドは、エディタが現在読み取り用にロックされているかどうかを報告します。

エディタの内部ロックは、`lock` への呼び出しの影響を受けません。

エディタについての場所に依存しない情報を報告するメソッドは、ロックを引き起こしません。場所情報を報告するメソッドは、エディタへの最後の変更以降に関連情報が計算されていなければ、フロー・ロックまたは書き込みロックを引き起こすことがあります（`editor<%>` の `locations-computed?` を参照）。選択位置の設定など、どのような仕方でもエディタを変更するメソッドは、読み取りロック、フロー・ロック、または書き込みロックを引き起こし得ます。

### 5.10 エディタとスレッド

エディタは、キャンバスに表示される程度（キャンバスはイベントスペースを持つ）を除き、特定のスレッドやイベントスペースに結び付いていません。エディタへの並行アクセスは、エディタが壊れることがないという意味では常に安全です。ただし、エディタへのアクセスはロックを引き起こし得るため、並行アクセスは契約失敗や予期しない結果を生み得ます。

エディタは、特定の並行パターンを信頼できる仕方で支えます。一つの関連パターンは、あるスレッドでエディタを更新しつつ、別の（ハンドラ）スレッドが管理するキャンバスにエディタを表示することです。キャンバスの再描画がフロー用ロック中に行われないこと、および再描画がエディタ変更を妨げないことを保証するため、次のことが保証されます。

- 編集シーケンス中（`begin-edit-sequence` で始まり `end-edit-sequence` で終わる）にエディタの `refresh` メソッドが呼ばれると、要求された再描画領域は記録されますが、再描画は実行されません。代わりに、再描画は編集シーケンスの終わりまで遅延されます。
- 再描画の進行中に編集シーケンスを開始しようとすると、再描画が完了するまでブロックします。
- `on-display-size-when-ready` メソッドは、エディタが再描画中でなく、編集シーケンスも進行中でないときだけ `on-display-size` を呼びます。最初の場合、`on-display-size` の呼び出しは再描画スレッドへ委譲され、再描画完了後に呼ばれます。二番目の場合、`on-display-size` の呼び出しは編集シーケンス・スレッドへ委譲され、編集シーケンス完了時に呼ばれます。

したがって、`editor-canvas%` オブジェクトを（`enable` を使って）無効化するだけで、キャンバスが表示するエディタをバックグラウンド・スレッドが変更できることが保証されます。ただし、すべての変更は編集シーケンス内になければなりません。バックグラウンドの変更はキャンバス再描画を最小限かつ一時的に損ない、再描画はバックグラウンド・スレッドでの変更を損ないません。

第二の支えられたパターンは、他のスレッドでエディタが操作され得るあいだに、バックグラウンド・スレッドでエディタを読むことです。場所に依存しない読み取りはロックを導入しないため、バックグラウンド・スレッドでのそのような読み取りは他のスレッドを損ないません。ただし、他のスレッドがバックグラウンド・スレッドを妨害し、誤ったまたは古い内容情報を受け取らせることがあります。この片側の保証は、エディタが変更されたときにバックグラウンド・スレッドの作業を捨てられる場合に有用です。
