これが標準的な試験問題に出たら、次のように答えるかもしれない。

```racket
(define (f x) (expt x 3))
```

しかし、次の定義を退ける根拠もない。

```racket
(define (f x) (if (= x 2) 8 (* x x)))
```

テストが教えるのは、ある入力についてプログラムが期待どおりに動くということだけである。

同じ精神で、特定の入力に対するプログラム適用の評価を計時しても分かるのは、それらの入力について答えを計算するのにどれだけ時間がかかるか——それだけである。同じ入力に対して同じ答えを計算する2つのプログラム——`prog-linear` と `prog-square`——があり、選んだすべての入力で `prog-linear` の方が常に `prog-square` より速く答えを出す、ということがあり得る。Making Choices はちょうどそのようなプログラム対を示す。構造再帰のプログラム `gcd` と、等価だが生成的再帰のプログラム `gcd-generative` である。計時比較は、後者の方が前者よりはるかに速いことを示唆する。

## 具体的な時間、抽象的な時間（Concrete Time, Abstract Time）

Making Choices は `gcd` と `gcd-generative` の実行時間を比較する。加えて、後者の方が常に前者より少ない再帰ステップで答えを計算するので優れている、と論じる。この考えを出発点に、Designing with Self-Referential Data Definitions の単純なプログラム `how-many` の性能を解析する。

```racket
(define (how-many a-list)
  (cond
    [(empty? a-list) 0]
    [else (+ (how-many (rest a-list)) 1)]))
```

未知の空でないリストの長さを計算するのにどれだけかかるかを知りたいとしよう。Intermezzo 1: Beginning Student Language の計算規則を使い、この過程を一連の代数的操作として見ることができる。

```racket
(how-many some-non-empty-list)
==
(cond
  [(empty? some-non-empty-list) 0]
  [else (+ (how-many (rest some-non-empty-list)) 1)])
==
(cond
  [#false 0]
  [else (+ (how-many (rest some-non-empty-list)) 1)])
==
(cond
  [else (+ (how-many (rest some-non-empty-list)) 1)])
==
(+ (how-many (rest some-non-empty-list)) 1)
```

最初のステップは、`how-many` の定義中の `a-list` を実際の引数 `some-non-empty-list` で置き換えることであり、これで最初の `cond` 式が得られる。次に評価しなければならないのは

```racket
(empty? some-non-empty-list)
```

である。仮定により結果は `#false` である。問題は、この結果を判定するのにどれだけかかるかだ。正確な時間量は分からないが、リストのコンストラクタを調べるには小さく固定の時間がかかると言って差し支えない。実際、次のステップ——`cond` が第一条件の値を調べるとき——についても同じ仮定が成り立つ。それが `#false` なので、最初の `cond` 行は落とされる。`cond` 行が `else` で始まるかどうかの検査も同様に速い。すると残るのは

```racket
(+ (how-many (rest some-non-empty-list)) 1)
```

である。

最後に、`rest` がリストの残りを固定時間で取り出すと安全に仮定できる。しかしそれ以外は行き詰まったように見える。`how-many` があるリストの長さを決めるのにどれだけかかるかを計算するには、そのリストの残りに何項目あるかを `how-many` が数えるのにどれだけかかるかを知る必要がある。

別の見方として、述語とセレクタが何らかの固定時間を要すると仮定すれば、`how-many` がリストの長さを決めるのにかかる時間は、それが行う再帰ステップの回数に依存する。もう少し正確に言えば、`(how-many some-list)` の評価は、リストの長さ——あるいは同等に、プログラムが再帰する回数——を `n` として、おおよそ `n` 倍の固定量かかる。

この例から一般化すると、実行時間は入力のサイズに依存し、再帰ステップの回数が評価列の長さのよい見積りになる、と示唆される。このため計算機科学者は、プログラムの**抽象実行時間（abstract running time）**を、入力のサイズと評価中の再帰ステップ数の関係として論じる。最初の例では、入力のサイズはリスト上の項目数である。したがって1項目のリストは1回の再帰ステップを要し、2項目なら2ステップ、`n` 項目のリストでは `n` ステップである。

> **注:** 「抽象」と呼ぶのは、この尺度がプリミティブな各ステップにどれだけ時間がかかるかの詳細を無視するからである。

計算機科学者は、プログラム `f` が「`n` ステップのオーダーで（on the order of n steps）」かかる、という言い回しを使い、`f` の抽象実行時間についての主張を定式化する。この言い回しを正しく使うには、`n` の説明が付いていなければならない。たとえば「与えられたリスト上の項目数を数える」とか「与えられた数の桁数である」などである。そのような説明がなければ、もともとの言い回しは実際には無意味である。

すべてのプログラムが `how-many` のような単純な抽象実行時間を持つわけではない。本書で最初の再帰プログラムを見てみよう。

```racket
(define (contains-flatt? lo-names)
  (cond
    [(empty? lo-names) #false]
    [(cons? lo-names)
     (or (string=? (first lo-names) 'flatt)
         (contains-flatt? (rest lo-names)))]))
```

`'flatt` で始まるリスト、たとえば

```racket
(contains-flatt?
  (list "flatt" "robot" "ball" "game-boy" "pokemon"))
```

については、プログラムは再帰ステップを必要としない。対照的に、`'flatt` がリストの末尾にある場合、たとえば

```racket
(contains-flatt?
  (list "robot" "ball" "game-boy" "pokemon" "flatt"))
```

では、評価にはリスト中の項目数と同じ回数の再帰ステップが必要になる。

この第二の解析は、プログラム解析の第二の重要な考え、すなわち**どのような種類の解析を行うか**へ導く。

- **最良ケース解析（best-case analysis）**は、プログラムが容易に答えを見つけられる入力のクラスに焦点を当てる。実行中の例では、`'flatt` で始まるリストが最良の種類の入力である。
- 一方、**最悪ケース解析（worst-case analysis）**は、プログラムを最も酷使する入力に対してどれだけ悪く振る舞うかを決める。`contains-flatt?` 関数は、`'flatt` が入力リストの末尾にあるときに最悪の性能を示す。
- 最後に、**平均解析（average analysis）**は、プログラマは入力が常に最良の形だと仮定できず、また最悪の形でもないことを願うしかない、という考えから始まる。多くの場合、プログラムがかかる**平均時間**を見積もらなければならない。たとえば `contains-flatt?` は、平均して、入力リストのどこか真ん中で `'flatt` を見つける。したがって後者が `n` 項目からなるなら、`contains-flatt?` の平均実行時間は [image: pict_238.png]（およそ `n/2`）である。つまり、入力上の項目数の半分だけ再帰する。

したがって計算機科学者は通常、「オーダーで」という言い回しを「平均で」または「最悪ケースで」と組み合わせて用いる。

## 「オーダーで」の定義（The Definition of “On the Order Of”）

前節は、「オーダーで」という言い回しのすべての鍵となる材料に触れた。いまこそこの言い回しの厳密な記述を導入する時である。前節が展開した2つの考えから始めよう。

1. 性能の抽象的な測定は、2つの量の関係である。入力のサイズと、答えを決めるのに必要な再帰ステップ数である。この関係は実際には、ある自然数（入力のサイズ）を別の自然数（必要な時間）に写す数学的関数である。
2. したがって、プログラムの性能についての一般的な主張は関数についての主張であり、2つのプログラムの性能比較は、そのような2つの関数の比較を求める。

そのような一方の関数が他方より「良い」かどうかを、どう決めるか。

> **注:** 演習245は別の問い——2つの他のプログラムが等しいかどうかを判定するプログラムを定式化できるか——に取り組む。この Intermezzo ではプログラムを書いていない。平明な数学的議論を使っている。

導入部の想像上のプログラム `prog-linear` と `prog-square` に戻ろう。それらは同じ結果を計算するが、性能は異なる。`prog-linear` プログラムは「`n` ステップのオーダー」を要し、`prog-square` は「`n²` ステップのオーダー」を使う。数学的に言えば、`prog-linear` の性能関数は

この最後の洞察が、「オーダー」という言い回しの正確な定義への鍵である。自然数上の関数 `f` が、すべての自然数についてある関数 `g` より大きな数を生成するなら、`f` は明らかに `g` より大きい。しかしこの比較がわずかな入力——たとえば 1000 や 1000000——についてだけ失敗し、他のすべてについて成り立つならどうか。その場合でも、`f` は `g` より良いと言いたい。これが次の定義につながる。

> **定義** 自然数上の関数 `g` が与えられたとき、`O(g)`（読み：ジーのビッグオー、big-O of g）は自然数上の関数のクラスである。関数 `f` が `O(g)` のメンバーであるとは、数 `c` と `bigEnough` が存在して、

## なぜプログラムは述語とセレクタを使うのか（Why Do Programs Use Predicates and Selectors?）

「オーダーで」という概念は、設計レシピがなぜよく組織され「性能の良い（performant）」プログラムを生むのかを説明する。この洞察を、数のリスト中で数を探すプログラムの設計という、単一の例で示す。ここにシグネチャ、目的ステートメント、テストとして定式化した例がある。

```racket
; Number [List-of Number] -> Boolean
; is x in l

(check-expect (search 0 '(3 2 1 0)) #true)
(check-expect (search 4 '(3 2 1 0)) #false)
```

これらの期待に応える定義が2つある。

> (define (searchL x l)
>   (cond
>     [(empty? l) #false]
>     [else
>      (or (= (first l) x)
>          (searchL x (rest l)))]))
>
> (define (searchS x l)
>   (cond
>     [(= (length l) 0) #false]
>     [else
>      (or (= (first l) x)
>          (searchS x (rest l)))]))

左側のプログラムの設計は設計レシピに従う。特に、テンプレートの開発は、データ定義の節ごとに構造的述語の使用を求める。この助言に従うと、最初の `cond` 行が空リストを扱い、第二行がそれ以外すべてを扱う条件付きプログラムが得られる。最初の `cond` 行の質問は `empty?` を使い、第二は `cons?` または `else` を使う。

`searchS` の設計は、構造的設計レシピに達していない。代わりに、リストはサイズを持つコンテナだという考えから着想を得る。したがってプログラムはこのサイズが 0 かを検査でき、それは空であることの検査と等価である。

> **注:** 実際には生成的再帰を使っている。

この考えは機能的には正しいが、`*SL` が提供する演算のコストが固定定数だという仮定を置く。しかし `length` が `how-many` のようであれば、`searchS` は `searchL` より遅くなる。新しい用語を使えば、`searchL` は `O(n)` の再帰ステップを使い、一方 `searchS` は `n` 項目のリストについて `O(n²)` ステップを要する。要するに、条件を定式化するのに任意の `*SL` 演算を使うと、性能をある関数クラスから、はるかに悪いクラスへ移してしまうことがある。

`length` が定数時間の関数か、それとも与えられたリストの長さに比例して時間を消費するかを確かめる実験で、この Intermezzo を締めくくろう。最も簡単な方法は、長いリストを作り、探索プログラムの各版がどれだけ時間を要するかを決めるプログラムを定義することである。

```racket
; N -> [List Number Number]
; how long do searchS and searchL take
; to look for n in (list 0... (- n 1))
(define (timing n)
  (local ((define long-list
            (build-list n (lambda (x) x))))
    (list
      (time (searchS n long-list))
      (time (searchL n long-list)))))
```

いまこのプログラムを 10000 と 20000 で実行せよ。`length` が `empty?` のようであれば、2回目の実行の時間は1回目のおよそ2倍になる。そうでなければ、`searchS` の時間は劇的に増える。

ストップ！ 実験を行え。

実験を終えたと仮定すると、`length` が与えられたリストのサイズに比例して時間を要ることがいま分かる。`searchS` の「S」は「squared（二乗）」を表す。その実行時間が `O(n²)` だからである。しかし、この種の推論がこれから出会うあらゆるプログラミング言語に当てはまる、と結論を急がないこと。多くはコンテナを `*SL` とは違う扱い方をする。それがどう行われるかを理解するには、あと1つの設計概念——蓄積子（accumulators）——が必要であり、それが本書最終部の関心である。

> **注:** 他の言語がコンテナのサイズをどう追跡するかは、Data Representations with Accumulators を見よ。
