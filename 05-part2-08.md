# 8 リスト（Lists）

**原本:** `extracted/original_markdown_05_part_two.md`（第8章）

これまで自己参照的な定義に出会ったことはおそらくないでしょう。国語（英語）の教師はこうした定義を確実に避けますし、多くの数学の授業でもこうした定義は曖昧に扱われます。しかし、プログラマは曖昧さを許容できません。プログラマの仕事には正確さが必要です。一般に、定義には自分自身への参照が複数含まれることもありますが、この章ではリストを皮切りに、自己参照をちょうど1つだけ必要とする有用な例を紹介します。

リストを導入することで、私たちが学べるアプリケーションの幅も広がります。この章では例を通じて直感を丁寧に築き上げるとともに、自己参照データを体系的に処理する関数の作り方を説明する次の章での設計レシピの改訂へと動機づけます。

## 8.1 リストの作成（Creating Lists）

私たちは皆、日常的にリストを作っています。食料品の買い物に行く前には、買いたい品目のリストを書き留めます。毎朝 ToDo リストを書く人もいます。12月には、多くの子どもたちがクリスマスの欲しいものリストを用意します。パーティーを計画するときは、招待者のリストを作ります。情報をリストの形で整理することは、私たちの生活のいたるところにあります。

情報がリストの形で与えられる以上、そうしたリストを初級学生言語（BSL）のデータとして表現する方法を、はっきりと学ばなければなりません。実際、リストはとても重要なので、BSL にはデカルト座標の点（`posn`）と同様に、リストの作成と操作のための組み込みサポートが備わっています。点とは対照的に、リストのデータ定義はいつも自分で書くことになります。しかし、まずは順を追って進めましょう。リストの作成から始めます。

リストを作るときは、いつも空リストから始めます。BSL では、空リストを次のように表します。

```racket
'()
```

これは「エンプティ（empty）」と発音され、「空リスト（empty list）」の略です。`#true` や `5` と同様に、`'()` は単なる定数です。リストに何かを加えるときは、別のリストを構築します。BSL では、`cons` 操作がこの目的を果たします。例えば、

```racket
(cons "Mercury" '())
```

は、`'()` リストと文字列 `"Mercury"` からリストを構築します。図44は、このリストを構造体と同じ絵で示したものです。`cons` の箱には `first` と `rest` という2つのフィールドがあります。この具体例では、`first` フィールドに `"Mercury"` が入り、`rest` フィールドに `'()` が入っています。

```racket

```

また、別の例も示します。

```racket
(cons "Earth" (cons "Mercury" '()))
```



このリストを構築するには、10 回のリスト構築と 1 つの `'()` が必要です。例えば、3 つの任意の数値からなるリスト、

```racket

  (cons e
    (cons -22.3 '())))
```

を構築するには、3 つの `cons` が必要です。

一般に、リストは1種類だけの値を含む必要はなく、任意の値を含んで構いません。

```racket
(cons "Robbie Round"
  (cons 3
    (cons #true
      '())))
```

このリストの最初の項目は文字列、2番目は数値、最後はブール値です。このリストは、従業員の氏名、その会社での在籍年数、会社経由で健康保険に加入しているかどうかという3つのデータからなる人事レコードの表現と考えることができます。あるいは、あるゲームの仮想プレイヤーを表していると考えることもできます。データ定義がなければ、そのデータが何についてのものなのかは分かりません。

> **注:** とはいえ、もしこのリストが固定個数の部品からなるレコードを表すつもりなら、代わりに構造体型を使ってください。

以下は、`cons` を含む最初のデータ定義です。

```racket
; A 3LON is a list of three numbers:
;   (cons Number (cons Number (cons Number '())))
; interpretation a point in 3-dimensional space
```

もちろん、このデータ定義は、他のデータ定義が構造体インスタンスのコンストラクタを使うのと同じように `cons` を使っており、ある意味で `cons` は特別なコンストラクタにすぎません。このデータ定義が示せていないのは、任意の長さのリスト——何も含まないリスト、1項目、2項目、10項目、あるいは 1,438,901 項目のリスト——をどう形成するかです。

そこで、もう一度試してみましょう。

```racket
; A List-of-names is one of:
; – '()
; – (cons String List-of-names)
; interpretation a list of invitees, by last name
```

深呼吸をして、もう一度読んでください。このデータ定義は、これまでに出会った中で最も珍しい定義の1つです——自分自身を参照する定義を見たことがないからです。これが意味をなすかどうかさえ明らかではありません。結局のところ、国語の先生に「『机』とは『机』のことである」と「机」という言葉を定義したと伝えたら、いちばんありそうな反応は「ナンセンス！」でしょう。自己参照的な定義は、言葉の意味を説明しないからです。

しかし、コンピュータ科学やプログラミングにおいては、自己参照的な定義が中心的な役割を果たします。そして注意を払えば、そうした定義は実際に意味をなします。ここで「意味をなす」とは、データ定義を本来の目的——定義されているクラスに属するデータの例を生成したり、与えられたデータ片が定義されたクラスに属するかどうかを確認したりすること——に使える、ということです。この観点から見ると、`List-of-names` の定義は完全に意味をなしています。最低限、項目化の最初の節を使って、1つの例として `'()` を生成できます。`'()` が `List-of-names` の要素であることが分かれば、2つ目の例を作るのも簡単です。

```racket
(cons "Findler" '())
```

ここでは、文字列（`String`）と `List-of-names` に属する唯一のリストを使って、項目化の2番目の節に従ってデータ片を生成しています。同じ規則で、この種のリストをさらに多く生成できます。

```racket
(cons "Flatt" '())
(cons "Felleisen" '())
(cons "Krishnamurthi" '())
```

これらのリストにはいずれも1つの名前（文字列として表現されたもの）しか含まれていませんが、データ定義の2行目を使って、より多くの名前を含むリストを作ることもできます。

```racket
(cons "Felleisen" (cons "Findler" '()))
```

このデータ片は `List-of-names` に属します。なぜなら、`"Felleisen"` は `String` であり、`(cons "Findler" '())` はすでに確認された `List-of-names` だからです。

## 8.2 `'()` とは何か、`cons` とは何か（What Is '(), What Is cons）

少し立ち止まって、`'()` と `cons` をよく見てみましょう。前述のとおり、`'()` は単なる定数です。`5` や `"this is a string"` などの定数と比べると、関数名や変数のように見えます。しかし、`#true` や `#false` と比べれば、これが BSL における空リストの表現にすぎないことがすぐに分かるはずです。

評価規則の観点では、`'()` は数値・ブール値・文字列などとは異なる、新しい種類の原子的な値です。また、`Posn` のような複合値でもありません。実際、`'()` は非常に独特で、それ自体だけで1つの値のクラスに属します。そのため、この値のクラスには、`'()` だけを認識し、他には何も認識しない述語が付いています。

```racket
; Any -> Boolean
; is the given value '()
(define (empty? x)...)
```

すべての述語と同様に、`empty?` は BSL の値の宇宙の任意の値に適用できます。これが `#true` を生成するのは、まさに `'()` に適用されたときです。

```racket
> (empty? '())
#true
> (empty? 5)
#false
> (empty? "hello world")
#false
> (empty? (cons 1 '()))
#false
> (empty? (make-posn 0 0))
#false
```

次に `cons` に移ります。これまで見てきたことはすべて、`cons` が構造体型定義によって導入されるコンストラクタと同じであることを示唆しています。より正確には、`cons` は2フィールド構造体のコンストラクタであるように見えます。最初のフィールドは任意の種類の値、2番目のフィールドはリストらしい任意の値です。次の定義は、この考えを BSL に写したものです。

```racket
(define-struct pair [left right])
; A ConsPair is a structure:
;   (make-pair Any Any).

; Any Any -> ConsPair
(define (our-cons a-value a-list)
  (make-pair a-value a-list))
```

唯一の問題は、`our-cons` が第2引数としてあらゆる可能な BSL 値を受け入れるのに対し、`cons` はそうではないという点です。次の実験がそれを確かめます。

```racket
> (cons 1 2)
cons:second argument must be a list, but received 1 and 2
```

言い換えれば、`cons` は実際には「項目化と構造体」で論じた種類の、検査付き関数です。そこで次のような洗練が示唆されます。

```racket
; A ConsOrEmpty is one of:
; – '()
; – (make-pair Any ConsOrEmpty)
; interpretation ConsOrEmpty is the class of all lists

; Any Any -> ConsOrEmpty
(define (our-cons a-value a-list)
  (cond
    [(empty? a-list) (make-pair a-value a-list)]
    [(pair? a-list) (make-pair a-value a-list)]
    [else (error "cons: second argument...")]))
```

もし `cons` が検査付きのコンストラクタ関数なら、結果の構造体から部品をどう取り出すのか疑問に思うかもしれません。結局のところ、「構造体の追加」では、構造体を使ったプログラミングにはセレクタが必要だと述べています。`cons` 構造体には2つのフィールドがあるので、セレクタも2つあります。`first` と `rest` です。これらも、私たちの `pair` 構造体を使って簡単に定義できます。

```racket
; ConsOrEmpty -> Any
; extracts the left part of the given pair
(define (our-first a-list)
  (if (empty? a-list)
      (error 'our-first "...")
      (pair-left a-list)))
```

立ち止まって、`our-rest` を定義せよ。

もしプログラムが `pair` の構造体型定義にアクセスできると、右フィールドに `'()` や別の `pair` を含まないペアを簡単に作れてしまいます。そうした悪いインスタンスが意図的に作られようと偶発的に作られようと、関数やプログラムを奇妙な仕方で壊しがちです。そのため BSL は、こうした問題を避けるために `cons` の実際の構造体型定義を隠しています。「局所定義」では、プログラムがそうした定義を隠す方法の1つを示しますが、今のところその力は必要ありません。

```
+------------------------------------------------------+
| Figure 46: List primitives                           |
|                                                      |
| +--------+--+--------------------------------------+ |
| | '()    |  | a special value, mostly to represen… | |
| +--------+--+--------------------------------------+ |
| | empty? |  | a predicate to recognize '() and no… | |
| | cons   |  | a checked constructor to create two… | |
| | first  |  | the selector to extract the last it… | |
| | rest   |  | the selector to extract the extende… | |
| | cons?  |  | a predicate to recognize instances … | |
| +--------+--+--------------------------------------+ |
+------------------------------------------------------+
```

図46はこの節をまとめたものです。鍵となる洞察は、`'()` が一意な値であること、そして `cons` がリスト値を生成する検査付きコンストラクタであることです。さらに、`first`、`rest`、`cons?` は、通常の述語とセレクタに対する別々の名前にすぎません。したがって、この章が教えるのはデータの新しい作り方ではなく、データ定義を定式化する新しい仕方です。

## 8.3 リストを使ったプログラミング（Programming with Lists）

友人のリストを何らかの形で管理しているとします。そのリストが大きくなりすぎて、ある名前がリストにあるかどうかを判定するプログラムが必要になったとしましょう。この考えを具体的にするために、演習として述べます。

> **注:** ここでいう「友人」は、現実世界ではなく、ソーシャルネットワークの意味で使っています。

> **サンプル問題** あなたは、ある新しい携帯電話の連絡先リストに取り組んでいます。所有者はさまざまな場面でこのリストを更新・参照します。当面、あなたに割り当てられた仕事は、この連絡先リストを受け取り、名前「Flatt」が含まれているかどうかを判定する関数を設計することです。

このサンプル問題の解ができたら、リスト上の任意の名前を探す関数へと一般化します。

前節の `List-of-names` のデータ定義は、関数が探索すべき名前のリストを表すのに適しています。次にヘッダー材料に移ります。

```racket
; List-of-names -> Boolean
; determines whether "Flatt" is on a-list-of-names
(define (contains-flatt? a-list-of-names)
  #false)
```

`a-list-of-names` は関数が受け取る名前のリストとして良い名前ですが、長すぎるので `alon` に短縮します。

一般的な設計レシピに従い、次に関数の目的を説明する例をいくつか作ります。まず、もっとも単純な入力 `'()` に対する出力を決めます。このリストには文字列がまったく含まれないので、「Flatt」も含まれません。

```racket
(check-expect (contains-flatt? '()) #false)
```

次に、1項目だけのリストを考えます。次の2つの例があります。

```racket
(check-expect (contains-flatt? (cons "Find" '()))
              #false)
(check-expect (contains-flatt? (cons "Flatt" '()))
              #true)
```

最初の場合、リスト上の唯一の項目は `"Flatt"` ではないので答えは `#false` です。2番目の場合、唯一の項目は `"Flatt"` なので答えは `#true` です。最後に、より一般的な例を示します。

```racket
(check-expect
  (contains-flatt?
    (cons "A" (cons "Flatt" (cons "C" '()))))
  #true)
```

この場合も、リストに `"Flatt"` が含まれるので答えは `#true` でなければなりません。立ち止まって、答えが `#false` でなければならない一般的な例を作れ。

一息ついてください。プログラムを実行してください。ヘッダーは関数の「ダミー」定義です。例があり、それらはテストに変えられており、何よりそのうちいくつかは実際に成功します。間違った理由で成功していますが、成功は成功です。ここまで納得できたら、読み進めてください。

第4のステップは、データ定義に対応する関数テンプレートを設計することです。文字列のリストのデータ定義には2つの節があるので、関数本体は2つの節を持つ `cond` 式でなければなりません。2つの条件は、関数が受け取ったのが2種類のリストのうちどちらであるかを判定します。

```racket
(define (contains-flatt? alon)
  (cond
    [(empty? alon)...]
    [(cons? alon)...]))
```

第2節では `(cons? alon)` の代わりに `else` を使うこともできます。

`cond` 式の各節を順に調べることで、テンプレートにもう1つヒントを加えられます。具体的には、設計レシピは、対応する入力クラスが複合データからなる場合、各節にセレクタ式を注記するよう勧めています。この場合、`'()` は複合データを持たないので、構成要素はありません。そうでなければ、リストは文字列ともう1つの文字列リストから構築されているので、`(first alon)` と `(rest alon)` をテンプレートに加えて、この事実を思い出します。

```racket
(define (contains-flatt? alon)
  (cond
    [(empty? alon)...]
    [(cons? alon)
     (... (first alon)... (rest alon)...)]))
```

いよいよ本格的なプログラミングの仕事、設計レシピの第5ステップです。テンプレートから始め、各 `cond` 節を別々に扱います。`(empty? alon)` が真なら、入力は空リストであり、関数は結果 `#false` を生成しなければなりません。2番目の場合、`(cons? alon)` が真です。テンプレートの注記は、先頭の文字列と、リストの残りがあることを思い出させてくれます。そこで、この範疇に入る例を考えてみましょう。

```racket
(cons "A"
  (cons...
... '()))
```

関数は、人と同じように、最初の項目を `"Flatt"` と比較しなければなりません。この例では、最初のものは `"A"` であり `"Flatt"` ではないので、比較は `#false` になります。代わりに別の例、例えば

```racket
(cons "Flatt"
  (cons...
... '()))
```

を考えていたら、関数は入力の最初の項目が `"Flatt"` であると判定し、したがって `#true` で応答するでしょう。これは、`cond` 式の2行目に、リスト上の最初の名前を `"Flatt"` と比較する式を含めるべきことを意味します。

```racket
(define (contains-flatt? alon)
  (cond
    [(empty? alon) #false]
    [(cons? alon)
     (... (string=? (first alon) "Flatt")
... (rest alon)...)]))
```

さらに、比較が `#true` なら、関数は `#true` を生成しなければなりません。比較が `#false` なら、残るのは別の文字列リスト `(rest alon)` です。明らかに、この場合関数は最終的な答えを知り得ません。答えは「…」が何を表すかに依存するからです。言い換えれば、最初の項目が `"Flatt"` でないなら、リストの残りに `"Flatt"` が含まれるかどうかを調べる何らかの方法が必要です。

幸い、`contains-flatt?` があり、それはその役割にぴったりです。目的ステートメントによれば、それはリストに `"Flatt"` が含まれるかどうかを判定します。そのステートメントは、`(contains-flatt? l)` が文字列のリスト `l` に `"Flatt"` が含まれるかどうかを教えてくれることを意味します。同様に、`(contains-flatt? (rest alon))` は、`"Flatt"` が `(rest alon)` のメンバーかどうかを判定し、それはまさに私たちが知る必要があることです。

要するに、最後の行は `(contains-flatt? (rest alon))` であるべきです。

```racket
; List-of-names -> Boolean
(define (contains-flatt? alon)
  (cond
    [(empty? alon) #false]
    [(cons? alon)
     (... (string=? (first alon) "Flatt")...
... (contains-flatt? (rest alon))...)]))
```

いまの技巧は、2つの式の値を適切な仕方で組み合わせることです。前述のとおり、最初の式が `#true` なら、リストの残りを探す必要はありません。しかしそれが `#false` なら、2番目の式がなお `#true` になるかもしれません。つまり、名前 `"Flatt"` がリストの残りにあるということです。これらすべてから、`(contains-flatt? alon)` の結果は、最後の行の最初の式または2番目の式が `#true` を生成するなら `#true` である、ということが示唆されます。

```
+--------------------------------------------------------------+
| Figure 47: Searching a list                                  |
|                                                              |
|; List-of-names -> Boolean                                   |
|; determines whether "Flatt" occurs on alon                  |
| (check-expect                                                |
|   (contains-flatt? (cons "X" (cons "Y"  (cons "Z" '()))))    |
|   #false)                                                    |
| (check-expect                                                |
|   (contains-flatt? (cons "A" (cons "Flatt" (cons "C" '())))) |
|   #true)                                                     |
| (define (contains-flatt? alon)                               |
|   (cond                                                      |
|     [(empty? alon) #false]                                   |
|     [(cons? alon)                                            |
|      (or (string=? (first alon) "Flatt")                     |
|          (contains-flatt? (rest alon)))]))                   |
+--------------------------------------------------------------+
```

図47は完成した定義を示しています。全体として、本の最初の章の定義とあまり違って見えません。シグネチャ、目的ステートメント、2つの例、そして定義からなります。この関数定義がこれまでに見たものと違う唯一の点は、自己参照——つまり、`define` の本体の中での `contains-flatt?` への参照——です。とはいえ、データ定義も自己参照的なので、ある意味では関数の自己参照もあまり驚くことではありません。

## 8.4 リストの計算（Computing with Lists）

私たちはまだ BSL を使っているので、代数の規則——インターメッツォ1を参照——が、次のような式の値をどう決定するかを教えてくれます。

```racket
(contains-flatt? (cons "Flatt" (cons "C" '())))
```

DrRacket なしでも、です。プログラマはこの種の計算がどう働くかについて直感的な理解を持たなければならないので、この単純な例について一歩ずつ見ていきます。

図48は最初のステップを示しており、適用の値を決めるためにいつもの置換規則を使います。結果は条件式です。代数の先生なら言うように、関数が段階的に定義されているからです。

```
+-----------------------------------------------------------------+
| Figure 49: Computing with lists, step 2                         |
|                                                                 |
|...                                                             |
| ==                                                              |
| (cond                                                           |
|  [#false #false]                                                |
|  [(cons? (cons "Flatt" (cons "C" '())))                         |
|   (or (string=? (first (cons "Flatt" (cons "C" '()))) "Flatt")  |
|       (contains-flatt? (rest (cons "Flatt" (cons "C" '())))))]) |
| ==                                                              |
| (cond                                                           |
|  [(cons? (cons "Flatt" (cons "C" '())))                         |
|   (or (string=? (first (cons "Flatt" (cons "C" '()))) "Flatt")  |
|       (contains-flatt? (rest (cons "Flatt" (cons "C" '())))))]) |
| ==                                                              |
| (cond                                                           |
|  [#true                                                         |
|   (or (string=? (first (cons "Flatt" (cons "C" '()))) "Flatt")  |
|       (contains-flatt? (rest (cons "Flatt" (cons "C" '())))))]) |
| ==                                                              |
| (or (string=? (first (cons "Flatt" (cons "C" '()))) "Flatt")    |
|     (contains-flatt? (rest (cons "Flatt" (cons "C" '())))))     |
+-----------------------------------------------------------------+
```

計算は図49で続きます。`cond` 式の正しい節を見つけるには、条件の値を1つずつ決めなければなりません。`cons` されたリストは空ではないので、最初の条件の結果は `#false` であり、したがって最初の `cond` 節を取り除きます。最後に、第2節の条件は `#true` に評価されます。`cons` されたリストに対する `cons?` が成り立つからです。

```
+-------------------------------------------------------------+
| Figure 50: Computing with lists, step 3                     |
|                                                             |
|...                                                         |
| ==                                                          |
| (or (string=? "Flatt" "Flatt")                              |
|     (contains-flatt? (rest (cons "Flatt" (cons "C" '()))))) |
| == (or #true (contains-flatt?...))                         |
| == #true                                                    |
+-------------------------------------------------------------+
```

ここから最終結果を得るまで、あとは3ステップの算術です。図50はその3ステップを示しています。最初のステップは、`first` の法則により `(first (cons "Flatt" ...))` を `"Flatt"` に評価します。2番目は、`"Flatt"` が文字列であり `"Flatt"` に等しいことを発見します。3番目は、`X` が何であれ `(or #true X)` が `#true` だと述べます。
