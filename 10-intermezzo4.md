数に関して、プログラミング言語は、下層のハードウェアと真の数学とのあいだの隔たりを仲介する。典型的なコンピュータのハードウェアは、固定サイズのデータ片で数を表す。また、そうしたデータ片だけを扱うプロセッサも備わっている。紙と鉛筆での計算では、何桁を処理するかは気にしない。原理的には、1桁の数、10桁の数、あるいは10,000桁の数でも扱える。したがって、プログラミング言語が下層のハードウェアの数を使うなら、その計算は可能な限り効率的になる。一方、数学から知る数に固執するなら、それらをハードウェアデータの片に変換し、また戻さなければならず——この変換には時間がかかる。このコストのため、ほとんどのプログラミング言語の設計者は、ハードウェアに基づく選択を採る。

> **注:** これらの
> データ片はビット、バイト、ワードと呼ばれる。

このインターメッツォは、数のハードウェア表現を、データ表現の演習として説明する。具体的には、第1節が数のための具体的な固定サイズのデータ表現を導入し、数をこの表現へ写像する方法を論じ、そうした数の上で計算がどう働くかを示唆する。第2節と第3節は、この選択の2つの最も根本的な問題——それぞれ算術オーバーフローとアンダーフロー——を示す。最後の節は、教育用言語における算術がどう働くかを概説する。その数体系は、今日のほとんどのプログラミング言語に見られるものを一般化したものである。最後の演習は、プログラムが数で計算するときに事態がどれほど悪くなり得るかを示す。

## 固定サイズ数の算術（Fixed-Size Number Arithmetic）

4桁を使って数を表せるとしよう。自然数を表すなら、表現可能な範囲の1つは [0,10000) である。実数については、0と1のあいだの10,000個の分数を選んでもよいし、0と1のあいだの5,000個ともう5,000個を1と2のあいだ、などと選んでもよい。いずれの場合も、4桁は選んだある区間について最大で10,000個の数を表せるにすぎない。一方、この区間の数直線には無限個の数が含まれる。

ハードウェアの数に対する一般的な選択は、いわゆる科学的記数法を使うことである。つまり、数は2つの部分で表される：

2. 指数（exponent）。10を底とする因子を決めるのに使われる。

```
+--------------------------------------------------------------+
| Figure 143: Functions for inexact representations            |
|                                                              |
|; N Number N -> Inex                                         |
|; makes an instance of Inex after checking the arguments     |
| (define (create-inex m s e)                                  |
|   (cond                                                      |
|     [(and (<= 0 m 99) (<= 0 e 99) (or (= s 1) (= s -1)))     |

|     [else (error "bad values given")]))                      |
|                                                              |
|; Inex -> Number                                             |
|; converts an inex into its numeric equivalent               |
| (define (inex->number an-inex)                               |
|   (* (inex-mantissa an-inex)                                 |
|      (expt                                                   |
|        10 (* (inex-sign an-inex) (inex-exponent an-inex))))) |
+--------------------------------------------------------------+
```

問題を理解するには、これらの選択を ISL+ でのデータ表現として具体化し、いくつか実験するのが最善である。固定サイズの数を、3つのフィールドをもつ構造体で表そう：

```racket
(define-struct inex [mantissa sign exponent])
; An Inex is a structure:
;   (make-inex N99 S N99)
; An S is one of:
; – 1
; – -1
; An N99 is an N between 0 and 99 (inclusive).
```

Inex のフィールドに対する条件は非常に厳しいので、この構造体型定義を具体化する関数 `create-inex` を定義する。図143を見よ。同図はまた、上記の式を使って Inex を数に変える `inex->number` も定義している。

上記の例1200を、われわれのデータ表現に翻訳してみよう：

```racket
(create-inex 12 1 2)
```

```racket

```

```racket

(make-inex 50 -1 20)
> (create-inex 5 -1 19)
(make-inex 5 -1 19)
```

`inex->number` を使って、これら2つの数が等価であることを確認せよ。

`create-inex` を使えば、表現可能な数の範囲を区切ることも容易である。多くの応用にとっては実際かなり狭い：

```racket

(define MIN-POSITIVE (create-inex 1 -1 99))
```

## オーバーフロー（Overflow）

科学的記数法は、固定サイズのデータ片で表せる数の範囲を広げるが、それでも有限である。固定サイズの数表現には収まらないほど大きすぎる数がある。例えば、

> [image: pict_173.png]

は表現できない。指数500が2桁に収まらず、仮数は法的に可能な限り大きいからである。

Inex にとって大きすぎる数は、計算の途中で生じ得る。例えば、表現できる2つの数が、表現できない数に足し合わさることがある：

```racket
(inex+ (create-inex 50 1 99) (create-inex 50 1 99))
==
(create-inex 100 1 99)
```

これはデータ定義に違反する。Inex 算術が表現するには大きすぎる数を生成するとき、（算術）オーバーフローと言う。

オーバーフローが起きると、言語実装によってはエラーを通知して計算を止める。別の実装は、そうした数を表すために無限大（infinity）と呼ばれる記号的な値を割り当て、算術演算を通じてそれを伝播させる。

**注** Inex が仮数の符号フィールドをもっていたなら、2つの負の数が足し合わさって、表現できないほど負になることがある。これを負方向のオーバーフローと呼ぶ。終わり。

**演習 415. ISL+ はオーバーフローの扱いに `+inf.0` を使う。整数 n を求めよ。ただし**

```racket
(expt #i10.0 n)
```

が不正確数であり、一方 `(expt #i10. (+ n 1))` が `+inf.0` で近似されるような n である。ヒント：n を計算する関数を設計せよ。

## アンダーフロー（Underflow）

スペクトルの反対端には、Inex に表現をもたない小さな数がある。例えば [image: pict_174.png] は0ではないが、表現できる最小の非ゼロ数より小さい。（算術）アンダーフローは、2つの小さな数を掛けて結果が Inex には小さすぎるときに生じる：

```racket
(inex* (create-inex 1 -1 10) (create-inex 1 -1 99))
==
(create-inex 1 -1 109)
```

これはエラーを通知する。

アンダーフローが起きると、言語実装によってはエラーを通知し、別の実装は結果の近似に0を使う。アンダーフローの近似に0を使うことは、Inex における数の近似表現を選ぶこととは質的に異なる。具体的には、1250を `(create-inex1212)` で近似すると、仮数から有効桁が落ちるが、結果はつねに表すべき数の10%以内にある。しかしアンダーフローを近似することは、仮数全体を落とすことを意味し、結果は真の結果の予測可能な百分率の範囲内にはない。

**演習 416.** ISL+ はアンダーフローの近似に `#i0.0` を使う。`(expt #i10.0 n)` がいまだ不正確な ISL+ の数であり、`(expt #i10. (- n 1))` が0で近似されるような最小の整数 n を求めよ。ヒント：n を計算する関数を使え。この関数と演習415の解の上で抽象化することを検討せよ。

## *SL の数（*SL Numbers）

ほとんどのプログラミング言語は、数の近似表現と数に対する算術だけを支援する。典型的な言語は、整数を、それが動くハードウェアのデータ片のサイズに関連する区間に制限する。実数の表現は、前節までの概略にゆるく基づくが、Inex が使う4桁より大きなデータ片と、2進数体系の桁を使う。

> **注:** 不正確な実数表現にはいくつかの
> 種類がある：float、double、extflonum、など。

教育用言語は正確数と不正確数の両方を支援する。その整数と有理数は任意に大きく精密であり、制限はコンピュータ全体のメモリの絶対的な大きさだけである。これらの数に対する計算では、関与する有理数が支援されるデータ片に収まるかぎり下層のハードウェアを使い、この区間の外の数については自動的に別の表現と、算術演算の別版に切り替える。その実数には2種類ある：正確と不正確である。正確数は実数を真に表し、不正確数は前節までの精神で実数を近似する。算術演算は可能なときは正確さを保ち、必要なときは不正確な結果を生成する。したがって `sqrt` は、2の正確表現と不正確表現の両方に対して不正確数を返す。対照的に、`sqrt` は正確な4が与えられると正確な2を生成し、入力が `#i4.0` のときは `#i2.0` を生成する。最後に、教育用プログラム中の数値定数は、`#i` が前置されないかぎり正確な有理数と理解される。

素の Racket は、すべての十進数を不正確数として解釈する。また、正確か不正確かにかかわらず、すべての実数を十進として表示する。含意は、そうした数はすべて危険だということである。真の数の不正確な近似である可能性が高いからである。プログラマは、ドット付きの数を正確として解釈するよう、数値定数に `#e` を前置することで Racket に強制できる。

この時点で、プログラムがこれらの不正確数を使う場合、結果が真の結果からどれだけずれるか疑問に思うかもしれない。この問いは、初期の計算機科学者が大いに苦心した問いであり、過去数十年にわたってこれらの研究は、数値解析（numerical analysis）と呼ばれる独立した分野を生み出した。すべての計算機科学者、そして実際、コンピュータとソフトウェアを使うすべての人は、その存在と、数値プログラムの働きに関するいくつかの基本的な洞察を意識すべきである。最初の味わいとして、次の演習は事態がどれほど悪くなり得るかを示す。これらに取り組み、不正確数の問題を決して見失わないようにせよ。

不正確な計算に関する一般的な助言は、加算を最も小さな数から始めよとプログラマに言う。大きな数を2つの小さな数に足すと大きな数だけが得られるかもしれないが、小さな数を先に足すと大きな数ができ、結果が変わるかもしれない：

```racket
#i9007199254740992.0
> (sum (list #i1.0 #i1.0 (expt 2 #i53.0)))
#i9007199254740994.0
```

この技巧は効かないかもしれない。上記の JANUS の対話を見よ。

ISL+ のような言語では、数を正確な有理数に変換し、それらに正確な算術を使い、結果を戻すことができる：

```racket
(exact->inexact (sum (map inexact->exact JANUS)))
```

この式を評価し、結果を上の3つの和と比較せよ。いまウェブからの助言についてどう思うか。

**演習 420. JANUS は固定のリストにすぎないが、次の関数を見よ：**

```racket
(define (oscillate n)
  (local ((define (O i)
            (cond
              [(> i n) '()]
              [else
               (cons (expt #i-0.99 i) (O (+ i 1)))])))
    (O 1)))
```

`oscillate` を自然数 n に適用すると、数学的な級数の最初の n 要素が生成される。これは図145のようなグラフとして理解するのが最善である。DrRacket で `(oscillate15)` を実行し、結果を検査せよ。

```

```

その結果を左から右へ足すと、右から左へ足すのと異なる結果が計算される：

```racket
> (sum (oscillate #i1000.0))
#i-0.49746596003269394
> (sum (reverse (oscillate #i1000.0)))
#i-0.4974659600326953
```

ふたたび、差は小さく見えるかもしれないが、文脈を見ると：

```racket
> (- (* 1e+16 (sum (oscillate #i1000.0)))     (* 1e+16 (sum (reverse (oscillate #i1000.0)))))
#i14.0
```

この差は問題になり得るか。コンピュータを信頼できるか。

問いは、選択が与えられたとき、プログラマがプログラムでどの数を使うべきかである。答えはもちろん文脈による。財務諸表の世界では、数値定数は正確数として解釈されるべきであり、財務諸表の計算操作は、数学的演算の正確さ保持の性質に頼れるべきである。結局のところ、法は不正確数とその演算がもたらす深刻な誤差を容認できない。しかし科学計算では、正確な結果を出すために必要な余分な時間が過大な負担になるかもしれない。科学者はしたがって不正確数を使う傾向があるが、プログラムの出力の用途に対して数値誤差が許容できることを確かめるため、プログラムを注意深く解析する。
