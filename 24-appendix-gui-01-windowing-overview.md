## 1 ウィンドウ機構

ウィンドウ機構ツールボックスは、GUI プログラムの基本構成要素を提供します。フレーム（トップレベル・ウィンドウ）、モーダル・ダイアログ、メニュー、ボタン、チェックボックス、テキストフィールド、ラジオボタンなどが、いずれもクラスとして用意されています。

> **注:** Racket におけるクラスとインタフェースの入門については、「Classes and Objects」を参照してください。

### 1.1 ウィンドウの作成

新しいトップレベル・ウィンドウを作るには、`frame%` クラスをインスタンス化します。

```racket
; Make a frame by instantiating the frame% class
(define frame (new frame% [label "Example"]))

; Show the frame by calling its show method
(send frame show #t)
```

組み込みクラスは、GUI イベントを扱うためのさまざまな仕組みを提供します。たとえば `button%` クラスをインスタンス化するとき、ユーザがボタンをクリックしたときに呼ばれるイベント・コールバック手続きを渡せます。次の例プログラムは、テキスト・メッセージとボタンを持つフレームを作ります。ユーザがボタンをクリックすると、メッセージが変わります。

```racket
; Make a frame by instantiating the frame% class
(define frame (new frame% [label "Example"]))

; Make a static text message in the frame
(define msg (new message% [parent frame]
                          [label "No events so far..."]))

; Make a button in the frame
(new button% [parent frame]
             [label "Click Me"]
; Callback procedure for a button click:
             [callback (lambda (button event)
                         (send msg set-label "Button click"))])

; Show the frame by calling its show method
(send frame show #t)
```

プログラマが GUI のイベント・ループを直接実装することはありません。その代わり、ウィンドウ機構システムが内部キューから各イベントを自動的に取り出し、適切なウィンドウへディスパッチします。ディスパッチは、ウィンドウのコールバック手続きを呼び出すか、ウィンドウのメソッドの一つを呼び出します。上のプログラムでは、ユーザが「Click Me」をクリックするたびに、ウィンドウ機構システムが自動的にボタンのコールバック手続きを呼び出します。

ウィンドウが複数種類のイベントを受け取る場合、イベントはコールバック手続きではなく、ウィンドウのクラスのメソッドへディスパッチされます。たとえば描画キャンバスは、更新イベント、マウス・イベント、キーボード・イベント、サイズ変更イベントを受け取ります。これらを扱うには、組み込みの `canvas%` クラスから新しいクラスを派生し、イベント処理メソッドをオーバーライドします。次の式は、上で作ったフレームに、マウスとキーボードのイベントを扱うキャンバスを追加します。

```racket
; Derive a new canvas (a drawing window) class to handle events
(define my-canvas%
  (class canvas%; The base class is canvas%
; Define overriding method to handle mouse events
    (define/override (on-event event)
      (send msg set-label "Canvas mouse"))
; Define overriding method to handle keyboard events
    (define/override (on-char event)
      (send msg set-label "Canvas keyboard"))
; Call the superclass init, passing on all init args
    (super-new)))

; Make a canvas that handles events in the frame
(new my-canvas% [parent frame])
```

上のコードを実行したあと、フレームを手動でリサイズすると、新しいキャンバスが見えます。カーソルをキャンバス上で動かすと、キャンバスの `on-event` メソッドが、移動イベントを表すオブジェクトとともに呼ばれます。キャンバスをクリックすると `on-event` が呼ばれます。キャンバスがキーボード・フォーカスを持っているあいだキーボードを打つと、キャンバスの `on-char` メソッドが呼び出されます。

ウィンドウ機構システムは GUI イベントを逐次的にディスパッチします。つまり、イベント処理のコールバックやメソッドを呼び出したあと、そのハンドラが戻るまで次のイベントをディスパッチしません。イベントの逐次性を示すため、フレームをさらに拡張し、「Pause」ボタンを追加します。

```racket
(new button% [parent frame]
             [label "Pause"]
             [callback (lambda (button event) (sleep 5))])
```

ユーザが「Pause」をクリックすると、フレーム全体が5秒間反応しなくなります。`sleep` の呼び出しが戻るまで、ウィンドウ機構システムはそれ以上のイベントをディスパッチできないからです。イベント・ディスパッチの詳細は、「イベントのディスパッチとイベントスペース」を参照してください。

イベントのディスパッチに加え、GUI クラスはウィンドウのグラフィカルなレイアウトも扱います。例のフレームは単純なレイアウトを示しています。フレームの要素は上から下へ並びます。一般にプログラマは、各 GUI 要素を親コンテナに割り当てることでウィンドウのレイアウトを指定します。フレームのような縦方向コンテナは子を列に並べ、横方向コンテナは子を行に並べます。コンテナは別のコンテナの子にもなれます。たとえば、フレーム内に二つのボタンを横に並べるには、新しいボタン用の横方向パネルを作ります。

```racket
(define panel (new horizontal-panel% [parent frame]))
(new button% [parent panel]
             [label "Left"]
             [callback (lambda (button event)
                         (send msg set-label "Left click"))])
(new button% [parent panel]
             [label "Right"]
             [callback (lambda (button event)
                         (send msg set-label "Right click"))])
```

ウィンドウ・レイアウトとコンテナの詳細は、「ジオメトリ管理」を参照してください。

### 1.2 キャンバスへの描画

キャンバスの内容は `on-paint` メソッドによって決まります。既定の `on-paint` は、キャンバス作成時に渡された `paint-callback` 関数を呼び出します。`on-paint` メソッドは引数を取らず、キャンバスの `get-dc` メソッドで描画コンテキスト（DC）を得て描画に使います。既定の `on-paint` は、キャンバスとこの DC を `paint-callback` 関数に渡します。DC に対する `racket/draw` ツールボックスの描画操作は、画面上のキャンバスの内容に反映されます。

たとえば次のプログラムは、大きく親しみやすい文字を表示するキャンバスを作ります。

```racket
(define frame (new frame%
                   [label "Example"]
                   [width 300]
                   [height 300]))
(new canvas% [parent frame]
             [paint-callback
              (lambda (canvas dc)
                (send dc set-scale 3 3)
                (send dc set-text-foreground "blue")
                (send dc draw-text "Don't Panic!" 0 0))])
(send frame show #t)
```

キャンバスの背景色は `set-canvas-background` メソッドで設定できます。キャンバスを透明にする（親の色とテクスチャを初期内容として引き継ぐ）には、キャンバス作成時に `style` 引数へ `'transparent` を渡します。

`racket/draw` ライブラリでの描画の概要は、The Racket Drawing Toolkit の Overview を参照してください。キャンバス描画のより高度な情報は、「キャンバスでのアニメーション」を参照してください。

### 1.3 コア・ウィンドウ機構クラス

ウィンドウ機構ツールボックスにおける基本的なグラフィカル要素は、**領域（area）** です。次のクラスが、ウィンドウ機構ツールボックス内のさまざまな種類の領域を実装します。

- **コンテナ（Containers）** — 他の領域を含むことができる領域：
  - `frame%` — フレームは、ユーザが移動・リサイズできるトップレベル・ウィンドウです。
  - `dialog%` — ダイアログはモーダルなトップレベル・ウィンドウです。ダイアログが表示されているあいだ、それが閉じられるまで他のトップレベル・ウィンドウは無効化されます。
  - `panel%` — パネルはコンテナ内のサブコンテナです。ツールボックスは `panel%` の三つのサブクラス、`vertical-panel%`、`horizontal-panel%`、`tab-panel%` を提供します。
  - `pane%` — ペインは軽量なパネルです。グラフィカルな表現やイベント処理能力は持ちません。`pane%` クラスには三つのサブクラス、`vertical-pane%`、`horizontal-pane%`、`grow-box-spacer-pane%` があります。
- **被包含要素（Containees）** — 他の領域の中に含まれなければならない領域：
  - `panel%` — パネルは被包含要素であると同時にコンテナでもあります。
  - `pane%` — ペインは被包含要素であると同時にコンテナでもあります。
  - `canvas%` — キャンバスは画面に描画するためのサブウィンドウです。
  - `editor-canvas%` — エディタ・キャンバスは、テキスト・エディタまたはペーストボード・エディタを表示するためのサブウィンドウです。`editor-canvas%` クラスは、「Editors」のエディタ・クラスとともに文書化されています。
  - **コントロール（Controls）** — ユーザが操作できる被包含要素：
    - `message%` — メッセージは、ユーザ操作のない静的テキスト・フィールドまたはビットマップです。
    - `button%` — ボタンはクリック可能なコントロールです。
    - `check-box%` — チェックボックスはクリック可能なコントロールで、ユーザがクリックしてチェック印を付けたり外したりします。
    - `radio-box%` — ラジオボックスは、互いに排他的なラジオボタンの集まりです。ユーザがあるラジオボタンをクリックするとそれが選択され、以前に選択されていたラジオボタンの選択が解除されます。
    - `choice%` — チョイス項目は、テキスト選択肢のポップアップ・メニューです。ユーザはコントロール内の項目を一つ選びます。
    - `list-box%` — リストボックスは、スクロール可能なテキスト選択肢のリストです。ユーザはリスト内の一つ以上の項目を選びます（リストボックスのスタイルによる）。
    - `text-field%` — テキストフィールドは、単純なテキスト入力用のボックスです。
    - `combo-field%` — コンボフィールドは、テキストフィールドと選択肢のポップアップ・メニューを組み合わせたものです。
    - `slider%` — スライダは、固定範囲内の整数値を選ぶドラッグ可能なコントロールです。
    - `gauge%` — ゲージは、固定範囲内の整数値を報告するための出力専用コントロールです（ユーザは値を変更できません）。

上の一覧が示すように、**コンテナ** と呼ばれる領域は、**被包含要素** と呼ばれる他の領域を管理します。パネルのように、コンテナと被包含要素の両方である領域もあります。

ほとんどの領域は **ウィンドウ** ですが、**非ウィンドウ** もあります。パネルのようなウィンドウはグラフィカルな表現を持ち、キーボードとマウスのイベントを受け取り、無効化や非表示にできます。対照的に、ペインのような非ウィンドウはジオメトリ管理にのみ役立ちます。非ウィンドウはマウス・イベントを受け取らず、無効化や非表示にもできません。

すべての領域は `area<%>` インタフェースのインスタンスです。各コンテナはさらに `area-container<%>` インタフェースのインスタンスでもあり、各被包含要素は `subarea<%>` のインスタンスです。ウィンドウは `window<%>` のインスタンスです。`area-container<%>`、`subarea<%>`、`window<%>` インタフェースは、いずれも `area<%>` のサブインタフェースです。

次の図は、`area<%>` の下の型階層の一部を示します。

メニューバー、メニュー、メニュー項目はグラフィカル要素ですが、領域ではありません（つまり、調整可能なグラフィカル・サイズなど、領域に共通する性質をすべて持つわけではありません）。その代わり、メニュー・クラスは別のコンテナ–被包含要素の階層を形成します。

- **メニュー項目コンテナ（Menu Item Containers）**
  - `menu-bar%` — メニューバーは、フレームに関連付けられたメニューのトップレベル集合です。
  - `menu%` — メニューはメニュー項目の集合を含みます。メニューはメニューバー内、ポップアップ・メニュー内、または別のメニュー内のサブメニューとして現れます。
  - `popup-menu%` — ポップアップ・メニューは、キャンバスやエディタ・キャンバスに動的に表示されるトップレベル・メニューです。
- **メニュー項目（Menu Items）**
  - `separator-menu-item%` — セパレータは、メニューまたはポップアップ・メニュー内の選択できない線です。
  - `menu-item%` — 通常のメニュー項目は、メニュー内の選択可能なテキスト項目です。項目が選ばれると、そのコールバック手続きが呼び出されます。
  - `checkable-menu-item%` — チェック可能なメニュー項目はメニュー内のテキスト項目で、ユーザがそれを選ぶと項目の隣のチェック印が切り替わります。
  - `menu%` — メニューは、メニュー項目であると同時にメニュー項目コンテナでもあります。

### 1.4 ジオメトリ管理

ウィンドウ機構ツールボックスのジオメトリ管理により、GUI 要素のグラフィカル表現がプラットフォームごとに異なっても、どのプラットフォームでも正しく見えるウィンドウを設計しやすくなります。ジオメトリ管理はコンテナを基盤とし、各コンテナは、フレームの現在のサイズやボタンの自然なサイズといった単純な制約に基づいて子を配置します。

組み込みのコンテナ・クラスには、子を行に揃える横方向パネル（およびペイン）と、子を列に揃える縦方向パネル（およびペイン）があります。横方向と縦方向のコンテナを入れ子にすることで、ほとんどのレイアウトを実現できます。たとえば、次のような形のダイアログを構成するには、

```
+--------------------------------------------------+
| `----------------------------------------------… |
+--------------------------------------------------+
| `|` `-------------------------------------` `|`  |
| `|` `Your name: |` `|` `|`                       |
| `|` `-------------------------------------` `|`  |
| `|` `--------` `----` `|`                        |
| `|` `( Cancel )` `( OK )` `|`                    |
| `|` `--------` `----` `|`                        |
| `----------------------------------------------… |
+--------------------------------------------------+
```

次のプログラムを使います。

```racket
; Create a dialog
(define dialog (instantiate dialog% ("Example")))

; Add a text field to the dialog
(new text-field% [parent dialog] [label "Your name"])

; Add a horizontal panel to the dialog, with centering for buttons
(define panel (new horizontal-panel% [parent dialog]
                                     [alignment '(center center)]))

; Add Cancel and Ok buttons to the horizontal panel
(new button% [parent panel] [label "Cancel"])
(new button% [parent panel] [label "Ok"])
(when (system-position-ok-before-cancel?)
  (send panel change-children reverse))

; Show the dialog
(send dialog show #t)
```

各コンテナは、各子の自然なサイズを使って子を配置します。自然なサイズは通常、ボタンのラベルやラジオボックスの選択肢の数など、子のインスタンス化パラメータに依存します。上の例では、ダイアログはテキストフィールドの最小幅に合わせて横に伸び、フィールドとボタンの合計の高さに合わせて縦に伸びます。次にダイアログは横方向パネルを伸ばしてダイアログ下半分を埋めます。最後に横方向パネルは、ボタンの最小幅の合計を使ってボタンを水平方向に中央揃えします。

この例が示すように、伸縮可能なコンテナは環境を埋めるように拡大し、余分な空間を伸縮可能な子のあいだに分配します。既定では、パネルは両方向に伸縮可能で、ボタンはどちらの方向にも伸縮しません。プログラマは、個々の GUI 要素が伸縮可能かどうかを変更できます。

次の各小節ではコンテナ・システムを詳しく説明します。まず「被包含要素」で被包含要素の属性を論じ、次に「コンテナ」でコンテナの属性を述べます。組み込みの縦・横方向コンテナに加え、プログラマは最終小節「新しい種類のコンテナの定義」で論じるように、新しい種類のコンテナを定義できます。

#### 1.4.1 被包含要素（Containees）

各被包含要素、すなわち子は、次の性質を持ちます。

- グラフィカル最小幅とグラフィカル最小高さ
- 要求最小幅と要求最小高さ
- 水平および垂直の伸縮可能性（オンまたはオフ）
- 水平および垂直のマージン

コンテナは、各被包含要素のこれら四つの性質に基づいて子を配置します。被包含要素の親コンテナは、被包含要素の作成時に指定されます。ウィンドウ被包含要素は親の中で非表示にしたり削除したりでき、再ペアレント（reparenting）によって親を変更することもできます。

特定の被包含要素のグラフィカル最小サイズは、`get-graphical-min-size` が報告するもので、プラットフォーム、被包含要素のラベル（コントロールの場合）、および作成時に指定されたスタイル属性に依存します。たとえばボタンの最小グラフィカル・サイズは、ラベルのテキスト全体が見えることを保証します。コントロール（ボタンなど）のグラフィカル最小サイズは変更できません。作成時に固定されます。（コントロールの最小サイズは、ラベルが変わっても再計算されません。）パネルやペインのグラフィカル最小サイズは、子の最小サイズの合計と、それらの配置の仕方に依存します。

被包含要素のサイズを選ぶとき、親コンテナはグラフィカル最小サイズではなく、被包含要素の**要求最小サイズ**を考慮します（要求最小がグラフィカル最小より大きいと仮定して）。グラフィカル最小と異なり、被包含要素の要求最小サイズは、`min-width` と `min-height` メソッドを使ってプログラマがいつでも変更できます。

被包含要素が（特定の方向に）伸縮可能でない限り、それは常に（対応する方向の）最小サイズに縮みます。そうでなければ、被包含要素はコンテナ内の利用可能な空間を埋めるように伸ばされます。各被包含要素は既定の伸縮可能性から始まります。たとえばボタンは最初は伸縮可能ではなく、一行テキストフィールドは最初は水平方向に伸縮可能です。プログラマは `stretchable-width` と `stretchable-height` メソッドで、被包含要素の伸縮可能性をいつでも変更できます。

**マージン** は被包含要素を囲む空間です。各被包含要素のマージンは最小サイズとは独立ですが、コンテナの観点からは、マージンは被包含要素の最小サイズを実質的に増やします。たとえばボタンの垂直マージンが 2 なら、コンテナはボタンの最小高さ分の空間に加え、ボタンの上下に2ピクセルの余白を残すだけの空間を割り当てなければなりません。プログラマは `horiz-margin` と `vert-margin` で被包含要素のマージンを調整できます。既定のマージンは、コントロールでは 2、それ以外の種類の被包含要素では 0 です。

実際には、コントロールの要求最小サイズとマージンが変更されることはまれですが、キャンバスではしばしば変更されます。伸縮可能性は、プログラマが望む視覚効果に応じて、どの種類の被包含要素でもよく調整されます。

#### 1.4.2 コンテナ（Containers）

コンテナは次の性質を持ちます。

- （削除されていない）子被包含要素のリスト
- 要求最小幅と要求最小高さ
- 子のあいだに使う間隔（spacing）
- 子の集合全体の周囲に使う境界マージン
- 水平および垂直の伸縮可能性（オンまたはオフ）
- 余った空間の位置決めのための揃え設定

これらの性質は、コンテナ自身のサイズ計算と子の配置に組み込まれます。被包含要素でもあるコンテナ（例：パネル）では、コンテナの要求最小サイズと伸縮可能性は、被包含要素としてのそれと同じです。

被包含要素の親コンテナは、被包含要素の作成時に指定されます。被包含要素ウィンドウは親コンテナ内で非表示にしたり削除したりでき、再ペアレントによって親を変更することもできます（ただし非ウィンドウの被包含要素は、非表示・削除・再ペアレントできません）。

- **非表示の子** はユーザには見えませんが、コンテナ内では各非表示の子のために空間が割り当てられたままです。子を非表示または表示するには、子の `show` メソッドを呼びます。
- **削除された子** は非表示であり、コンテナが他の子を配置するときに無視されるため、削除された子のためにコンテナ内に空間は予約されません。子を削除状態または非削除状態にするには、コンテナの `delete-child` または `add-child` メソッドを呼びます（これらは子の `show` メソッドを呼び出します）。
- ウィンドウ被包含要素を再ペアレントするには、`reparent` メソッドを使います。ウィンドウは新しい親の中でも、非表示または削除の状態を保持します。

子が作られたとき、それは最初は表示され、非削除です。削除された子は、その子への外部参照がなくなればガベージ・コレクションの対象になります。非削除の子（非表示かどうかにかかわらず）のリストは、コンテナの `get-children` メソッドから得られます。

コンテナの非削除リスト内の子の順序は重要です。たとえば縦方向パネルは、リストの最初の子をパネルの上端に置き、以下同様です。新しい子が作られると、それはコンテナの子リストの末尾に置かれます。コンテナのリストの順序は、`change-children` メソッドで動的に変更できます。（`change-children` メソッドは、子の有効化・無効化にも使えます。）

`get-graphical-min-size` が報告するコンテナのグラフィカル最小サイズは、子の最小サイズを（コンテナのレイアウト戦略に応じて合計または最大を取るなどして）組み合わせ、さらにコンテナの間隔と境界マージンを加えて計算されます。プログラマは `min-width` と `min-height` メソッドでより大きな最小を指定できます。コンテナの計算上の最小がプログラマ指定の最小より大きいときは、プログラマ指定の最小は無視されます。

コンテナの**間隔**は、子のマージンが要求する空間に加えて、隣接する子のあいだに残す空間の量を決めます。コンテナの**境界マージン**は、子の集合の周囲に追加する空間の量を決め、子を置けるコンテナ内の領域を実質的に狭めます。プログラマは `border` と `spacing` メソッドで、コンテナの境界と間隔を動的に調整できます。すべてのコンテナ型の既定の境界と間隔は 0 です。

パネルやペインは被包含要素であると同時にコンテナでもあるため、境界マージンに加えて被包含要素マージンを持ちます。パネルでは、これら二つのマージンは冗長ではありません。パネルはグラフィカルな境界を持てるからです。境界はパネルの被包含要素マージンの内側、パネルの境界マージンの外側に描かれます。

フレームやダイアログのようなトップレベル・ウィンドウ・コンテナでは、コンテナの伸縮可能性は、ユーザがウィンドウを最小サイズより大きくリサイズできるかどうかを決めます。したがって、伸縮可能でないフレームはユーザがリサイズできません。他の種類のコンテナ（すなわちパネルとペイン）では、コンテナの伸縮可能性は、別のコンテナ内の被包含要素としての伸縮可能性です。すべての種類のコンテナは、最初は両方向に伸縮可能です——`grow-box-spacer-pane%` のインスタンスを除きます。これは有用なコンテナ・クラスというより軽量なスペーサ・クラスとして意図されています——ただしプログラマは、`stretchable-width` と `stretchable-height` メソッドで領域の伸縮可能性をいつでも変更できます。

コンテナの**揃え**指定は、コンテナに余った空間があるとき子をどう位置決めするかを決めます。（コンテナは、その方向に伸縮可能な子が一つもないときだけ、その方向に余った空間を持てます。）たとえば、コンテナの水平揃えが `'left` のとき、子はコンテナ内で左揃えされ、余った空間は右側に集まります。コンテナの水平揃えが `'center` のとき、各子はコンテナ内で水平方向に中央揃えされます。コンテナの揃えは `set-alignment` メソッドで変更します。

#### 1.4.3 新しい種類のコンテナの定義

入れ子の横・縦方向コンテナでほとんどのレイアウト・パターンを表現できますが、プログラマは明示的なレイアウト手続きを持つ新しい種類のコンテナを定義することもできます。プログラマは `panel%` または `pane%` からクラスを派生し、`container-size` と `place-children` メソッドをオーバーライドすることで、新しい種類のコンテナを定義します。`container-size` メソッドは、各子のサイズ仕様のリストを受け取り、コンテナの最小幅と最小高さの二つの値を返します。`place-children` メソッドは、コンテナのサイズと各子のサイズ仕様のリストを受け取り、サイズと配置のリストを返します（元のリストと並行）。

入力のサイズ仕様は、四つの値のリストです。

- 子の最小幅
- 子の最小高さ
- 子の水平伸縮可能性（`#t` は伸縮可能、`#f` は非伸縮）
- 子の垂直伸縮可能性

`place-children` では、出力の位置とサイズの仕様は四つの値のリストです。

- 子の新しい水平位置（親からの相対）
- 子の新しい垂直位置
- 子の新しい実際の幅
- 子の新しい実際の高さ

入力と出力の両方の幅と高さには、子のマージンが含まれます。各子に対して返される位置は、コントロールを置くときに子のマージンを考慮して自動的に加算されます。

### 1.5 マウスとキーボードのイベント

ユーザがマウスを動かす、マウスボタンをクリックまたは離す、あるいはキーボードのキーを押すたびに、あるウィンドウに対してイベントが生成されます。イベントを受け取るウィンドウは、グラフィック表示の現在の状態に依存します。

- **マウス・イベントの受信ウィンドウ** は通常、マウスが動かされたりクリックされたりしたときのカーソル下のウィンドウです。マウスが子ウィンドウの上にある場合、親ではなく子ウィンドウがイベントを受け取ります。

  ユーザがウィンドウ内をクリックすると、そのウィンドウがマウスを「つかみ」、マウスボタンが離されるまで（カーソルの位置にかかわらず）すべてのマウス・イベントがそのウィンドウに行きます。その結果、ユーザはスクロールバーのつまみをクリックして、カーソルをスクロールバー・コントロール内に厳密に保たなくてもドラッグできます。

  マウスのボタン解放イベントは、通常は各ボタン押し下げイベントに対して生成されますが、ボタン解放イベントが落ちることがあります。たとえば、モーダル・ダイアログが現れてマウスを奪うことがあります。より一般に、原理的にはどのようなマウス・イベントも落ちうるので、厳密なマウス・イベント列に依存するアルゴリズムは避けてください。たとえばマウス追跡ハンドラは、ドラッグ以外のイベントを受け取ったときに追跡状態をリセットすべきです。

- **キーボード・イベントの受信ウィンドウ** は、イベント時点でキーボード・フォーカスを所有するウィンドウです。フォーカスを所有するウィンドウは常に一つだけで、フォーカスの所有は通常、ウィンドウがある方法で表示します。たとえばテキストフィールド・コントロールは、点滅するキャレットを表示してフォーカスの所有を示します。

  トップレベル・ウィンドウ内では、プラットフォームの慣習に応じて、特定の種類のサブウィンドウだけがフォーカスを持てます。さらに、最初にフォーカスを所有するサブウィンドウもプラットフォーム固有です。ユーザはさまざまな方法でフォーカスを移せますが、通常は対象ウィンドウをクリックします。プログラムは `focus` メソッドを使って、サブウィンドウへフォーカスを移したり初期フォーカスを設定したりできます。

  `'wheel-up` または `'wheel-down` イベントは、オペレーティング・システムがホイール・イベントをどう扱うかによって、キーボード・フォーカスを持つウィンドウ以外に送られることがあります。

  キー押しイベントは、実際のキー押し、またはオートキー繰り返しのどちらにも対応します。間にキー解放イベントがない複数のキー押しイベントは、通常オートキーを示します。ただし他の入力イベントと同様、キー解放イベントも（たとえばモーダル・ダイアログの出現などにより）落ちることがあります。

ボタンやリストボックスなどのコントロールは、キーボードとマウスのイベントを自動的に処理し、最終的にはコントロール作成時に渡されたコールバック手続きを呼び出します。キャンバスは、マウスとキーボードのイベントをそれぞれ `on-event` と `on-char` メソッドへ伝播します。

マウスとキーボードのイベントは、そのウィンドウへ特別な方法で届けられます。受信ウィンドウの各祖先は、`on-subwindow-event` と `on-subwindow-char` メソッドを通じてイベントを横取りする機会を得ます。詳細はメソッドの説明を参照してください。

トップレベル・ウィンドウの既定の `on-subwindow-char` メソッドは、キーボード・イベントを横取りして、メニュー・ショートカット・イベントとフォーカス移動イベントを検出します。詳細は `frame%` の `on-subwindow-char` と `dialog%` の `on-subwindow-char` を参照してください。特定の OS 固有のキー組み合わせは低レベルで捕捉され、オーバーライドできません。たとえば Windows と Unix では、Alt を押して離すと常にキーボード・フォーカスがメニューバーへ移ります。同様に、Windows では Alt-Tab で別のアプリケーションへ切り替わります。（Windows では Alt-Space がシステム・メニューを呼び出しますが、このショートカットは `on-system-menu-char` によって実装され、`frame%` の `on-subwindow-char` と `dialog%` の `on-subwindow-char` から呼ばれます。）

### 1.6 イベントのディスパッチとイベントスペース

グラフィカル・ユーザ・インタフェースは本質的にマルチスレッドのシステムです。一方のスレッドは画面上のウィンドウを管理するプログラムであり、もう一方のスレッドはマウスを動かしキーボードを打つユーザです。GUI プログラムは通常、少なくともプログラマの観点からは、このマルチスレッド・システムを逐次的なものに翻訳するために**イベント・キュー**を使います。各ユーザ操作は一度に一つずつ処理され、前の操作が完全に処理されるまで、それ以降のユーザ操作は無視されます。マルチスレッド過程からシングルスレッド過程への変換は、GUI プログラムの実装を大幅に単純化します。

純粋に逐次的なイベント・キューがもたらすプログラミング上の便宜にもかかわらず、次のような状況ではユーザとの対話がより柔軟である必要があります。

- **入れ子のイベント処理：** イベントを処理する過程で、ユーザからさらに情報を得る必要が生じることがあります。通常、そのような情報はモーダル・ダイアログを通じて得られます。入力の得方にかかわらず、元のイベントが完全に処理される前に、さらにユーザ・イベントを受信・処理しなければなりません。イベントのさらなる処理を許すには、元のイベントのハンドラが明示的にシステムへ**yield** しなければなりません。yield すると、イベントは純粋に逐次的ではなく、入れ子の仕方で処理されます。
- **非同期のイベント処理：** アプリケーションは、ユーザとの独立した対話を表すウィンドウから成ることがあります。たとえば描画プログラムは複数の描画ウィンドウをサポートし、あるウィンドウでの特に時間のかかる作業（例：画像への特殊フィルタ効果）が、別のウィンドウでの作業を妨げてはなりません。そのようなアプリケーションは、個々のウィンドウごとに逐次的なイベント処理を必要としますが、ウィンドウ横断では非同期（潜在的に並列）のイベント処理を必要とします。言い換えると、アプリケーションはウィンドウごとに別のイベント・キューと、各イベント・キューごとの別のイベント処理スレッドを必要とします。

**イベントスペース（eventspace）** は、GUI イベントを処理するための文脈です。各イベントスペースは自身のイベント・キューを維持し、単一のイベントスペース内のイベントは、指定された**ハンドラ・スレッド**によって逐次的にディスパッチされます。このハンドラ・スレッドで走るイベント処理手続きは、`yield` を呼ぶことでシステムへ yield でき、その場合、他のイベント処理手続きが同じハンドラ・スレッド内で入れ子の（しかしシングルスレッドの）仕方で呼ばれることがあります。異なるイベントスペースからのイベントは、別々のハンドラ・スレッドによって非同期にディスパッチされます。

フレームやダイアログが親なしで作成されると、それらは「イベントスペースの作成と設定」で述べるように現在のイベントスペースに関連付けられます。トップレベル・ウィンドウとその子孫に対するイベントは、常にそのウィンドウのイベントスペースでディスパッチされます。すべてのダイアログはモーダルです。ダイアログの `show` メソッドは、ダイアログが表示されているあいだイベントを処理するために暗黙に `yield` を呼びます。（スレッドとモーダル・ダイアログについては、「イベントスペースとスレッド」も参照してください。）さらに、モーダル・ダイアログが表示されているとき、システムはダイアログのイベントスペース内の他のトップレベル・ウィンドウへのキーおよびマウスの押し/解放イベントを無効化しますが、他のイベントスペース内のウィンドウはモーダル・ダイアログの影響を受けません。（モーダル・ダイアログが表示されているときでも、マウスの移動、入出のイベントはすべてのウィンドウへ届けられます。）

#### 1.6.1 イベントの種類と優先度

ボタン・クリック、キー押し、更新などのユーザおよびウィンドウ機構の操作に対応するイベントに加え、システムは二種類の内部イベント——**タイマ・イベント**と**明示的にキューされたイベント**——をディスパッチします。

タイマ・イベントは `timer%` のインスタンスによって作られます。タイマが開始され期限が来ると、タイマは自身の `notify` メソッドを呼ぶイベントをキューします。トップレベル・ウィンドウと同様、各タイマは作成時に特定のイベントスペース（「イベントスペースの作成と設定」で述べる現在のイベントスペース）に関連付けられ、そのイベントスペースにイベントをキューします。

明示的にキューされたイベントは `queue-callback` で作られ、イベントを処理するコールバック手続きを受け取ります。イベントは `queue-callback` 呼び出し時点の現在のイベントスペースに、`queue-callback` の（省略可能な）第二引数で指定された高優先度または低優先度でエンキューされます。

イベントスペースのイベント・キューは実際には優先度キューで、イベントはその種類に応じて、最高優先度（最初にディスパッチ）から最低優先度（最後にディスパッチ）まで並べられます。

- `queue-callback` でインストールされた高優先度イベントが最高優先度です。
- `timer%` 経由のタイマ・イベントが二番目に高い優先度です。
- ウィンドウ更新イベントが三番目に高い優先度です。
- マウス・クリックやキー押しなどの入力イベントが、下から二番目の優先度です。
- `queue-callback` でインストールされた低優先度イベントが最低優先度です。

プログラマはイベントがディスパッチされる順序を直接制御できませんが、`event-dispatch-handler` パラメータ経由でイベント・ディスパッチ・ハンドラを設定することで、ディスパッチのタイミングを制御できます。このパラメータと他のイベントスペース手続きは、「Eventspaces」でより詳しく述べます。

#### 1.6.2 イベントスペースとスレッド

新しいイベントスペースが作られると、対応する**ハンドラ・スレッド**がイベントスペース用に作られます。初期イベントスペースは新しいハンドラ・スレッドを作らず、代わりに `racket/gui/base` がインスタンス化されたスレッドを初期イベントスペースのハンドラ・スレッドとして使います。Startup Actions も参照してください。

システムがイベントスペースのイベントをディスパッチするとき、それは常にイベントスペースのハンドラ・スレッドで行います。ハンドラ手続きは無期限に走る新しいスレッドを作れますが、ハンドラ・スレッドがハンドラ手続きを実行しているあいだは、対応するイベントスペースに対して新しいイベントをディスパッチできません。

ハンドラ・スレッドがダイアログを表示するとき、ダイアログの `show` メソッドは、ダイアログが表示されているあいだ暗黙に `yield` を呼びます。非ハンドラ・スレッドがダイアログを表示するとき、非ハンドラ・スレッドは単にダイアログが閉じられるまでブロックします。非ハンドラ・スレッドから引数なしで `yield` を呼んでも効果はありません。非ハンドラ・スレッドからセマフォ付きで `yield` を呼ぶことは、`semaphore-wait` を呼ぶのと等価です。

`racket/gui/base` のウィンドウ機構関数とメソッドはどのスレッドからでも呼べますが、スレッド間やハンドラ・スレッドとのあいだで競合状態を作らないよう注意してください。

- グラフィカル・オブジェクトはスレッドセーフですが、グラフィカル要素が複数のスレッドで操作されると、コールバックや他のイベント・ハンドラは変化するオブジェクト状態を見ることがあります。
- エディタ・クラスには特定のスレッド制約があります。「エディタとスレッド」を参照してください。

非ハンドラ・スレッドで GUI 要素を操作すると（コールバックがハンドラ・スレッドで走る可能性があるため）混乱しやすい競合状態を作りやすいので、すべての GUI の設定と操作はハンドラ・スレッドで行うのが最善です。`queue-callback` 関数は、他の任意のスレッドからハンドラ・スレッドでの作業をスケジュールするのに役立ちます。すでにハンドラ・スレッドで走っているときは、非 GUI イベントを待ちつつ GUI イベントを進めるために `yield` を使います。

#### 1.6.3 イベントスペースの作成と設定

フレーム、ダイアログ、またはタイマが作られるたびに、それらは `current-eventspace` パラメータ（Parameters を参照）が決める現在のイベントスペースに関連付けられます。

`make-eventspace` 手続きは新しいイベントスペースを作ります。次の例は、新しいイベントスペースとそのイベントスペース内の新しいフレームを作ります（`parameterize` 構文形式は一時的にパラメータ値を設定します）。

```racket
(let ([new-es (make-eventspace)])
  (parameterize ([current-eventspace new-es])
    (new frame% [label "Example"])))
```

イベントスペースが作られると、それは現在の**カストディアン（custodian）** の管理下に置かれます。カストディアンがイベントスペースをシャットダウンすると、そのイベントスペースに関連付けられたすべてのフレームとダイアログが破棄され（`top-level-window<%>` の `can-close?` や `on-close` は呼ばれません）、イベントスペース内のすべてのタイマが停止し、エンキューされたすべてのコールバックが削除されます。シャットダウンされたイベントスペースで新しいウィンドウ、タイマ、または明示的にキューされたイベントを作ろうとすると、`exn:fail` 例外が発生します。

イベントスペースは**同期可能なイベント**（GUI イベントと混同しないでください）なので、`sync` と一緒に使えます。同期可能なイベントとして、イベントスペースは、フレームが見えている、タイマが動いている、コールバックがキューされている、または `menu-bar%` が `'root` 親で作られているときにブロッキング状態にあります。イベントスペースのブロッキング状態は、ディスパッチ待ちのイベントがあるかどうかとは無関係であることに注意してください。また、イベントの処理中、タイマの発火中、コールバックの実行中にイベントスペースが必ずしもブロッキング状態にあるとは限らず、ハンドラ・スレッドが終了したあとイベントスペースがブロッキング状態のまま残ることもあることに注意してください。

#### 1.6.4 継続とイベント・ディスパッチ

システムがイベントをディスパッチするたびに、ハンドラへの呼び出しは、継続の中断（例外発生時など）とハンドラが捕捉する継続を区切る**継続プロンプト**（`call-with-continuation-prompt` を参照）で包まれます。区切られた継続プロンプトはイベント・ディスパッチ・ハンドラの呼び出しの外側にインストールされるため、捕捉された継続にはイベント・ディスパッチ・ハンドラの呼び出しが含まれます。

たとえばボタンのコールバックが例外を起こすと、既定の例外ハンドラが行う中断は、プログラムを終了させたり囲む `(yield)` を超えたりするのではなく、イベント・ディスパッチ地点へ戻ります。ただし `with-handlers` が、ボタン・コールバックによって起こされた例外につながる `(yield)` を包んでいる場合、その例外は `with-handlers` によって捕捉され得ます。

同様の線で、ボタン・コールバックが（既定の継続プロンプト・タグを使って）継続を捕捉した場合、その継続を適用すると、ハンドラが戻るまでの作業だけが再インストールされます。ボタン・コールバックを呼び出すディスパッチ機構は継続に含まれません。したがって、ボタン・コールバック中に捕捉された継続は、同じコールバックの外でも潜在的に有用です。

#### 1.6.5 ロギング

GUI システムは、イベントがいつ処理されたか、処理にどれだけかかったかのタイミングをログに記録します。Racket コードへのコールバックを含む各イベントには二つのログ・イベントがあり、どちらも `gui-event` 構造体を使います。

```racket
(struct gui-event (start end name) #:prefab)
```

`start` フィールドは、イベント処理開始時の `(current-inexact-milliseconds)` の結果です。`end` フィールドは、イベント処理開始時のログ・メッセージでは `#f` で、終了時のログ・メッセージでは終了時の `(current-inexact-milliseconds)` の結果です。`name` フィールドはイベントを処理した関数の名前です。`queue-callback` ベースのイベントの場合は、`queue-callback` に渡されたサンクの名前です。

### 1.7 キャンバスでのアニメーション

キャンバスの内容はバッファされるため、キャンバスを再描画する必要があるとき、通常は `on-paint` メソッドや `paint-callback` 関数を再び呼ぶ必要はありません。ちらつきをさらに減らすため、`on-paint` メソッドや `paint-callback` 関数が呼ばれているあいだ、ウィンドウ機構システムはキャンバス内容バッファの画面へのフラッシュを避けます。

キャンバスの内容は、キャンバスの `get-dc` メソッドの結果でいつでも描画して更新でき、描画はスレッドセーフです。キャンバス内容への変更は周期的に画面へフラッシュされます（必ずしもイベント処理の境界でとは限りません）が、フラッシュが中断されていなければ、`flush` メソッドは直ちに画面へフラッシュします。`suspend-flush` と `resume-flush` メソッドは、自動および明示的なフラッシュの両方を中断・再開します。ただし一部のプラットフォームでは、まれに自動フラッシュが強制されることがあります。

ほとんどのアニメーション用途では、`suspend-flush`、`resume-flush`、`flush` を使って、ちらつきとアニメーション用の追加描画バッファの必要性を避けられます。アニメーション中は、部分的に描かれたフレームが画面へフラッシュされないよう、各アニメーション・フレームの構築を `suspend-flush` と `resume-flush` で囲みます。すぐに `suspend-flush` が続く場合でも、キャンバス内容が用意できたときにフラッシュされるよう `flush` を使ってください。フラッシュが頻繁に中断されると、画面へのフラッシュ処理が飢餓状態になり得るからです。`canvas%` の `refresh-now` メソッドは、この一連の操作を便利にカプセル化します。

### 1.8 画面解像度とテキストのスケーリング

Mac OS では、画面サイズは描画単位でユーザに説明されます。Retina ディスプレイは描画単位あたり2ピクセルを提供し、描画単位はウィンドウ・サイズ、子ウィンドウ位置、キャンバス描画に一貫して使われます。フォント・サイズのための「ポイント」は描画単位と等価です。

Windows と Unix では、画面サイズはピクセルでユーザに説明されますが、テキストや他の項目に適用するスケールをユーザが独立に選べます。典型的なテキスト・スケールは 125%、150%、200% です。`racket/gui` ライブラリはこのスケールを、画面、ウィンドウ、ボタン、キャンバス描画を含むすべての GUI 要素に使います。たとえばスケールが 200% なら、`get-display-size` が報告する画面サイズは、各次元のピクセル数の半分になります。丸め誤差により、ウィンドウの報告サイズが、ちょうど設定したサイズと異なることがあるので注意してください。フォント・サイズのための「ポイント」は `(/ 96 72)` 描画単位と等価です。

Unix では、環境変数 `PLT_DISPLAY_BACKING_SCALE` が正の実数に設定されている場合、それが `racket/gui` のスケーリングに関する特定のシステム設定を上書きします。GTK+ 3（Platform Dependencies を参照）では、環境変数はシステム全体のテキスト・スケーリングを上書きします。GTK+ 2 では、環境変数はテキストとコントロールの両方のスケーリングを上書きします。ただしメニュー、既定のラベル・フォントを使うコントロール・ラベル、ラベル以外のコントロール部分は、`PLT_DISPLAY_BACKING_SCALE` で指定されたスケールを使いません。

`gui-lib` パッケージのバージョン 1.14 での変更：Unix でのスケーリングのサポートを追加しました。
