# 10 リストの更なる世界 (More on Lists)

リストは、現在ほとんどすべての言語に備わっている、非常に汎用性の高いデータ形式です。プログラマーはリストを使って、大規模なアプリケーション、人工知能、分散システムなどを構築してきました。この章では、この世界におけるいくつかのアイデアを説明します。リストを生成する関数、リストの中に構造体を必要とするデータ表現、そしてテキストファイルをリストとして表現することなどが含まれます。

## 10.1 リストを生成する関数 (Functions that Produce Lists)

以下は、時給制の従業員の賃金を計算する関数です。
```racket
; Number -> Number
; 労働時間 h に対する賃金を計算する
(define (wage h)
  (* 12 h))
```
この関数は労働時間数を受け取り、賃金を算出します。しかし、給与計算ソフトウェアを必要としている会社は、この関数単体には興味がありません。会社が欲しいのは、すべての従業員の賃金を計算する関数です。

この新しい関数を `wage*` と呼びましょう。その役割は、すべての従業員の労働時間を処理し、それぞれに支払うべき賃金を決定することです。簡単にするため、入力は数値のリスト（各数値は1人の従業員が働いた時間数を表す）であり、期待される結果も週給のリスト（これも数値のリストで表される）であると仮定しましょう。

入力と出力に対するデータ定義はすでに持っているので、すぐに設計の第2ステップに進むことができます。
```racket
; List-of-numbers -> List-of-numbers
; 週ごとの労働時間リストから週給のリストを計算する
(define (wage* whrs)
  '())
```

次に、入力の例とそれに対応する出力の例が必要です。そこで、週の労働時間を表す短い数値リストをいくつか作ってみましょう。

| 入力 (given) | 期待される出力 (expected) |
| :--- | :--- |
| `'()` | `'()` |
| `(cons 28 '())` | `(cons 336 '())` |
| `(cons 4 (cons 2 '()))` | `(cons 48 (cons 24 '()))` |

出力を計算するために、与えられた入力リストの各数値に対して週の賃金を決定します。最初の例では、入力リストに数値がないので、出力は `'()` です。2番目と3番目の期待される出力がなぜそうなるのか、理解するようにしてください。

`wage*` は「リスト」の章の他のいくつかの関数と同じ種類のデータを消費し、テンプレートはデータ定義の形にのみ依存するため、このテンプレートを再利用できます。
```racket
(define (wage* whrs)
  (cond
    [(empty? whrs) ...]
    [else (... (first whrs) ...
               ... (wage* (rest whrs)) ...)]))
```

テンプレートの作成を練習したい場合は、図52の質問を使用してください。

これで最も創造的な設計ステップに進む準備ができました。設計レシピに従って、テンプレートの各 `cond` 行を個別に考えます。
再帰しないケースでは、`(empty? whrs)` が真であり、入力は `'()` であることを意味します。上記の例から、望ましい答えは `'()` であると指定されているので、これで完了です。

2番目のケースでは、設計の質問が、テンプレートの各式が何を計算するかを述べるよう指示しています。
- `(first whrs)` は `whrs` の最初の数値を返します。これは最初の労働時間数です。
- `(rest whrs)` は与えられたリストの残りの部分です。
- `(wage* (rest whrs))` は、残りの部分が私たちが今定義している関数自体によって処理されることを意味します。いつものように、この式のシグネチャと目的ステートメントを使って結果がどうなるかを考えます。シグネチャからは数値のリストであることがわかり、目的ステートメントからは、このリストが残りの時間リストに対する賃金のリストを表していることがわかります。

重要なのは、関数がまだ完全に定義されていなくても、このケースで結果を計算する式を定式化するときに、これらの事実を頼りにすることです。

`whrs` の最初の項目を除くすべての要素に対する賃金のリストはすでに得られているので、関数は `whrs` 全体に対する期待される出力を生成するために2つの計算を行う必要があります。`(first whrs)` の週賃金を計算し、さらに `whrs` のすべての週賃金を表すリストを構築することです。最初の部分には `wage` を再利用します。2番目の部分には、2つの情報を `cons` で1つのリストにまとめます。
```racket
(cons (wage (first whrs)) (wage* (rest whrs)))
```

これで定義が完成しました。図64を参照してください。

**図64: すべての従業員の賃金を計算する**
```racket
; List-of-numbers -> List-of-numbers
; 与えられたすべての週労働時間から週給を計算する
(define (wage* whrs)
  (cond
    [(empty? whrs) '()]
    [else (cons (wage (first whrs))
                (wage* (rest whrs)))]))

; Number -> Number
; 労働時間 h に対する賃金を計算する
(define (wage h)
  (* 12 h))
```

## 10.2 リスト内の構造体 (Structures in Lists)

週の労働時間を単なる数値として表すのは、あまり良い選択ではありません。なぜなら、給与小切手の印刷には、週あたりの労働時間以上の情報が必要だからです。また、すべての従業員が同じ時給であるとは限りません。幸いなことに、リストにはアトミックな値以外の項目を含めることができます。実際、リストには必要なあらゆる値、特に構造体を含めることができます。

私たちの進行中の例では、まさにそのようなデータ表現が必要です。数値の代わりに、従業員と彼らの労働時間および賃金率を表す構造体を使用します。
```racket
(define-struct work [employee rate hours])
; A Work is a structure:
;   (make-work String Number Number)
; 解釈: (make-work n r h) は従業員名 n、時給率 r、労働時間 h を組み合わせる。
```

この表現はまだ単純ですが、構造体を含むリストのデータ定義を定式化する必要があるため、追加の課題として十分です。
```racket
; Low (list of works の略) は以下のいずれか:
; -- '()
; -- (cons Work Low)
; 解釈: Low のインスタンスは、複数の従業員が働いた時間を表す。
```

以下は `Low` の3つの要素の例です。
```racket
'()
(cons (make-work "Robby" 11.95 39)
      '())
(cons (make-work "Matthew" 12.95 45)
      (cons (make-work "Robby" 11.95 39)
            '()))
```

データ定義を使って、これらのデータがなぜ `Low` に属するのか説明してください。
ストップ！また、データ定義を使って、さらに2つの例を作成してください。

> 実際のプロジェクトで作業するときは、このようなサフィックス（.v2など）は使用せず、代わりに異なるバージョンのコードを管理するためのツールを使用します。

`Low` の定義が理にかなっていることが分かったので、単なる数値リストではなく、`Low` の要素を消費するように関数 `wage*` を再設計する時が来ました。
```racket
; Low -> List-of-numbers
; 与えられたレコードから週給を計算する
(define (wage*.v2 an-low)
  '())
```

関数名の最後にあるサフィックス ".v2" は、これが関数の改訂された第2バージョンであることを読者に伝えます。この場合、改訂は新しいシグネチャと適応された目的ステートメントから始まります。ヘッダーは上記と同じです。

設計レシピの第3ステップは、例を作成することです。上記の2番目のリストから始めましょう。これには1つの作業レコード、すなわち `(make-work "Robby" 11.95 39)` が含まれています。その解釈は、"Robby" が39時間働き、時給11.95ドル支払われるということです。したがって、彼のその週の賃金は `(* 11.95 39)`、つまり 466.05ドルです。したがって、`wage*.v2` に対する期待される結果は `(cons 466.05 '())` です。当然ながら、入力リストに2つの作業レコードが含まれている場合、この種の計算を2回実行し、結果は2つの数値のリストになります。ストップ！上記の3番目のデータ例に対する期待される結果を決定してください。

**数値に関する注意**
BSLは、他のほとんどのプログラミング言語とは異なり、小数を正確な分数として理解することに注意してください。例えばJavaのような言語では、最初の作業レコードの期待される賃金に対して `466.04999999999995` を生成します。小数を扱う演算がこのような奇妙な動作をするタイミングを予測することはできないため、他のプログラミング言語に備えるためにも、以下のように例を記述しておくのが得策です。
```racket
(check-expect
  (wage*.v2 (cons (make-work "Robby" 11.95 39) '()))
  (cons (* 11.95 39) '()))
```
このように例を書き留めることは、実際に賃金を計算する方法を本当に理解したことを意味します。**[注意終わり]**

ここからテンプレートの開発に移ります。テンプレートに関する質問を使用すると、すぐに次のようになります。
```racket
(define (wage*.v2 an-low)
  (cond
    [(empty? an-low) ...]
    [(cons? an-low)
     (... (first an-low) ...
      ... (wage*.v2 (rest an-low)) ...)]))
```
データ定義が2つの節で構成されており、最初の節で `'()`、2番目の節で `cons` された構造体を導入しているためです。しかし、入力についてこのテンプレートが表現している以上のことが分かっていることにも気づくでしょう。例えば、`(first an-low)` は与えられたリストから3つのフィールドを持つ構造体を取り出します。これは、テンプレートにさらに3つの式を追加することを示唆しているようです。
```racket
(define (wage*.v2 an-low)
  (cond
    [(empty? an-low) ...]
    [(cons? an-low)
     (... (first an-low) ...
      ... (work-employee (first an-low)) ...
      ... (work-rate (first an-low)) ...
      ... (work-hours (first an-low)) ...
      ... (wage*.v2 (rest an-low)) ...)]))
```
この詳細なテンプレートは、2つの指針を組み合わせた結果です。入力データに自己参照リストが含まれているため、テンプレートには自己参照関数呼び出しが含まれます。また、リストには構造体が含まれているため、テンプレートには構造体からフィールド値を抽出するためのセレクタ式が含まれます。

これで最も創造的なステップの準備が整いました。最初の `cond` 節は、空リスト `'()' に対するものであり、答えは `'()` です。

2番目の節については、テンプレートに用意されている構成要素が何を計算するかを書き留めます。
- `(first an-low)` はリスト `an-low` の最初の `Work` 構造体です。
- `(work-employee (first an-low))` は最初の従業員の名前です。
- `(work-rate (first an-low))` は最初の従業員の時給です。
- `(work-hours (first an-low))` は最初の従業員の労働時間です。
- `(wage*.v2 (rest an-low))` は、リストの残りの作業レコードに対する賃金のリストを計算します。

残りのレコードに対する賃金リストはすでに得られているので、必要なのは最初の作業レコードの賃金を計算し、それを再帰の結果に `cons` することだけです。最初の作業レコードの賃金は、その時給に労働時間を掛けることで計算できます。
```racket
(* (work-rate (first an-low)) (work-hours (first an-low)))
```
これら2つの値を `cons` すると、完全な関数定義が得られます。図65を参照してください。

**図65: 従業員レコードのリストから賃金を計算する**
```racket
(define-struct work [employee rate hours])
; A Work is a structure:
;   (make-work String Number Number)
; 解釈: (make-work n r h) は従業員名、時給率 r、労働時間 h を組み合わせる

; Low (list of works の略) は以下のいずれか:
; -- '()
; -- (cons Work Low)
; 解釈: Low のインスタンスは、複数の従業員が働いた時間を表す

; Low -> List-of-numbers
; 与えられたすべての週労働時間から週給を計算する
(define (wage*.v2 an-low)
  (cond
    [(empty? an-low) '()]
    [else (cons (* (work-rate (first an-low))
                   (work-hours (first an-low)))
                (wage*.v2 (rest an-low)))]))
```

この定義は完璧ですが、設計上のアドバイスの最初の部分に少し違反しています。それは、1つの関数に1つのタスクを持たせるという点です。ここでの2番目の `cond` 節は、作業レコードを処理してその週の賃金を計算することと、その賃金を残りの賃金リストと組み合わせることの2つを行っています。これらのタスクを分離するために、1つの作業レコードを受け取ってその賃金を計算するヘルパー関数を「願望（wish）」します。
```racket
; Work -> Number
; 与えられた作業レコード w に対する賃金を計算する
(define (wage.v2 w)
  (* (work-rate w) (work-hours w)))
```
このヘルパー関数を使用すると、`wage*.v2` の定義は非常に簡潔になり、設計も分かりやすくなります。
```racket
(define (wage*.v2 an-low)
  (cond
    [(empty? an-low) '()]
    [else (cons (wage.v2 (first an-low))
                (wage*.v2 (rest an-low)))]))
```
このようにプログラムを補助関数に分けることで、より複雑なデータ表現が必要になったとき（例えば、残業手当の計算方法の追加など）の拡張が非常に容易になります。

## 10.3 リストのリスト、およびテキストファイル (Lists in Lists, and Text Files)

「リストのリスト」という考え方は、最初は恐ろしく聞こえるかもしれませんが、心配する必要はありません。設計レシピは、このような複雑な場合でも有効に機能します。

ファイルの内容をリストとして表すことを考えてみましょう。これはリストのリストを使用する典型的な例です。たとえば、ファイルを「行のリスト」として読み込み、各行を「単語のリスト」として表す場合、それは「文字列のリストのリスト」になります。この表現は、ファイルの元の構造（空行など）を完全に維持できるという利点があります。

図67は、ファイルからテキストを読み込むためのいくつかの便利な関数を示しています。

**図67: ファイルの読み込み**
```racket
; String -> String
; ファイル f の内容を単一の文字列として返す
(define (read-file f) ...)

; String -> List-of-strings
; ファイル f の内容を行ごとの文字列 of リストとして返す
(define (read-lines f) ...)

; String -> List-of-strings
; ファイル f の内容を単語ごとの文字列 of リストとして返す
(define (read-words f) ...)

; String -> List-of-list-of-strings
; ファイル f の内容を「行のリスト（各行は単語のリスト）」として返す
(define (read-words/line f) ...)
```
これらの関数は、指定されたファイルの名前を文字列引数として受け取ります。指定されたファイルがプログラムと同じフォルダーに存在しない場合、エラーをシグナルします。

これらの関数はすべてのテキストファイルを処理するのに十分ではありませんが、私たちの目的であるシステム開発の原則を学ぶためには十分であり、面白いプログラムを設計する力を与えてくれます。

**練習問題171.** `List-of-strings` のデータ定義がどのようなものか、あなたは知っています。それを書き出してください。また、Piet Hein の詩を、各行が文字列として表されるインスタンス、および各単語が文字列として表されるインスタンスとして表現できることを確認してください。`read-lines` と `read-words` を使用して、表現の選択を確認してください。

次に、`List-of-list-of-strings` のデータ定義を作成してください。同じ詩を、各行が単語（文字列）のリストとして表され、詩全体がそれら行のリストであるインスタンスとして表現してください。`read-words/line` を使用して確認できます。

オペレーティングシステムには、ファイルを測定するプログラム（Unix の `wc` コマンドなど）が付属しています。あるプログラムは行数を数え、別のプログラムは各行に現れる単語数を測定します。後者を例にして、設計レシピが複雑な関数の設計にどのように役立つかを見てみましょう。

最初のステップは、必要なすべてのデータ定義を用意することです。上記の練習問題で入力データの定義を作成しました。また、前セクションで出力データを表す `List-of-numbers` を定義しました。簡潔にするために、`LN` を「文字列のリストのリスト」のクラスを表す名前として使用し、目的の関数のヘッダーを記述します。
```racket
; LN -> List-of-numbers
; 各行にある単語の数を決定する
(define (words-on-line lls)
  '())
```

次に必要なのは、データ例の作成です。
```racket
(define line0 (cons "hello" (cons "world" '())))
(define line1 '())

(define lls0 '())
(define lls1 (cons line0 (cons line1 '())))
```
`line0` は2つの単語を含み、`line1` は単語を含いません。`lls0` は行を含まず、`lls1` はこれら2つの行を含んでいます。

これらのデータ例から、関数例を作成します。`lls0` に `words-on-line` を適用すると、行がないため空リスト `'()` が返されるはずです。`lls1` に適用すると、2つの行があるため2つの数値のリスト `(cons 2 (cons 0 '()))` が返されるはずです。

これをテストケースに変換します。
```racket
(check-expect (words-on-line lls0) '())
(check-expect (words-on-line lls1) (cons 2 (cons 0 '())))
```

テンプレートの開発が、このサンプル問題の最も興味深いステップです。図52のテンプレートに関する質問に答えることで、おなじみのリスト処理テンプレートがすぐに得られます。
```racket
(define (words-on-line lls)
  (cond
    [(empty? lls) ...]
    [else (... (first lls) ...
           ... (words-on-line (rest lls)) ...)]))
```
前セクションと同様に、`(first lls)` は複雑な構造を持つ `List-of-strings` を返すことが分かっています。この知識を表現するために入れ子のテンプレートを挿入したくなりますが、より良いアプローチは、別個の補助テンプレート（ここでは `line-processor`）を用意し、最初のテンプレートからそれを参照することです。

補助テンプレートはリストを処理する関数のためのものなので、ほぼ同じに見えます。
```racket
(define (line-processor ln)
  (cond
    [(empty? ln) ...]
    [else (... (first ln) ...
           ... (line-processor (rest ln)) ...)]))
```
重要な違いは、`(first ln)` がリストからアトミックな値である文字列を抽出することです。このテンプレートを使用すると、`words-on-line` の第2のケースの最初の行を以下のように変更できます。
```racket
(... (line-processor (first lls)) ...
 ... (words-on-line (rest lls)) ...)
```
これは、第5のステップにおいて、`words-on-line` の定義が補助関数の設計を要求することを示唆しています。

さて、プログラミングの時間です。空リスト（行がない）に対する最初のケースは簡単で、答えは `'()` です。
2番目のケースにはいくつかの式があり、それぞれが何を計算するかを思い出します。
- `(first lls)` は最初の行を取り出します。
- `(line-processor (first lls))` は、この行を処理するための補助関数を設計する可能性を示唆します。
- `(rest lls)` は残りの行のリストです。
- `(words-on-line (rest lls))` は、残りの行に対する単語数のリストを計算します。

1つの行を受け取ってその中の単語数をカウントする補助関数（`words#` と呼びましょう）を設計できると仮定すれば、2番目のケースの本体は簡単に完成します。
```racket
(cons (words# (first lls)) (words-on-line (rest lls)))
```
この式は、最初の行の単語数を、残りの行の単語数を表すリストの先頭に `cons` します。

残るは `words#` 関数の設計です。そのテンプレートは `line-processor` であり、その目的はリスト内の文字列の数をカウントすることです。
```racket
; List-of-strings -> Number
; los に含まれる単語の数をカウントする
(define (words# los)
  0)
```
しかし、あなたは以前にこれとまったく同じ機能を持つ `how-many` 関数を設計したことを思い出すでしょう。既存の関数を再利用するのは良いことなので、以下のように定義できます。
```racket
(define (words# los)
  (how-many los))
```
実際には、BSLにはリストの要素数をカウントする標準の `length` 関数がすでに存在します。したがって、それを使用するのが最適です。完全な設計を図68にまとめます。

**図68: 各行の単語数をカウントする**
```racket
; LN (list of lines の略) は以下のいずれか:
; -- '()
; -- (cons List-of-strings LN)
; 解釈: 行のリスト。各行は文字列（単語）のリスト。

(define line0 (cons "hello" (cons "world" '())))
(define line1 '())
(define ln0 '())
(define ln1 (cons line0 (cons line1 '())))

; LN -> List-of-numbers
; 各行にある単語の数を決定する
(check-expect (words-on-line ln0) '())
(check-expect (words-on-line ln1) (cons 2 (cons 0 '())))

(define (words-on-line ln)
  (cond
    [(empty? ln) '()]
    [else (cons (length (first ln))
                (words-on-line (rest ln)))]))
```

これで、最初のファイルユーティリティを設計する準備が整いました。
```racket

; 与えられたファイル内の各行の単語数をカウントする
```

## 10.4 グラフィカルエディタの再訪 (A Graphical Editor, Revisited)

第5章で、私たちはインタラクティブなグラフィカルな1行エディタを設計しました。そこではエディタの状態を表現するために、2つの文字列を保持する構造体か、文字列とカーソル位置のインデックスを保持する構造体のいずれかを使用しました。

3番目の選択肢は、1Stringの2つのリストを組み合わせた構造体を使用することです。
```racket
(define-struct editor [pre post])
; An Editor is a structure:
;   (make-editor Lo1S Lo1S)

; An Lo1S (list of 1Strings の略) は以下のいずれか:
; -- '()
; -- (cons 1String Lo1S)
```
なぜこのような表現にするのかを考える前に、2つのデータ例を作成してみましょう。
```racket
(define good
  (cons "g" (cons "o" (cons "o" (cons "d" '())))))
(define all
  (cons "a" (cons "l" (cons "l" '()))))
(define lla
  (cons "l" (cons "l" (cons "a" '()))))

; データ例1:
(make-editor all good)

; データ例2:
(make-editor lla good)
```

これら2つの例は、解釈（interpretation）を書き留めることがいかに重要であるかを示しています。エディタの2つのフィールドがカーソルの左側と右側の文字を表すことは明らかですが、この構造体タイプを解釈する方法には少なくとも2つのアプローチがあります。

1. `(make-editor pre post)` は、`pre` の文字がカーソルの前にあり、`post` の文字がカーソルの後ろにあることを意味し、組み合わされたテキストは次のようになる。
   ```racket
   (string-append (implode pre) (implode post))
   ```
   `implode` 関数は 1String のリストを単一の `String` に変換することを思い出してください。

2. `(make-editor pre post)` は、`pre` の文字がカーソルの前に**逆順（reverse）**で並んでいることを意味する。この場合、エディタに表示されるテキストは次のようになります。
   ```racket
   (string-append (implode (reverse pre)) (implode post))
   ```

どちらの解釈も素晴らしい選択肢ですが、2番目の解釈を使用するとプログラムの設計が大幅に簡素化されることが分かっています。このセクションの残りの部分では、リストを構造体の中に組み込む使用法を示しながら、この点について説明します。

まず、データ定義を理解するために明らかに必要となる `rev` 関数（リストを逆順にする関数）から始めましょう。
```racket
; Lo1s -> Lo1s
; 与えられたリストを逆順にしたリストを生成する

(check-expect (rev (cons "a" (cons "b" (cons "c" '()))))
              (cons "c" (cons "b" (cons "a" '()))))

(define (rev l)
  l)
```

`rev` のテンプレートは、通常のリストテンプレートです。
```racket
(define (rev l)
  (cond
    [(empty? l) ...]
    [else (... (first l) ...
           ... (rev (rest l)) ...)]))
```

空リストの逆順は空リストなので、最初の節を埋めるのは簡単です。2番目の節については、設計の質問を使います。
- `(first l)` はリストの最初の 1String です。
- `(rest l)` はリストの残りの部分です。
- `(rev (rest l))` はリストの残りの部分の逆順です。

ヒントとして、図71の表を作成します。

**図71: `rev` のための表**

| l | (first l) | (rest l) | (rev (rest l)) | (rev l) |
| :--- | :--- | :--- | :--- | :--- |
| `(cons "a" '())` | `"a"` | `'()` | `'()` | `(cons "a" '())` |
| `(cons "a" (cons "b" (cons "c" '())))` | `"a"` | `(cons "b" (cons "c" '()))` | `(cons "c" (cons "b" '()))` | `(cons "c" (cons "b" (cons "a" '())))` |

2番目の例を見ると、`(rev (rest l))` が `(cons "c" (cons "b" '()))` を生成して、ほとんどの仕事を成し遂げていることが分かります。期待される結果は `(cons "c" (cons "b" (cons "a" '())))` なので、`rev` は再帰の結果の**末尾**に `"a"` （つまり `(first l)`）を追加する必要があります。

末尾に要素を追加する関数 `add-at-end` を「願望」し、それを使用して関数定義を完了させます。
```racket
(define (rev l)
  (cond
    [(empty? l) '()]
    [else (add-at-end (rev (rest l)) (first l))]))
```

願望リストの `add-at-end` の設計：
```racket
; Lo1s 1String -> Lo1s
; l の末尾に s を追加した新しいリストを生成する

(check-expect (add-at-end (cons "c" (cons "b" '())) "a")
              (cons "c" (cons "b" (cons "a" '()))))

(define (add-at-end l s)
  l)
```

`add-at-end` もリスト処理関数なので、そのテンプレートは次のようになります。
```racket
(define (add-at-end l s)
  (cond
    [(empty? l) ...]
    [else (... (first l) ...
           ... (add-at-end (rest l) s) ...)]))
```

最初のケース（空リスト）に対する結果は、要素 `s` のみを含むリスト、すなわち `(cons s '())` です。
2番目のケース（複雑なケース）について、構成要素が何を計算するかを考えます。
- `(first l)` はリストの最初の 1String。
- `(add-at-end (rest l) s)` は「`s` を `l` の残りの部分の末尾に追加した新しいリスト」。
ここから、再帰の結果の先頭に `(first l)` を追加し直すだけでよいことが分かります。
```racket
(define (add-at-end l s)
  (cond
    [(empty? l) (cons s '())]
    [else (cons (first l) (add-at-end (rest l) s))]))
```

テストを実行して、この関数が機能し、それによって `rev` も機能することを確認してください。もちろん、BSLには標準で任意のリストを反転させる `reverse` 関数がすでに用意されているため、実際にはそれを使用できます。

**練習問題178.** `editor-kh` のテンプレートが、長さ 1 の文字列をチェックする前に `"\t"` と `"\r"` を処理する理由を説明してください。

```racket
(define (editor-kh ed k)
  (cond
    [(key=? k "left") (editor-lft ed)]
    [(key=? k "right") (editor-rgt ed)]
    [(key=? k "\b") (editor-del ed)]
    [(key=? k "\t") ed]
    [(key=? k "\r") ed]
    [(= (string-length k) 1) (editor-ins ed k)]
    [else ed]))
```

ここで、`editor-ins` を設計します。
```racket
; Editor 1String -> Editor
; 1String k を pre と post の間に挿入する
(define (editor-ins ed k)
  ed)
```
この関数は、エディタの `pre` フィールドの先頭に `k` を追加した新しいエディタを作成するだけです。
```racket
(define (editor-ins ed k)
  (make-editor (cons k (editor-pre ed))
               (editor-post ed)))
```
`pre` が**逆順**で保持されているため、`pre` の先頭に `cons` することは、カーソル位置の直前に文字を挿入することに対応します。補助関数を使用せずに、単なる `cons` だけで挿入を実装できるのが、この表現形式の大きなメリットです。

**練習問題179.** 以下の関数を設計してください。
- `editor-lft`: カーソルを1文字分左に移動する（可能であれば）。
- `editor-rgt`: カーソルを1文字分右に移動する（可能であれば）。
- `editor-del`: カーソルの左側の文字を削除する（可能であれば）。
豊富な例を作成して、実装を検証してください。

最後に、エディタを描画する `editor-render` 関数を設計します。
`editor-render` はエディタの状態を受け取り、カーソルを挟んだ2つのテキスト画像を合成して表示します。

たとえば、`(create-editor "pre" "post")` の結果を描画するには、次のような式を使用します。
```racket
(place-image/align
  (beside (text "pre" FONT-SIZE FONT-COLOR)
          CURSOR
          (text "post" FONT-SIZE FONT-COLOR))
  1 1 "left" "top"
  MT)
```

この処理を行うためのテンプレートを記述します。
```racket
(define (editor-render e)
  (... (editor-pre e) ... (editor-post e) ...))
```
`editor-pre` と `editor-post` は 1String のリストなので、リストをテキスト画像に変換するヘルパー関数 `editor-text` を願望します。
```racket
; Editor -> Image
(define (editor-render e)
  (place-image/align
    (beside (editor-text (editor-pre e))
            CURSOR
            (editor-text (editor-post e)))
    1 1 "left" "top"
    MT))
```

`editor-text` は 1String のリストを受け取り、それをテキスト画像に変換します。
```racket
; Lo1s -> Image
; 1String のリストをテキスト画像として描画する
(define (editor-text s)
  (text (implode s) FONT-SIZE FONT-COLOR))
```

しかし、このままでテストを走らせると、`editor-render` のテストは失敗します。なぜなら、`pre` フィールドの文字リストは逆順で保持されているため、そのまま描画すると左側のテキストが逆順に表示されてしまうからです。

したがって、`editor-pre` のリストを `reverse` で元に戻してから `editor-text` に渡す必要があります。
```racket
(define (editor-render ed)
  (place-image/align
    (beside (editor-text (reverse (editor-pre ed)))
            CURSOR
            (editor-text (editor-post ed)))
    1 1 "left" "top"
    MT))
```
これでテストが成功します。

**練習問題180.** `implode` を使用せずに `editor-text` を設計してください。

**注意**
現代のアプリケーションでは、ユーザーがマウスでカーソル位置を指定できます。このような機能を追加することは可能ですが、それは第6部「アキュムレータを用いたエディタ」まで保留します。
