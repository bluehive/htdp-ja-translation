# 10 リストの更なる世界（More on Lists）

**原本:** `extracted/original_markdown_05_part_two.md`（第10章）

リストは、今やほとんどすべての言語に備わる、非常に汎用性の高いデータ形式である。プログラマはリストを使って、大規模なアプリケーション、人工知能、分散システムなどを構築してきた。この章では、この世界のいくつかのアイデアを示す。リストを生成する関数、リストの中に構造体を置くデータ表現、そしてテキストファイルをリストとして表すことなどである。

### 10.1 リストを生成する関数（Functions that Produce Lists）

以下は、時給制の従業員の賃金を求める関数である。

```racket
; Number -> Number
; computes the wage for h hours of work
(define (wage h)
  (* 12 h))
```

この関数は労働時間数を受け取り、賃金を生成する。しかし、給与計算ソフトウェアを使いたい会社は、この関数そのものには興味がない。会社が欲しいのは、すべての従業員の賃金を計算する関数である。

この新しい関数を `wage*` と呼ぼう。その役割は、すべての従業員の労働時間を処理し、それぞれに支払うべき賃金を求めることである。簡単のため、入力は数値のリスト——各数値は1人の従業員が働いた時間数を表す——であり、期待される結果も週給のリスト——やはり数値のリスト——だと仮定する。

入力と出力のデータ定義はすでに持っているので、すぐに設計の第2ステップへ進める。

```racket
; List-of-numbers -> List-of-numbers
; computes the weekly wages for the weekly hours
(define (wage* whrs)
  '())
```

次に、入力の例と対応する出力の例が必要である。そこで、週の労働時間を表す短い数値リストをいくつか作る。

| given（入力） | expected（期待） |
| :--- | :--- |
| `'()` | `'()` |
| `(cons 28 '())` | `(cons 336 '())` |
| `(cons 4 (cons 2 '()))` | `(cons 48 (cons 24 '()))` |

出力を計算するには、与えられた入力リスト上の各数値について週賃金を求める。最初の例では入力リストに数がないので、出力は `'()` である。2番目と3番目の期待出力がなぜその値になるか、理解しておくこと。

`wage*` は「リスト」の章の他のいくつかの関数と同じ種類のデータを消費する。またテンプレートはデータ定義の形だけに依存するので、次のテンプレートを再利用できる。

```racket
(define (wage* whrs)
  (cond
    [(empty? whrs)...]
    [else (... (first whrs)...
... (wage* (rest whrs))...)]))
```

テンプレートの作成を練習したい場合は、図52の質問を使うとよい。

いよいよ最も創造的な設計ステップである。設計レシピに従い、テンプレートの各 `cond` 行を個別に考える。
再帰しないケースでは `(empty? whrs)` が真であり、入力は `'()` である。上記の例が望ましい答え `'()` を指定しているので、これで済む。

2番目のケースでは、設計の質問が、テンプレートの各式が何を計算するかを述べよと指示する。

- `(first whrs)` は `whrs` 上の最初の数、すなわち最初の労働時間数を与える。
- `(rest whrs)` は与えられたリストの残りである。
- `(wage* (rest whrs))` は、残りをまさに定義中の関数で処理することを意味する。いつものように、シグネチャと目的文からこの式の結果を読み取る。シグネチャはそれが数値のリストだと教え、目的文は、そのリストが入力（時間リストの残り）に対する賃金のリストだと説明する。

肝心なのは、関数がまだ定義されていなくても、このケースで結果を計算する式を立てるときに、これらの事実に頼ることである。

`whrs` の先頭項目以外すべてに対する賃金のリストはすでに得られているので、関数は `whrs` 全体に対する期待出力を作るために2つの計算を行う必要がある。すなわち `(first whrs)` の週賃金を計算し、`whrs` のすべての週賃金を表すリストを組み立てることである。前者には `wage` を再利用する。後者には、2つの情報を `cons` して1つのリストにする。

```racket
(cons (wage (first whrs)) (wage* (rest whrs)))
```

これで定義は完成である。図64を見よ。

> **図64: Computing the wages of all employees**

```racket
; List-of-numbers -> List-of-numbers
; computes the weekly wages for all given weekly hours
(define (wage* whrs)
  (cond
    [(empty? whrs) '()]
    [else (cons (wage (first whrs)) (wage* (rest whrs)))]))

; Number -> Number
; computes the wage for h hours of work
(define (wage h)
  (* 12 h))
```

### 10.2 リスト内の構造体（Structures in Lists）

労働週を単なる数値で表すのは悪い選択である。給与小切手の印刷には、週あたりの労働時間以上の情報が要るからだ。また、すべての従業員が同じ時給とは限らない。幸いリストは原子的な値以外の項目も含める。実際、リストには望むどんな値でも、とりわけ構造体を含められる。

進行中の例は、まさにそのようなデータ表現を求めている。数値の代わりに、従業員とその労働時間・時給を表す構造体を使う。

```racket
(define-struct work [employee rate hours])
; A (piece of) Work is a structure:
;   (make-work String Number Number)
; interpretation (make-work n r h) combines the name
; with the pay rate r and the number of hours h
```

この表現はまだ単純だが、構造体を含むリストのデータ定義を立てるよう迫る分、ちょうどよい追加の挑戦になる。

```racket
; Low (short for list of works) is one of:
; – '()
; – (cons Work Low)
; interpretation an instance of Low represents the
; hours worked for a number of employees
```

以下は `Low` の3つの要素である。

```racket
'()
(cons (make-work "Robby" 11.95 39)
      '())
(cons (make-work "Matthew" 12.95 45)
      (cons (make-work "Robby" 11.95 39)
            '()))
```

データ定義を使って、これらのデータ片がなぜ `Low` に属するか説明せよ。

ストップ！ さらにデータ定義を使って、例をあと2つ作れ。

> **注:** 現実のプロジェクトでは、このような接尾辞は使わない。代わりに、コードの異なる版を管理する道具を使う。

`Low` の定義が意味をなすことが分かったので、`wage*` を再設計し、単なる数値のリストではなく `Low` の要素を消費するようにする番である。

```racket
; Low -> List-of-numbers
; computes the weekly wages for the given records
(define (wage*.v2 an-low)
  '())
```

関数名末尾の接尾辞「`.v2`」は、これが関数の第2の、改訂された版だとすべての読者に知らせる。この場合、改訂は新しいシグネチャと適合させた目的文から始まる。ヘッダ本体は上と同じである。

設計レシピの第3ステップは、例を通して考えることである。上の2番目のリストから始めよう。それは1つの work レコード、すなわち `(make-work "Robby" 11.95 39)` を含む。解釈は、`"Robby"` が39時間働き、時給は11.95ドルだということである。したがってその週の賃金は466.05ドル、すなわち `(* 11.95 39)` である。`wage*.v2` の望ましい結果はしたがって `(cons 466.05 '())` である。もちろん入力リストが2つの work レコードを含めば、この種の計算を2回行い、結果は2つの数値のリストになる。ストップ！ 上の3番目のデータ例について、期待される結果を求めよ。

**数についての注** BSLは——他のほとんどのプログラミング言語と異なり——十進数をあなたが理解するのと同じく、正確な分数として理解する点に留意せよ。たとえば Java のような言語は、最初の work レコードの期待賃金として 466.04999999999995 を生成するかもしれない。十進数に対する演算がいつこのように奇妙に振る舞うか予測できないので、そのような例は次のように書いておく方がよい。

```racket
(check-expect
  (wage*.v2
    (cons (make-work "Robby" 11.95 39) '()))
  (cons (* 11.95 39) '()))
```

他のプログラミング言語への備えにもなる。また、この書き方で例を書くことは、賃金の計算方法を本当に把握したことの証にもなる。以上。

ここからテンプレートの開発に移る。テンプレートの質問を使えば、すぐに次が得られる。

```racket
(define (wage*.v2 an-low)
  (cond
    [(empty? an-low)...]
    [(cons? an-low)
     (... (first an-low)...
... (wage*.v2 (rest an-low))...)]))
```

データ定義が2つの節からなり、第1節に `'()` を、第2節に `cons` された構造体を導入するからである、など。しかし入力について、このテンプレートが表す以上のことも分かっている。たとえば `(first an-low)` は与えられたリストから3フィールドの構造体を取り出す。これはテンプレートにさらに3つの式を加えることを示唆するように見える。

```racket
(define (wage*.v2 an-low)
  (cond
    [(empty? an-low)...]
    [(cons? an-low)
     (... (first an-low)...
...... (work-employee (first an-low))...
...... (work-rate (first an-low))...
...... (work-hours (first an-low))...
      (wage*.v2 (rest an-low))...)]))
```

このテンプレートは、潜在的に興味深いデータをすべて列挙している。

ここでは別の戦略を使う。具体的には、他のデータ定義を参照するデータ定義のためのテンプレートを作るときは、いつでも別の関数テンプレートを作り、それを参照することを勧める。

```racket
(define (wage*.v2 an-low)
  (cond
    [(empty? an-low)...]
    [(cons? an-low)
     (... (for-work (first an-low))
... (wage*.v2 (rest an-low))...)]))

; Work ->???
; a template for processing elements of Work
(define (for-work w)
  (... (work-employee w)...
... (work-rate w)...
... (work-hours w)...))
```

テンプレートを分割すると、仕事が関数へ、また関数どうしのあいだへ自然に分割される。どれも大きすぎず、すべてが特定のデータ定義と結びつく。

いよいよプログラムを書く番である。いつものように、単純に見えるケース——ここでは最初の `cond` 行——から始める。`wage*.v2` が `'()` に適用されれば `'()` が返るはずで、それで決着する。次に2行目へ進み、各式が何を計算するかを思い出す。

1. `(first an-low)` はリストから最初の work 構造体を取り出す。
2. `(for-work ...)` は work 構造体を処理する関数を設計したい、と言う。
3. `(rest an-low)` は与えられたリストの残りを取り出す。
4. `(wage*.v2 (rest an-low))` は、関数の目的文に従い、最初以外のすべての work レコードに対する賃金のリストを求める。

ここで行き詰まったら、図54の表の方法を使え。

すべて理解できれば、2つの式を `cons` すれば足りることが分かる。

```racket
... (cons (for-work (first an-low))
          (wage*.v2 (rest an-low)))...
```

ただし `for-work` が最初の work レコードの賃金を計算すると仮定する。要するに、願望リストにもう1項目加えて関数を完成させたことになる。

`for-work` はあくまで仮の名前であり、この関数には悪い名前なので、関数を `wage.v2` と呼び、完全な願望リスト項目を書こう。

```racket
; Work -> Number
; computes the wage for the given work record w
(define (wage.v2 w)
  0)
```

この種の関数の設計は「固定サイズのデータ」で詳しく扱っており、ここで追加の説明は要らない。図65は `wage.v2` と `wage*.v2` を開発した結果を示す。

> **図65: Computing the wages from work records**

```racket
; Low -> List-of-numbers
; computes the weekly wages for all weekly work records

(check-expect
  (wage*.v2 (cons (make-work "Robby" 11.95 39) '()))
  (cons (* 11.95 39) '()))

(define (wage*.v2 an-low)
  (cond
    [(empty? an-low) '()]
    [(cons? an-low) (cons (wage.v2 (first an-low))

; computes the wage for the given work record w
(define (wage.v2 w)

```

### 10.3 リストのリスト、ファイル（Lists in Lists, Files）

「関数とプログラム」は、テキストファイル全体を文字列として読む関数 `read-file` を導入する。言い換えれば、`read-file` の作者はテキストファイルを文字列として表すことを選び、関数は（名前で指定された）特定のファイルのためのデータ表現を作る。しかしテキストファイルは、単なる長い文章や文字列ではない。行と語、行とセル、その他多くの仕方で組織化されている。要するに、ファイルの内容を単なる文字列として表すのはまれにうまくいくが、通常は悪い選択である。

> **注:** 定義領域に `(require 2htdp/batch-io)` を追加せよ。

> **図66: Things take time**

```
ttt.txt
TTT
Put up in a place
where it's easy to see
the cryptic admonishment
T.T.T.
When you feel how depressingly
slowly you climb,
it's well to remember that
Things Take Time.
Piet Hein
```


具体例として、図66のサンプルファイルを見よ。それは Piet Hein の詩を含み、多くの行と語からなる。次のプログラムを使うと、

```racket
(read-file "ttt.txt")
```

このファイルを BSL の文字列にすると、次が得られる。

> **注:** 点々は、おそらく察したとおり、本当の結果の一部ではない。

```racket
"TTT\n \nPut up in a place\nwhere...."
```

ここで文字列内の `"\n"` は改行を示す。

文字列に対する原始操作——たとえば `explode`——でこの文字列を分解することは確かに可能だが、ほとんどのプログラミング言語——BSL を含む——は、ファイルのさまざまな表現と、既存のファイルからそうした表現を作る関数をサポートしている。

- 1つの表し方は行のリストであり、各行を1つの文字列として表す。
  `(cons "TTT" (cons "" (cons "Put up in a place" (cons ... '()))))`
  ここでリストの2番目の項目は空文字列である。ファイルが空行を含むからだ。
- 別の表し方は語のリストであり、やはり各語を文字列として表す。
  `(cons "TTT" (cons "Put" (cons "up" (cons "in" (cons ... '())))))`
  この表現では空の2行目が消えることに注意。結局、空行には語がない。
- 第3の表現は、語のリストのリストに頼る。
  `(cons (cons "TTT" '()) (cons '() (cons (cons "Put" (cons "up" (cons ... '()))) (cons ... '()))))`
  この表現は第2のものに対し、ファイルの組織——2行目が空であることも含む——を保つ利点がある。代償は、いきなりリストがリストを含むことである。

リストを含むリストという考えは初め恐ろしく聞こえるかもしれないが、心配は要らない。設計レシピは、そのような複雑さにも役立つ。

> **図67: Reading files**

```racket
; String -> String
; produces the content of file f as a string
(define (read-file f) ...)

; String -> List-of-string
; produces the content of file f as a list of strings,
; one per line
(define (read-lines f) ...)

; String -> List-of-string
; produces the content of file f as a list of strings,
; one per word
(define (read-words f) ...)

; String -> List-of-list-of-string
; produces the content of file f as a list of list of
; strings, one list per line and one string per word
(define (read-words/line f) ...)

; The above functions consume the name of a file as a String
; argument. If the specified file does not exist in the
; same folder as the program, they signal an error.
```


始める前に、図67を見よ。それは便利なファイル読み込み関数をいくつか紹介する。網羅的ではない。ファイルからテキストを扱う他の方法は多く、あらゆるテキストファイルに対処するにはさらに多くを知る必要がある。ここでの目的——体系的なプログラム設計の原理の教授と学習——には十分であり、それなりに興味深いプログラムを設計する力を与えてくれる。

図67は、まだ存在しない2つのデータ定義の名前を使う。リストを含むリストに関わるものも含む。いつものようにデータ定義から始めるが、今回はこの仕事をあなたに任せる。したがって先へ進む前に、次の練習問題を解け。図を完全に理解するには解答が必要であり、解答を通さなければこの節の残りを本当には理解できない。

**練習問題 171.** `List-of-strings` のデータ定義がどのようなものか、あなたは知っている。それを書き出せ。各行を文字列として表した定義のインスタンス、および各語を文字列として表したインスタンスとして、Piet Hein の詩を表現できることを確かめよ。`read-lines` と `read-words` を使って、表現の選択を確認せよ。

次に `List-of-list-of-strings` のデータ定義を開発せよ。再び、Piet Hein の詩を、各行を語ごとの文字列のリストとして表し、詩全体をそのような行表現のリストとして表した定義のインスタンスとして表現せよ。選択の確認には `read-words/line` を使ってよい。

おそらく知っているように、オペレーティングシステムにはファイルを測るプログラムが付いている。1つは行数を数え、別のものは行あたり何語あるかを求める。後者から始め、設計レシピが複雑な関数の設計にどう役立つかを示そう。

最初のステップは、必要なデータ定義をすべて揃えることである。上の練習問題を解いていれば、望む関数のすべての可能な入力のデータ定義があり、前節は可能なすべての出力を記述する `List-of-numbers` を定義している。短くするため、文字列のリストのリストのクラスを指すのに `LN` を使い、望む関数のヘッダ材料を書く。

```racket
; LN -> List-of-numbers
; determines the number of words on each line
(define (words-on-line lls) '())
```

関数を `words-on-line` と名付ける。適切であり、目的文を一句で捉えるからである。

しかし本当に必要なのは、データ例の集合である。

```racket
(define line0 (cons "hello" (cons "world" '())))
(define line1 '())

(define lls0 '())
(define lls1 (cons line0 (cons line1 '())))
```

最初の2つの定義は行の例を2つ導入する。1つは2語を含み、もう1つは語を含まない。最後の2つの定義は、これらの行例から `LN` のインスタンスをどう作るかを示す。関数にこれら2つの例を与えたときの期待結果を求めよ。

データ例ができれば、機能例を立てるのは簡単である。各データ例に関数を適用したところを想像すればよい。`words-on-line` を `lls0` に適用すれば、行がないので空リストが返るはずである。`words-on-line` を `lls1` に適用すれば、行が2つあるので2つの数値のリストが返るはずである。2つの数はそれぞれ2と0である。`lls1` の2行が2語と0語を含むからだ。

これらをすべてテストケースに直すと次のようになる。

```racket
(check-expect (words-on-line lls0) '())
(check-expect (words-on-line lls1)
              (cons 2 (cons 0 '())))
```

第2ステップの終わりでこうしておけば、完全なプログラムができている——実行すれば一部のテストケースが失敗するだけだ。

このサンプル問題で興味深いステップはテンプレートの開発である。図52のテンプレートの質問に答えれば、いつものリスト処理テンプレートがすぐに得られる。

```racket
(define (words-on-line lls)
  (cond
    [(empty? lls)...]
    [else
     (... (first lls); a list of strings
... (words-on-line (rest lls))...)]))
```

前節と同様、式 `(first lls)` が `List-of-strings` を取り出すことが分かっており、それ自体も複雑な組織を持つ。この知識を表すために入れ子のテンプレートを挿入したくなるが、思い出すべきは、第2の補助テンプレートを開発し、第2条件の最初の行をその補助テンプレートを参照するよう変える方がよい、ということである。

この補助テンプレートはリストを消費する関数用なので、前のものとほぼ同じ形になる。

```racket
(define (line-processor ln)
  (cond
    [(empty? ln)...]
    [else
     (... (first ln); a string
... (line-processor (rest ln))...)]))
```

重要な違いは、`(first ln)` がリストから文字列を取り出し、文字列を原子的な値とみなすことである。このテンプレートを手に、`words-on-line` の第2ケースの最初の行を次のように変えられる。

```racket
... (line-processor (first lls))...
```

これは第5ステップで、`words-on-line` の定義が補助関数の設計を求めるかもしれないことを思い出させる。

いよいよプログラムを書く番である。いつものように、図53の質問でこのステップを導く。行の空リストに関する第1ケースは簡単なケースである。例によれば、このケースの答えは `'()`、すなわち数値の空リストである。`cons` に関する第2ケースはいくつかの式を含み、それらが何を計算するかの確認から始める。

- `(first ln)` は（表現された）行の空でないリストから最初の行を取り出す。
- `(line-processor (first ln))` は、この行を処理する補助関数を設計したいかもしれないことを示唆する。
- `(rest ln)` は行のリストの残りである。
- `(words-on-line (rest ln))` は、リストの残りについて行ごとの語数のリストを計算する。なぜ分かるか。`words-on-line` のシグネチャと目的文でまさにそれを約束したからである。

行を消費して1行上の語数を数える補助関数——`words#` と呼ぼう——を設計できると仮定すれば、第2条件を完成させるのは簡単である。

```racket
(cons (words# (first ln))
      (words-on-line (rest ln)))
```

この式は、`ln` の最初の行上の語数を、`ln` の残りの行上の語数を表す数値のリストの先頭に `cons` する。

残るは `words#` 関数の設計である。そのテンプレートは `line-processor` と名付けられ、目的は行上の語数を数えること——行は単なる文字列のリストである。そこで願望リスト項目は次のとおり。

```racket
; List-of-strings -> Number
; counts the number of words on los
(define (words# los) 0)
```

この時点で、「自己参照データ定義を用いた設計」で設計レシピを示した例を思い出してもよい。その関数は `how-many` と呼ばれ、やはり文字列のリスト上の文字列の個数を数える。`how-many` の入力は名前のリストを表すことになっているが、その違いは問題にならない。文字列のリスト上の文字列の個数を正しく数えさえすれば、`how-many` は我々の問題を解く。

既存の関数を再利用するのはよいことなので、`words#` を次のように定義してよい。

```racket
(define (words# los)
  (how-many los))
```

しかし現実には、プログラミング言語にはそのような問題をすでに解く関数が付いている。BSL ではこの関数を `length` と呼び、値の種類を問わず、任意の値のリスト上の値の個数を数える。

### 10.4 グラフィカルエディタの再訪（A Graphical Editor, Revisited）

「グラフィカルエディタ」は、対話的な1行グラフィカルエディタの設計についてである。エディタの状態を表す2つの異なる方法を提案し、両方を探れと促す。文字列の対を含む構造体か、文字列と現在位置への添字を組み合わせた構造体である（練習問題87を見よ）。

第3の選択肢は、1String の2つのリストを組み合わせた構造体を使うことである。

```racket
(define-struct editor [pre post])
; An Editor is a structure:
;   (make-editor Lo1S Lo1S)
; An Lo1S is one of:
; – '()
; – (cons 1String Lo1S)
```

なぜか不思議に思う前に、データ例を2つ作ろう。

```racket
(define good
  (cons "g" (cons "o" (cons "o" (cons "d" '())))))
(define all
  (cons "a" (cons "l" (cons "l" '()))))
(define lla
  (cons "l" (cons "l" (cons "a" '()))))

; data example 1:
(make-editor all good)

; data example 2:
(make-editor lla good)
```

2つの例は、解釈を書き留めることがいかに重要かを示す。`editor` の2つのフィールドは明らかにカーソルの左と右の文字を表すが、2つの例は構造体型を解釈する仕方に少なくとも2つあることを示す。

1. `(make-editor pre post)` は、`pre` 内の文字がカーソルの前にあり、`post` 内の文字が後に続き、結合したテキストが
   `(string-append (implode pre) (implode post))`
   であることを意味しうる。`implode` が 1String のリストを `String` にすることを思い出そう。
2. `(make-editor pre post)` は同様に、`pre` 内の文字が逆順でカーソルの前にある、とも意味しうる。その場合、表示されるエディタ内のテキストは次のように得られる。
   `(string-append (implode (rev pre)) (implode post))`
   関数 `rev` は 1String のリストを消費し、それを逆順にしなければならない。

`rev` の完全な定義がなくても、それがどう動くかは想像できる。この理解を使い、第1のデータ例を第1の解釈に従って情報へ翻訳し、第2のデータ例を第2の解釈で扱うと、同じエディタ表示になることを確かめよ。

> [image: pict_101.png]

両方の解釈ともよい選択だが、第2のものを使うとプログラムの設計が大幅に簡単になることが分かる。この節の残りはこの点を示し、同時に構造体の中にリストを使うことを例示する。教訓を正しく味わうには、「グラフィカルエディタ」の練習問題を解いておくべきである。

データ定義を意味あるものにするために明らかに必要なので、`rev` から始めよう。そのヘッダ材料は素直である。

```racket
; Lo1s -> Lo1s
; produces a reverse version of the given list

(check-expect
  (rev (cons "a" (cons "b" (cons "c" '()))))
  (cons "c" (cons "b" (cons "a" '()))))

(define (rev l) l)
```

念のため、「明らかな」例を1つテストケースとして加えた。何が必要か確かめるため、追加の例を加えてもよい。

`rev` のテンプレートはいつものリストテンプレートである。

```racket
(define (rev l)
  (cond
    [(empty? l)...]
    [else (... (first l)...
... (rev (rest l))...)]))
```

ケースは2つあり、第2ケースにはいくつかのセレクタ式と自己参照の式がある。

テンプレートを埋めるのは第1節については簡単である。空リストの逆順版は空リストである。第2節については、再びコーディングの質問を使う。

- `(first l)` は 1String のリスト上の最初の項目である。
- `(rest l)` はリストの残りである。
- `(rev (rest l))` はリストの残りの逆順である。

ストップ！ これらのヒントで `rev` の設計を完成させよ。

```racket
; creates a new list by adding s to the end of l

(check-expect
  (add-at-end (cons "c" (cons "b" '())) "a")
  (cons "c" (cons "b" (cons "a" '()))))

(define (add-at-end l s)
  l)
```

「拡張」されているのは、テストケースとして定式化した例が付いているからである。例は `rev` の例から導かれ、実際、まさに願望リスト項目を動機づける例である。先を読む前に、`add-at-end` が空リストを消費する例を作れ。

`add-at-end` もリスト処理関数なので、テンプレートは、今ではよく知っているものの名前を変えただけである。

```racket
(define (add-at-end l s)
  (cond
    [(empty? l)...]
    [else (... (first l)...
... (add-at-end (rest l) s)...)]))
```

関数定義へ完成させるには、ステップ5のレシピの質問に従う。最初の質問は「基本」ケース——ここでは第1ケース——の答えを立てることである。提案した練習を終えていれば、次の結果が

```racket
(add-at-end '() s)
```

常に `(cons s '())` であることが分かる。結局、結果はリストでなければならず、リストは与えられた 1String を含まなければならない。

次の2つの質問は「複雑」または「自己参照」のケースに関わる。第2の `cond` 行の式が何を計算するかは分かっている。第1の式は与えられたリストから最初の 1String を取り出し、第2の式は「`s` を `(rest l)` の末尾に加えて新しいリストを作る」。つまり目的文が、ここで関数が何を生成すべきかを命じる。ここから、関数は再帰の結果に `(first l)` を戻して加えなければならないことが明らかである。

```racket
(define (add-at-end l s)
  (cond
    [(empty? l) (cons s '())]
    [else
     (cons (first l) (add-at-end (rest l) s))]))
```

例としてのテストを実行し、この関数が動き、したがって `rev` も動くことを確かめよ。もちろん、BSL がすでに任意のリスト——1String のリストを含む——を逆順にする関数を提供していることに驚くべきではない。そして当然、それは `reverse` と呼ばれる。

**練習問題 177.** 関数 `create-editor` を設計せよ。この関数は2つの文字列を消費し、`Editor` を生成する。最初の文字列はカーソルの左のテキスト、2番目は右のテキストである。この節の残りはこの関数に依存する。

この時点で、1行グラフィカルエディタのデータ表現を完全に理解しているはずである。「ワールドプログラムの設計」の対話型プログラムの設計戦略に従い、物理定数——たとえばエディタの幅と高さ——と、グラフィカル定数——たとえばカーソル——を定義すべきである。我々のものは次のとおり。

```racket
(define HEIGHT 20); the height of the editor
(define WIDTH 200); its width
(define FONT-SIZE 16); the font size
(define FONT-COLOR "black"); the font color

(define MT (empty-scene WIDTH HEIGHT))
(define CURSOR (rectangle 1 HEIGHT "solid" "red"))
```

しかし重要なのは、イベントハンドラとエディタの状態を描く関数の願望リストを書き留めることである。2htdp/universe ティーチパックがこれらの関数のヘッダ材料を定めることを思い出そう。

```racket
; Editor -> Image
; renders an editor as an image of the two texts
; separated by the cursor
(define (editor-render e) MT)

; Editor KeyEvent -> Editor
; deals with a key event, given some editor
(define (editor-kh ed ke) ed)
```

加えて、「ワールドプログラムの設計」はプログラムの main 関数を書き下ろすよう求める。

```racket
; main: String -> Editor
; launches the editor given some initial string
(define (main s)
   (big-bang (create-editor s "")
     [on-key editor-kh]
     [to-draw editor-render]))
```

このプログラムの初期エディタを決めるため、練習問題177を再読せよ。

次にどの願望に取り組むかはどちらでもよいが、我々はまず `editor-kh` を、次に `editor-render` を設計する。ヘッダ材料はあるので、キーイベントハンドラの働きを2つの例で説明しよう。

```racket
(check-expect (editor-kh (create-editor "" "") "e")
              (create-editor "e" ""))
(check-expect
  (editor-kh (create-editor "cd" "fgh") "e")
  (create-editor "cde" "fgh"))
```

これら2つの例は、キーボードで文字「e」を押したときに何が起きるかを示す。コンピュータはエディタの現在状態と `"e"` に対して関数 `editor-kh` を実行する。最初の例ではエディタは空なので、結果は文字 `"e"` だけがありその後にカーソルがあるエディタである。2番目の例では、カーソルは文字列 `"cd"` と `"fgh"` のあいだにあり、したがって結果はカーソルが `"cde"` と `"fgh"` のあいだにあるエディタである。要するに、関数は任意の通常の文字を常にカーソル位置へ挿入する。

先を読む前に、バックスペース（`"\b"`）キーで文字を消すとき、`"left"` と `"right"` 矢印キーでカーソルを動かすとき、あるいは他の矢印キーを押すときに `editor-kh` がどう動くかを示す例を作れ。いずれの場合も、エディタが空のとき、カーソルが空でない文字列の左端・右端にあるとき、中央にあるときに何が起きるべきかを考えよ。ここでは区間を扱っていないが、「極端な」ケースの例を作るのは依然よい考えである。

テストケースができたら、テンプレートを開発する番である。`editor-kh` の場合、2つの複雑な形のデータを消費する関数を扱っている。1つはリストを含む構造体、もう1つは文字列の大きな列挙である。一般にこの設計ケースは改良された設計レシピを求めるが、このような場合、まず入力の一方——すなわちキー入力——を扱うべきことも明らかである。

そう言った上で、テンプレートは関数が受け取った `KeyEvent` を検査する大きな `cond` 式にすぎない。

```racket
(define (editor-kh ed k)
  (cond
    [(key=? k "left")...]
    [(key=? k "right")...]
    [(key=? k "\b")...]
    [(key=? k "\t")...]
    [(key=? k "\r")...]
    [(= (string-length k) 1)...]
    [else...]))
```

この `cond` 式は `KeyEvent` のデータ定義と完全には一致しない。一部の `KeyEvent` は特別な注意を要し（`"left"`、`"\b"` など）、一部は特別なので無視する必要があり（`"\t"` と `"\r"`）、一部は1つの大きなグループ（通常キー）に分類すべきだからである。

**練習問題 178.** `editor-kh` のテンプレートが、長さ1の文字列を検査する前に `"\t"` と `"\r"` を扱う理由を説明せよ。

第5ステップ——関数の定義——では、条件の各節を別々に扱う。第1節は、カーソルを動かしエディタの文字列内容はそのままにする結果を求める。第2節も同様である。しかし第3節は、文字があればエディタの内容から文字を削除することを求める。最後に、第6の `cond` 節はカーソル位置への文字の追加に関わる。最初の基本指針に従い、願望リストを広く使い、タスクごとに1関数を想像する。

```racket
(define (editor-kh ed k)
  (cond
    [(key=? k "left") (editor-lft ed)]
    [(key=? k "right") (editor-rgt ed)]
    [(key=? k "\b") (editor-del ed)]
    [(key=? k "\t") ed]
    [(key=? k "\r") ed]
    [(= (string-length k) 1) (editor-ins ed k)]
    [else ed]))
```

`editor-kh` の定義から分かるように、4つの願望リスト関数のうち3つは同じシグネチャを持つ。

```racket
; Editor -> Editor
```

最後の1つは引数を1つではなく2つ取る。

```racket
; Editor 1String -> Editor
```

最初の3関数の願望の正式な定式化はあなたに任せ、4番目に焦点を当てる。

目的文と関数ヘッダから始めよう。

```racket
; insert the 1String k between pre and post
(define (editor-ins ed k)
  ed)
```

目的は問題文からそのまま取れる。関数ヘッダの構成には `Editor` のインスタンスが要る。`pre` と `post` は現在のものの部品なので、それらを単に戻して組み合わせる。

次に `editor-ins` の例を `editor-kh` のものから導く。

```racket
(check-expect
  (editor-ins (make-editor '() '()) "e")
  (make-editor (cons "e" '()) '()))

(check-expect
  (editor-ins
    (make-editor (cons "d" '())
                 (cons "f" (cons "g" '())))
    "e")
  (make-editor (cons "e" (cons "d" '()))
               (cons "f" (cons "g" '()))))
```

これらの例を `Editor` の解釈を使って辿れ。つまり、与えられたエディタが情報として何を意味し、関数呼び出しがそれらの言葉で何を達成すべきかを理解せよ。この場合は、情報をよく表すので、エディタの視覚的表現を描くのが最善である。

第4ステップはテンプレートの開発を求める。第1引数は構造体であることが保証され、第2は文字列、すなわち原子的なデータ片である。言い換えれば、テンプレートは与えられたエディタ表現から部品を取り出すだけである。

```racket
(define (editor-ins ed k)
  (... ed... k...
... (editor-pre ed)...
... (editor-post ed)...))
```

テンプレートがパラメータを列挙するのは、それらも利用可能だからであることを思い出せ。

テンプレートと例から、`editor-ins` は与えられたエディタの `pre` と `post` フィールドからエディタを作り、前者の先頭に `k` を加えるべきだと比較的容易に結論できる。

```racket
(define (editor-ins ed k)
  (make-editor (cons k (editor-pre ed))
               (editor-post ed)))
```

`(editor-pre ed)` も `(editor-post ed)` も 1String のリストだが、補助関数を設計する必要はない。望ましい結果を得るには、リストを作る `cons` を使えば足りる。

この時点で2つのことをすべきである。第一に、この関数のテストを実行せよ。第二に、`Editor` の解釈を使い、なぜこの関数が挿入を行うかを抽象的に説明せよ。それでも足りなければ、この単純な定義を練習問題84のものと比較し、なぜあちらは補助関数が要りこちらは要らないかを考えよ。

**練習問題 179. 次の関数を設計せよ。**

```racket
; Editor -> Editor
; moves the cursor position one 1String left,
; if possible
(define (editor-lft ed) ed)

; Editor -> Editor
; moves the cursor position one 1String right,
; if possible
(define (editor-rgt ed) ed)

; Editor -> Editor
; deletes a 1String to the left of the cursor,
; if possible
(define (editor-del ed) ed)
```

再び、幅広い例を通すことが重要である。

`Editor` の描画関数の設計は、新しいが小さな課題をいくつか課す。最初は、十分に多くのテストケースを開発することである。一方では、可能な組み合わせの覆いが要る。カーソルの左が空文字列、右が空、両方が空である。他方では、2htdp/image ティーチパックが提供する関数での実験も要る。具体的には、テキスト画像として描かれた2つの文字列片を合成する方法と、テキスト画像を空の画像フレーム（`MT`）へ置く方法が要る。`(create-editor "pre" "post")` の結果の画像を作るために我々が行うことは次のとおり。

```racket
(place-image/align
  (beside (text "pre" FONT-SIZE FONT-COLOR)
          CURSOR
          (text "post" FONT-SIZE FONT-COLOR))
  1 1
  "left" "top"
  MT)
```

これを上のエディタ画像と比べると違いに気づくが、それで構わない。正確なレイアウトはこの練習の目的に本質的ではなく、改訂されたレイアウトが問題を自明にしているわけでもないからである。いずれにせよ、DrRacket の対話領域で実験し、好みのエディタ表示を見つけよ。

これでテンプレートを開発する準備が整い、次程度のものが得られるはずである。

```racket
(define (editor-render e)
  (... (editor-pre e)... (editor-post e)))
```

与えられた引数は2フィールドの構造体型にすぎない。しかしそれらの値は 1String のリストであり、テンプレートをさらに洗練したくなるかもしれない。するな！ 代わりに、あるデータ定義が別の複雑なデータ定義を参照するときは、願望リストを使う方がよいことを心に留めよ。

十分な数の例を通していれば、願望リストに何が欲しいかも分かる。文字列を適切なサイズと色のテキストにする1つの関数である。この関数を `editor-text` と呼ぼう。すると `editor-render` の定義は、`editor-text` を2回使い、結果を `beside` と `place-image` で合成するだけである。

```racket
; Editor -> Image
(define (editor-render e)
  (place-image/align
    (beside (editor-text (editor-pre e))
            CURSOR
            (editor-text (editor-post e)))
    1 1
    "left" "top"
    MT))
```

この定義は式を3段入れ子にしているが、想像上の `editor-text` 関数の使用でかなり読みやすくなっている。

残るは `editor-text` の設計である。`editor-render` の設計から、`editor-text` は 1String のリストを消費しテキスト画像を生成することが分かる。

```racket
; Lo1s -> Image
; renders a list of 1Strings as a text image
(define (editor-text s)
  (text "" FONT-SIZE FONT-COLOR))
```

このダミー定義は空のテキスト画像を生成する。

`editor-text` が何を計算すべきかを示すため、例を1つ辿る。例の入力は

```racket
(create-editor "pre" "post")
```

これは `editor-render` の説明にも使ったもので、次と等価である。

```racket
(make-editor
  (cons "e" (cons "r" (cons "p" '())))
  (cons "p" (cons "o" (cons "s" (cons "t" '())))))
```

2番目のリストを `editor-text` のサンプル入力に取り、`editor-render` の例から期待結果が分かる。

```racket
(check-expect
  (editor-text
   (cons "p" (cons "o" (cons "s" (cons "t" '())))))
  (text "post" FONT-SIZE FONT-COLOR))
```

先を読む前に、2つ目の例を作っておくとよい。

`editor-text` が 1String のリストを消費するので、テンプレートは手間なく書ける。

```racket
(define (editor-text s)
  (cond
    [(empty? s)...]
    [else (... (first s)
... (editor-text (rest s))...)]))
```

結局、テンプレートは関数入力を記述するデータ定義が命じる。しかし `Editor` の解釈を理解し心に留めていれば、テンプレートは要らない。それは文字列を 1String のリストへするために `explode` を使う。当然、逆の計算を行う関数 `implode` があり、すなわち

```racket
> (implode   (cons "p" (cons "o" (cons "s" (cons "t" '())))))
"post"
```

この関数を使えば、`editor-text` の定義は例から関数本体への小さな一歩にすぎない。

```racket
(define (editor-text s)
  (text (implode s) FONT-SIZE FONT-COLOR))
```

**練習問題 180.** `implode` を使わずに `editor-text` を設計せよ。

2つの関数をテストすると、本当の驚きが来る。`editor-text` のテストは成功するが、`editor-render` のテストは失敗する。失敗を調べると、カーソルの左の文字列——`"pre"`——が後ろ向きに組版されている。エディタ状態のこの部分が逆順で表されていることを忘れていたのである。幸い、2つの関数の単体テストは、どの関数が間違っているかを特定し、何が間違っているか、どう直せばよいかまで教えてくれる。

```racket
(define (editor-render ed)
  (place-image/align
    (beside (editor-text (reverse (editor-pre ed)))
            CURSOR
            (editor-text (editor-post ed)))
    1 1
    "left" "top"
    MT))
```

この定義は `ed` の `pre` フィールドに `reverse` 関数を使う。

**注** 現代のアプリケーションは、ユーザーがマウス（や他のジェスチャ機器）でカーソル位置を指定できる。原理的にはこの能力をエディタに加えることは可能だが、「マウス付きグラフィカルエディタ」まで保留する。
