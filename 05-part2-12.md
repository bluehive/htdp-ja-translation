# 12 プロジェクト: リスト（Projects: Lists）

**原本:** `extracted/original_markdown_05_part_two.md`（第12章）

この章では、設計の要素に対する理解をより確固たるものにすることを目指した、いくつかの拡張的な練習問題を紹介します。対象となるのは、バッチプログラムと対話的プログラムの設計、合成による設計、願望リスト（ウィッシュリスト）の設計、そして関数の設計レシピです。最初のセクションでは、英語の辞書や iTunes ライブラリといった現実世界のデータを扱う問題を取り上げます。ワードゲーム（単語パズル）の問題には 2 つのセクションが必要です。1 つは合成による設計を示すため、もう 1 つは問題の核心に取り組むためです。残りのセクションは、ゲームと有限状態機械についてです。

> **注意:** この章は `2htdp/batch-io` ティーチパックに依存します。

## 12.1 現実世界のデータ: 辞書（Real-World Data: Dictionaries）

現実世界の情報は大量に存在する傾向があり、だからこそプログラムで処理することに大きな意味があります。たとえば、辞書には十数語だけでなく、何十万もの単語が含まれます。このような大量の情報を処理したいときは、小さな例を使ってプログラムを慎重に設計しなければなりません。プログラムが正しく動くと自分で納得できたら、現実世界のデータに対して実行して実際の結果を得ます。プログラムがこの大量のデータを処理するには遅すぎる場合は、各関数とその動作を振り返り、冗長な計算を取り除けないかを問いかけてください。

> **注意:** 性能に関する懸念については、「生成的再帰」を参照してください。ここからそこまでは、体系的にプログラムを設計することに焦点を当てます。そうすれば、その後で性能問題を適切に探れるようになります。

**図74: 辞書の読み込み（Reading a dictionary）**
```
; On OS X:
(define LOCATION "/usr/share/dict/words")
; On LINUX: /usr/share/dict/words or /var/lib/dict/words
; On WINDOWS: borrow the word file from your Linux friend

; A Dictionary is a List-of-strings.
(define AS-LIST (read-lines LOCATION))
```

図74は、英語の辞書全体を読み込むために必要な 1 行のコードを示しています。そのような辞書がどれほど大きいかを知るには、図のコードを自分のコンピュータ向けに調整し、`length` を使って辞書にいくつの単語があるかを調べてください。著者らの辞書には、2017年7月25日時点で 235,886 語がありました。

以下の練習問題では、文字（letter）が重要な役割を果たします。図74の調整に加えて、プログラムの先頭に次を追加するとよいでしょう。

```racket
; A Letter is one of the following 1Strings:
; – "a"
; –...
; – "z"
; or, equivalently, a member? of this list:
(define LETTERS
  (explode "abcdefghijklmnopqrstuvwxyz"))
```

ヒント: 練習問題の例やテストを定式化するときは `list` を使ってください。

**練習問題195.** 関数 `starts-with#` を設計してください。この関数は `Letter` と `Dictionary` を受け取り、与えられた辞書のうち、与えられた文字で始まる単語の個数を数えます。関数が正しく動くと分かったら、自分のコンピュータの辞書で `"e"` で始まる単語がいくつあるか、また `"z"` で始まる単語がいくつあるかを調べてください。

**練習問題196.** `count-by-letter` を設計してください。この関数は `Dictionary` を受け取り、与えられた辞書の中で各文字が単語の先頭文字として何回使われているかを数えます。結果は `Letter-Count` のリストです。`Letter-Count` は文字とカウントを組み合わせたデータです。

関数の設計が終わったら、自分のコンピュータの辞書について、すべての文字に対して何語あるかを調べてください。

**練習問題197.** `most-frequent` を設計してください。この関数は `Dictionary` を受け取り、与えられた辞書の中で単語の先頭文字として最も頻繁に現れる文字についての `Letter-Count` を生成します。

自分のコンピュータの辞書で最も頻繁に使われる先頭文字は何で、それは何回使われますか？

*設計の選択肢に関する注意* この練習問題では、直前の練習問題の解と、`Letter-Count` のリストから正しいペアを選ぶ関数との合成が求められます。後者の関数の設計には 2 通りあります。

- カウントが最大のペアを選ぶ関数を設計する。
- ソート済みのペアのリストから先頭を選ぶ関数を設計する。

両方の設計を検討してください。どちらが好みですか？なぜですか？

**練習問題198.** `words-by-first-letter` を設計してください。この関数は `Dictionary` を受け取り、文字ごとに 1 つずつの `Dictionary` からなるリストを生成します。

この新しい関数を使って、練習問題197の `most-frequent` を再設計してください。新しい関数を `most-frequent.v2` と呼びます。設計を終えたら、2 つの関数が自分のコンピュータの辞書に対して同じ結果を計算することを確認してください。

```racket
(check-expect
  (most-frequent AS-LIST)
  (most-frequent.v2 AS-LIST))
```

*設計の選択肢に関する注意* `words-by-first-letter` では、与えられた辞書に特定の文字の単語が 1 つもない場合の扱いについて選択肢があります。

- 1 つの代替案は、結果として生じる空の辞書を全体の結果から除外することです。こうすると関数のテストも `most-frequent.v2` の設計も簡単になりますが、補助関数の設計が必要になります。
- もう 1 つは、その文字の単語を探した結果として、たとえ 1 つもなくても `'()` を含めることです。この代替案は最初の案に必要な補助関数を避けますが、`most-frequent.v2` の設計に複雑さが加わります。

*中間データとデフォレステーションに関する注意* 単語カウント関数のこの第 2 版は、部分ごとに数えられる以外に実質的な目的のない大きな中間データ構造を作ることで、望ましい結果を計算します。場合によっては、プログラミング言語が 2 つの関数を 1 つに融合することでそれらを自動的に取り除きます。プログラムに対するこの変換はデフォレステーション（deforestation）とも呼ばれます。言語がプログラムをデフォレストしないと分かっているときは、プログラムがデータを十分速く処理しないなら、そのようなデータ構造を取り除くことを検討してください。

## 12.2 現実世界のデータ: iTunes（Real-World Data: iTunes）

Apple の iTunes ソフトウェアは、音楽、ビデオ、テレビ番組などを集めるために広く使われています。自分の iTunes アプリケーションが集めた情報を分析したくなるかもしれません。そのデータベースを取り出すのは実はかなり簡単です。アプリケーションの `File` メニューを選び、`Library`、次に `Export` を選ぶと—これで、iTunes 情報のいわゆる XML 表現をエクスポートできます。XML の処理は「プロジェクト: XML の商取引」でいくらか詳しく扱います。ここでは `2htdp/itunes` ティーチパックに頼って情報を手に入れます。具体的には、ティーチパックにより iTunes ライブラリに含まれる音楽トラックを取得できます。

> **注意:** このようなプロプライエタリな形式は定期的に変わるため、本書には 2 つのサンプルファイルが付属しています: `itunes.xml` と `itunes2.xml`。自分のファイルの代わりにこれらを使っても構いません。

詳細は異なりますが、iTunes ライブラリは各音楽トラックについて次のような種類の情報の一部を保持します（場合によってはもう少し少なくなります）。

- **Track ID** — ライブラリに対するトラックの一意の識別子。例: `442`
- **Name** — トラックのタイトル。`Wild Child`
- **Artist** — 制作アーティスト。`Enya`
- **Album** — 所属アルバムのタイトル。`A Day Without`
- **Genre** — トラックに割り当てられた音楽ジャンル。`New Age`
- **Kind** — 音楽のエンコーディング。MPEG audio file
- **Size** — ファイルサイズ。`4562044`
- **Total Time** — トラックの長さ（ミリ秒）。`227996`
- **Track Number** — アルバム内でのトラック位置。`2`
- **Track Count** — アルバムのトラック数。`11`
- **Year** — リリース年。`2000`
- **Date Added** — トラックが追加されたとき。`2002-7-17 3:55:14`
- **Play Count** — 再生回数。`20`
- **Play Date** — 最後に再生されたとき。`3388484113` Unix 秒
- **Play Date UTC** — 最後に再生されたとき。`2011-5-17 17:35:13`

いつものように、最初の仕事はこの情報に対する BSL のデータ表現を選ぶことです。このセクションでは、音楽トラックについて 2 つの表現を使います。構造体に基づくものと、リストに基づくものです。前者はトラックあたり固定個数の属性を記録し、すべての情報が揃っている場合にのみ使えます。後者は、利用可能な情報をデータとして表したものが何であれ付いてきます。それぞれ特定の用途に向いており、用途によっては両方の表現が役立ちます。

> **注意:** `2htdp/batch-io` ティーチパックに加えて、このセクションは `2htdp/itunes` ティーチパックに依存します。

**図75: iTunes トラックの構造体表現（構造体）（Representing iTunes tracks as structures (the structures)）**
```
; the 2htdp/itunes teachpack documentation, part 1:

; An LTracks is one of:
; – '()
; – (cons Track LTracks)

(define-struct track
  [name artist album time track# added play# played])
; A Track is a structure:
; (make-track String String String N N Date N Date)
; interpretation An instance records in order: the track's
; title, its producing artist, to which album it belongs,
; its playing time in milliseconds, its position within the
; album, the date it was added, how often it has been
; played, and the date when it was last played

(define-struct date [year month day hour minute second])
; A Date is a structure:
; (make-date N N N N N N)
; interpretation An instance records six pieces of information:
; the date's year, month (between 1 and 12 inclusive),
; day (between 1 and 31), hour (between 0
; and 23), minute (between 0 and 59), and
; second (also between 0 and 59).
```

**図76: iTunes トラックの構造体表現（関数）（Representing iTunes tracks as structures (the functions)）**
```
; Any Any Any Any Any Any Any Any -> Track or #false
; creates an instance of Track for legitimate inputs
; otherwise it produces #false
(define (create-track name artist album time
                      track# added play# played)
...)

; Any Any Any Any Any Any -> Date or #false
; creates an instance of Date for legitimate inputs
; otherwise it produces #false
(define (create-date y mo day h m s)
...)

; String -> LTracks
; creates a list-of-tracks representation from the
; text in file-name (an XML export from iTunes)
(define (read-itunes-as-tracks file-name)
...)
```

図75と図76は、`2htdp/itunes` ティーチパックが実装するトラックの構造体ベースの表現を紹介します。`track` 構造体型には 8 つのフィールドがあり、それぞれがトラックの特定の性質を表します。ほとんどのフィールドは `String` や `N` のようなアトミックな種類のデータを含みます。他のフィールドは `Date` を含みます。`Date` は 6 つのフィールドを持つ構造体型です。`2htdp/itunes` ティーチパックは `track` と `date` 構造体型のすべての述語とセレクタをエクスポートしますが、コンストラクタの代わりに検査付きコンストラクタを提供します。

`2htdp/itunes` ティーチパックの説明の最後の要素は、iTunes の XML ライブラリ記述を読み込み、トラックのリスト `LTracks` を返す関数です。ある iTunes アプリから XML ライブラリをエクスポートしたら、次のコード片を実行してすべてのレコードを取得できます。

```racket
; modify the following to use your chosen name
(define ITUNES-LOCATION "itunes.xml")

; LTracks
(define itunes-tracks
  (read-itunes-as-tracks ITUNES-LOCATION))
```

このスニペットを、iTunes の XML エクスポートと同じフォルダに保存してください。例のために `itunes-tracks` を使わないことを覚えておいてください。それはその目的にははるかに大きすぎます。実際、毎回 BSL プログラムを DrRacket で実行するたびにファイルを読むのにかなり時間がかかるほど大きいかもしれません。したがって、関数を設計しているあいだは、この 2 行目をコメントアウトしたくなるかもしれません。自分の iTunes コレクションについての情報を計算したいときだけコメントを外してください。

**練習問題199.** 重要なデータ定義はすでに与えられていますが、設計レシピの第 1 ステップはまだ不完全です。`Date`、`Track`、`LTracks` の例を作ってください。これらの例は、以下の練習問題の入力として役立ちます。

**図77: iTunes トラックのリスト表現（Representing iTunes tracks as lists）**
```
; the 2htdp/itunes teachpack documentation, part 2:

; An LLists is one of:
; – '()
; – (cons LAssoc LLists)

; An LAssoc is one of:
; – '()
; – (cons Association LAssoc)
;
; An Association is a list of two items:
; (cons String (cons BSDN '()))

; A BSDN is one of:
; – Boolean
; – Number
; – String
; – Date

; String -> LLists
; creates a list of lists representation for all tracks in
; file-name, which must be an XML export from iTunes
(define (read-itunes-as-lists file-name)
...)
```

図77は、`2htdp/itunes` ティーチパックがリストでトラックを表す方法を示しています。`LLists` はトラック表現のリストであり、各トラック表現は `String` と 4 種類の値をペアにしたリストのリストです。関数 `read-itunes-as-lists` は iTunes の XML ライブラリを読み込み、`LLists` の要素を生成します。したがって、次の定義をプログラムに追加すれば、すべてのトラック情報にアクセスできます。

```racket
; modify the following to use your chosen name
(define ITUNES-LOCATION "itunes.xml")

; LLists
(define list-tracks
  (read-itunes-as-lists ITUNES-LOCATION))
```

そして、それを iTunes ライブラリが保存されているのと同じフォルダに保存してください。

*注意* リストベースの表現は、構造体ベースのものより少し整理されていません。この文脈では半構造化（semi-structured）という言葉がときどき使われます。そのようなリスト表現は、まれにしか現れず構造体型には収まらない性質を収容します。人々はしばしば未知の情報を探るためにそのような表現を使い、形式がよく分かってから構造体を導入します。可能なとき `LAssoc` を `Track` に変換する関数 `track-as-struct` を設計してください。

## 12.3 ワードゲーム、合成の例示（Word Games, Composition Illustrated）

新聞や雑誌の単語パズルを解く人もいるでしょう。次を試してみてください。

> **サンプル問題** ある単語が与えられたとき、同じ文字からできているすべての単語を見つけよ。たとえば “cat” は “act” とも綴れる。

例を通して考えてみましょう。「dear」が与えられたとします。4 文字の可能な並び替えは 24 通りあります。

> ader aedr aerd adre arde ared daer eadr eard dare rade raed dear edar erad drae rdae read dera edra erda drea rdea reda

このリストには、正当な英単語が 3 つあります。「read」「dear」「dare」です。

*注意* 単語に同じ文字が 2 回含まれると、すべての並び替えの集まりに同じ文字列のコピーがいくつか含まれることがあります。私たちの目的では、これは許容できます。現実的なプログラムでは、リストの代わりに集合を使って重複エントリを避けたくなるかもしれません。「リストと集合に関する注意」を参照してください。

可能なすべての並び替えを体系的に列挙することは明らかにプログラムの仕事であり、英語辞書での検索も同様です。このセクションでは検索関数の設計を扱い、もう 1 つの問題の解は次のセクションに残します。2 つを分けることで、この最初のセクションは体系的なプログラム設計の高水準の考えに集中できます。

> **注意:** 現実世界の辞書の扱いについては「現実世界のデータ: 辞書」を参照してください。

この問題を手でどう解くかを少し想像してみましょう。十分な時間があれば、与えられた単語のすべての文字の可能な並び替えを列挙し、辞書にも現れる変種だけを選べばよいでしょう。明らかにプログラムもこのように進められ、これは合成による自然な設計を示唆します。しかし、いつものように体系的に進め、入力と出力のデータ表現を選ぶところから始めます。

少なくとも一見したところでは、単語を `String` として、結果を単語のリストまたは `List-of-strings` として表すのが自然です。この選択に基づき、シグネチャと目的ステートメントを定式化できます。

```racket
; String -> List-of-strings
; finds all words that use the same letters as s
(define (alternative-words s)
...)
```

次に、いくつかの例が必要です。与えられた単語が “cat” なら、3 文字 *c*、*a*、*t* を扱っています。少し弄ると、これらの文字の 6 通りの並びが示唆されます: *cat*、*cta*、*tca*、*tac*、*act*、*atc*。このうち実際の単語は 2 つです。「cat」と「act」です。`alternative-words` は `String` のリストを生成するので、結果の表し方は 2 通りあります: `(list "act" "cat")` と `(list "cat" "act")` です。幸い BSL には、関数が 2 つの可能な結果のどちらかを返すと言う方法があります。

```racket
(check-member-of (alternative-words "cat")
                 (list "act" "cat")
                 (list "cat" "act"))
```

ストップ！ ドキュメントで `check-member-of` について読んでください。

この例を通すと 2 つの問題が露呈します。

- 1 つ目はテストについてです。仮に単語 “rat” を使ったとすると、代替は 3 つあります。「rat」「tar」「art」です。この場合、関数の結果になりうるリストを 6 つ定式化しなければなりません。4 つの可能な代替がある “dear” のような単語では、テストの定式化はさらに難しくなります。
- 2 つ目の問題は、単語表現の選択に関するものです。`String` は最初は自然に見えますが、例から分かるように、関数の一部は単語を文字の列として見なければならず、それらを自由に並べ替えられる必要があります。`String` 内の文字を並べ替えることも可能ですが、この目的には明らかに文字のリストの方が適しています。

これらの問題を 1 つずつ扱い、テストから始めます。

`alternative-words` と `"rat"` のテストを定式化したいとします。上から、結果は `"rat"`、`"tar"`、`"art"` を含まなければならないことは分かりますが、これらの単語が結果のどの順で現れるかは分かりません。

この状況では `check-satisfied` が便利です。`String` のリストが私たちの 3 つの `String` を含むかどうかを検査する関数とともに使えます。

> **注意:** 「間奏曲 1: Beginning Student Language」を参照してください。

```racket
  (and (member? "rat" w)
       (member? "art" w)
       (member? "tar" w)))
```

## 12.4 ワードゲーム、問題の核心（Word Games, the Heart of the Problem）

目標は `arrangements` を設計することです。これは `Word` を受け取り、その単語の文字ごとの並び替えのリストを生成する関数です。この拡張的な練習問題は、深い願望リストの必要性を強化します。つまり、関数を 1 つ終えるたびに増えていくように見える、望ましい関数のリストです。

> **注意:** 数学用語では順列（permutations）です。

述べたように、`String` は単語の表現として使えそうですが、`String` はアトミックであり、`arrangements` がその文字を並べ替える必要があるという事実自体が、別の表現を求めています。したがって、私たちが選んだ単語のデータ表現は `1String` のリストであり、入力の各項目が 1 文字を表します。

```racket
; A Word is one of:
; – '() or
; – (cons 1String Word)
; interpretation a Word is a list of 1Strings (letters)
```

**練習問題212.** `List-of-words` のデータ定義を書き下してください。`Word` と `List-of-words` の例を作ってください。最後に、上の関数例を `check-expect` で定式化してください。完全な例の代わりに、たった 2 文字、たとえば `"d"` と `"e"` の単語で作業することを検討してください。

`arrangements` のテンプレートは、リスト処理関数のそれです。

```racket
; Word -> List-of-words
; creates all rearrangements of the letters in w
(define (arrangements w)
  (cond
    [(empty? w)...]
    [else (... (first w)...
... (arrangements (rest w))...)]))
```

第 5 ステップの準備として、テンプレートの `cond` 行を見てみましょう。

1. 入力が `'()` なら、入力の可能な並び替えは 1 つだけです。空の単語 `'()` です。したがって結果は `(list '())`、つまり空リストを唯一の項目として含むリストです。
2. そうでなければ、単語には最初の文字があり、`(first w)` がその文字です。また、再帰は単語の残りの部分についての可能なすべての並び替えのリストを生成します。たとえば、リストが `(list "d" "e" "r")` なら、再帰は `(arrangements (list "e" "r"))` です。それは結果
   `(cons (list "e" "r") (cons (list "r" "e") '()))`
   を生成します。リスト全体についての可能なすべての並び替えを得るには、今、最初の項目（私たちの場合は `"d"`）を、これらのすべての単語の、可能なすべての文字のあいだ、および先頭と末尾に挿入しなければなりません。

私たちの分析は、1 つの文字を多くの異なる単語のすべての位置に挿入できれば、`arrangements` を完成できることを示唆します。このタスク記述の最後の側面は暗黙にリストに言及しており、この章の助言に従って、補助関数を求めます。この関数を `insert-everywhere/in-all-words` と呼び、それを使って `arrangements` の定義を完成させましょう。

```racket
(define (arrangements w)
  (cond
    [(empty? w) (list '())]
    [else (insert-everywhere/in-all-words (first w)
            (arrangements (rest w)))]))
```

**練習問題213.** `insert-everywhere/in-all-words` を設計してください。これは `1String` と単語のリストを受け取ります。結果は第 2 引数のような単語のリストですが、与えられたリストのすべての単語の先頭、すべての文字のあいだ、および末尾に第 1 引数が挿入されています。

完全な願望リスト項目から始めてください。空リスト、1 文字の単語のリスト、2 文字の単語の別のリストなどについてのテストでそれを補ってください。続ける前に、次の 3 つのヒントを注意深く研究してください。

ヒント (1) 上の例を再考してください。それは `"d"` を単語 `(list "e" "r")` と `(list "r" "e")` に挿入する必要があると言っています。したがって次の適用は自然な例の候補の 1 つです。

```racket

  (cons (list "e" "r")
    (cons (list "r" "e")
      '())))
```

(2) BSL+ の操作 `append` を使いたくなるでしょう。これは 2 つのリストを受け取り、2 つのリストの連結を生成します。

```racket

(list "a" "b" "c" "d" "e")
```

(3) この練習問題の解は一連の関数です。忍耐強く設計レシピに従い、願望リストを体系的に片付けてください。

**練習問題214.** `arrangements` を「ワードゲーム、合成の例示」の部分的なプログラムと統合してください。テスト一式がすべて通ることを確認したあと、お気に入りの例のいくつかで実行してください。

ヒント (1) 「食べる」を解釈する 1 つの方法は、頭が食べ物があった場所へ動き、尾が頭があった場所に挿入されて 1 セグメント伸びる、と言うことです。なぜこの解釈は関数として設計しやすいのでしょうか？(2) この最後のステップでは、`worm-main` 関数に 2 番目のパラメータを追加すると便利だと分かりました。`big-bang` が世界の現在状態を別ウィンドウに表示するかどうかを決める真偽値です。この情報をどう求めるかは `state` のドキュメントを参照してください。

この最後の練習問題を終えたら、完全なワームゲームができています。次に `worm-main` 関数を変更して、最終的なワームの長さを返すようにしてください。それから DrRacket の `Create Executable`（`Racket` メニューの下）を使って、BSL+ を知っている人だけでなく誰でも起動できるものにプログラムを変えてください。

ゲームに余分なひねりを加えて、本当に自分のゲームにしたくなるかもしれません。私たちは面白いゲーム終了メッセージ、複数の異なる食べ物のかけらを置くこと、部屋に余分な障害物を置くこと、その他いくつかのアイデアを試しました。あなたは何を思いつきますか？

テトリスはソフトウェアの初期の頃からのもう 1 つのゲームです。本格的なテトリスゲームの設計は、わずかな利益に対して多くの労力を要求するため、このセクションは簡略版に焦点を当てます。野心的なら、テトリスが本当にどう動くかを調べ、本格版を設計してください。

**図81: シンプルなテトリス（Simple Tetris）**
```

[image: pict_109.png]
[image: pict_110.png]
[image: pict_108.png] [image: pict_109.png] [image: pict_110.png]
```

私たちの簡略版では、ゲームは個々のブロックがシーンの上から落ちてくるところから始まります。その 1 つが地面に着地すると静止し、別のブロックがランダムな場所から落ち始めます。プレイヤーは `"left"` と `"right"` の矢印キーで落ちているブロックを制御できます。ブロックがキャンバスの床、またはすでに静止しているブロックの上に着地すると、静止して動かせなくなります。短い時間でブロックは積み上がり、ブロックの山がキャンバスの天井に達するとゲームオーバーです。当然このゲームの目的は、できるだけ多くのブロックを着地させることです。アイデアの図示は図81を参照してください。

この記述が与えられたので、「World プログラムの設計」からの対話的プログラムの設計指針に取りかかれます。それらは定数の性質と可変の性質を分けることを求めます。前者は「物理的」およびグラフィカルな定数として書き下せます。後者は、シンプルなテトリスゲームの可能なすべての状態を構成するデータを示唆します。いくつか例を挙げます。

- ゲームの幅と高さは固定であり、ブロックも同様です。BSL+ では、次のような定義が欲しくなります。

    (define WIDTH 10) ; # of blocks, horizontally
    (define SIZE 10)  ; blocks are squares
    (define SCENE-SIZE (* WIDTH SIZE))
    (define BLOCK
      ; red squares with black rims
      (overlay
        (square (- SIZE 1) "solid" "red")
        (square SIZE "outline" "black")))

読み進める前に、これらの定義を説明してください。

- ブロックの「風景」はゲームごと、またクロックティックごとに異なります。もう少し正確にしましょう。ブロックの見た目は同じままです。位置が異なります。

今や残っている中心問題は、落ちているブロックと地面のブロックの風景のデータ表現を設計することです。落ちているブロックについては、再び 2 つの可能性があります。1 つは「物理的」表現を選ぶこと、もう 1 つは「論理的」表現です。物理的表現はキャンバス上のブロックの実際の物理的位置を追跡します。論理的表現は、ブロックが左端と上端から何ブロック幅離れているかを数えます。静止しているブロックについては、個々のブロックよりもさらに多くの選択肢があります。物理的位置のリスト、論理的位置のリスト、山の高さのリスト、などです。

> **注意:** 関連する設計判断については練習問題215を参照してください。

このセクションでは、データ表現を私たちが選びます。

```racket
(define-struct tetris [block landscape])
(define-struct block [x y])

; A Tetris is a structure:
;   (make-tetris Block Landscape)
; A Landscape is one of:
; – '()
; – (cons Block Landscape)
; A Block is a structure:
;   (make-block N N)

; interpretations
; (make-block x y) depicts a block whose left
; corner is (* x SIZE) pixels from the left and
; (* y SIZE) pixels from the top;
; (make-tetris b0 (list b1 b2...)) means b0 is the
; dropping block, while b1, b2, and... are resting
```

これが私たちが論理的表現と呼んだものです。座標はブロックの物理的位置を反映せず、原点から何ブロックサイズ離れているかだけを反映するからです。私たちの選択は、`x` が常に 0 と `WIDTH` のあいだ（排他的）にあり、`y` が 0 と `HEIGHT` のあいだ（排他的）にあることを含意しますが、この知識は無視します。

**練習問題220.** テトリスゲームの状態のような複雑なデータ定義が提示されたら、さまざまなデータ集まりのインスタンスを作ることから始めます。後で関数例に使える例のための、示唆に富む名前をいくつか挙げます。

```racket
(define landscape0...)
(define block-dropping...)
(define tetris0...)
(define tetris0-drop...)
...
(define block-landed (make-block 0 (- HEIGHT 1)))
...
(define block-on-block (make-block 0 (- HEIGHT 2)))
```

与えられた `Tetris` のインスタンスを `Image` に変えるプログラム `tetris-render` を設計してください。DrRacket の相互作用領域を使って、（極端に）単純なデータ例のいくつかを描画する式を開発してください。それから関数例を単体テストとして定式化し、関数自体も定式化してください。

**練習問題221.** 対話的プログラム `tetris-main` を設計してください。これはキャンバスの上から直線に落ち、床またはすでに静止しているブロックの上に着地するブロックを表示します。`tetris-main` への入力は、クロックがティックする速さを決めるべきです。速さをどう指定するかは `on-tick` のドキュメントを参照してください。

ブロックが着地したかどうかを発見するには、それを落として、床の上にあるか、静止ブロックのリスト上のブロックの 1 つと重なるかを検査することを提案します。ヒント: `member?` プリミティブについて調べてください。

ブロックが着地したら、プログラムは直ちに、現在のものの右隣の列に降りる別のブロックを作るべきです。現在のブロックがすでに右端の列にあるなら、次のブロックは左端の列を使うべきです。あるいは、現在の列とは異なる列をランダムに選ぶ関数 `block-generate` を定義してください。着想は練習問題219を参照してください。

**練習問題222.** 練習問題221のプログラムを変更して、プレイヤーが落ちているブロックの水平移動を制御できるようにしてください。プレイヤーが `"left"` 矢印キーを押すたびに、落ちているブロックは、列 0 にあるか、左にすでに静止ブロックの山がある場合を除き、1 列左へずれるべきです。同様に、プレイヤーが `"right"` を押すたびに、可能なら落ちているブロックは 1 列右へ動くべきです。

**練習問題223.** 練習問題222のプログラムに `stop-when` 句を装備してください。ゲームは、列の 1 つがキャンバスの上端に「触れる」のに十分なブロックを含むときに終了します。

練習問題223を解いたら、必要最小限のテトリスゲームができています。友人に見せる前に少し磨きたくなるかもしれません。たとえば、最終キャンバスにプレイヤーが積み上げられたブロック数を示すテキストを表示できます。あるいはすべてのキャンバスにそのようなテキストを含められます。選択はあなた次第です。

## 12.7 本格的なスペースウォー（Full Space War）

「項目化と構造体」は、動作の少ないスペースインベーダー風ゲームに言及しています。プレイヤーは地上部隊を前後に動かすことしかできません。「リストと World」では、プレイヤーは望むだけ多くのショットを撃てるようになります。このセクションは、このゲームを完成させるのに役立つ練習問題を提示します。

いつものように、UFO が地球に着陸しようとしています。プレイヤーの仕事は UFO の着陸を防ぐことです。そのために、ゲームには任意個数のショットを撃てるタンクが付いています。これらのショットの 1 つが UFO の重心に十分近づくと、ゲームは終了しプレイヤーの勝ちです。UFO が地面に十分近づくと、プレイヤーの負けです。

**練習問題224.** 直前の 2 つのセクションから学んだ教訓を使い、ゲームの拡張をゆっくり設計してください。ゲームの機能を 1 つずつ追加します。常に設計レシピを使い、補助関数の指針に頼ってください。ゲームが気に入ったら、他の機能を追加してください。走るテキストを表示する、タンクを排除できるチャージを UFO に装備する、攻撃する UFO の艦隊全体を作る、そして何より想像力を使ってください。

UFO と互いに撃ち合うタンクが好きでなければ、同じアイデアを使って似た、穏やかなゲームを作ってください。

**練習問題225.** 消火ゲームを設計してください。

ゲームの舞台は、広大な森林を火災が駆け抜ける西部諸州です。空からの消火活動をシミュレートします。具体的には、プレイヤーは地上の火災に水の積荷を落とす飛行機のパイロットとして行動します。プレイヤーは飛行機の水平移動と水の積荷の投下を制御します。

ゲームソフトウェアは地上のランダムな場所に火災を発生させます。火災の数を制限し、現在燃えている火災の数や他の要因の関数にしたくなるかもしれません。ゲームの目的は、限られた時間内にすべての火災を消すことです。ヒント: この章で示したような反復的な設計アプローチを使ってこのゲームを作ってください。

## 12.8 有限状態機械（Finite State Machines）

有限状態機械（FSM）と正規表現は、プログラミングのいたるところにある要素です。「有限状態 World」が説明するように、状態機械は World プログラムについて考える 1 つの方法です。逆に、練習問題109は、FSM を実装し、プレイヤーが特定の一連のキーストロークを押すかどうかを検査する World プログラムの設計方法を示します。

また思い出されるように、有限状態機械は正規表現と等価です。したがって計算機科学者は、FSM がある特定の正規表現に一致するキーストロークを受理すると言う傾向があります。たとえば練習問題109からの次のようなものです。

> `a (b|c)* d`

異なるパターン、たとえば

> `a (b|c)* a`

を認識するプログラムが欲しければ、既存のプログラムを適切に修正するだけでよいでしょう。2 つのプログラムは互いに似ており、いくつかの異なる正規表現についてこの演習を繰り返せば、似た見た目のプログラムがひとまとまりできあがります。

自然な考えは一般解を探すことです。つまり、FSM のデータ表現を受け取り、プレイヤーが一致するキー列を押すかどうかを認識する World プログラムです。このセクションは、まさにそのような World プログラムの設計を提示します。ただし大幅に簡略化されたものです。とくに、FSM には初期状態も最終状態もなく、照合は実際のキーストロークを無視します。代わりに、任意のキーが押されるたびに 1 つの状態から別の状態への遷移が起きます。さらに、状態は色の文字列であることを要求します。そうすれば、FSM を解釈するプログラムは現在状態を色として単に表示できます。

*設計の選択肢に関する注意* 一般化する別の試みは次です。

> **サンプル問題** 特定の `KeyEvent` のリスト上で与えられた FSM を解釈するプログラムを設計せよ。つまり、プログラムは FSM のデータ表現と文字列を受け取る。結果は、文字列が FSM に対応する正規表現に一致すれば `#true`、そうでなければ `#false` である。

しかし結局のところ、このプログラムは第 1 部と第 2 部の原則では設計できません。実際、この問題を解くのは「バックトラックするアルゴリズム」まで待たねばなりません。練習問題476を参照してください。

**図82: 有限状態機械の一般的な表現と解釈（Representing and interpreting finite state machines in general）**
```
; An FSM is one of:
; – '()
; – (cons Transition FSM)

(define-struct transition [current next])
; A Transition is a structure:
; (make-transition FSM-State FSM-State)

; FSM-State is a Color.

; interpretation An FSM represents the transitions that a
; finite state machine can take from one state to another
; in reaction to keystrokes
```

簡略化された問題文は、FSM の表現のためのデータ定義の必要性、その状態の性質、画像としての見た目を含むいくつかの点を命じます。図82はこの情報をまとめています。それは FSM のデータ定義から始まります。ご覧のとおり、FSM は単なる `Transition` のリストです。リストを使わなければならないのは、World プログラムが任意の FSM で動くようにしたいからであり、それは有限だが任意に大きい数の状態を意味します。各 `Transition` は 2 つの状態を構造体に組み合わせます。現在状態と次の状態、つまりプレイヤーがキーを押したときに機械が遷移する先です。データ定義の最後の部分は、状態はただ色の名前であると言います。

**練習問題226.** 状態のための等価述語 `state=?` を設計してください。

この定義は複雑なので、設計レシピに従い例を作ります。

```racket
(define fsm-traffic
  (list (make-transition "red" "green")
        (make-transition "green" "yellow")
        (make-transition "yellow" "red")))
```

おそらく推測したでしょうが、この遷移表は信号機を記述しています。最初の遷移は信号機が `"red"` から `"green"` へ飛ぶことを教え、2 番目は `"green"` から `"yellow"` への遷移を表し、最後は `"yellow"` から `"red"` のためのものです。

**練習問題227.** BW Machine は、キーイベントごとに黒から白へ、そしてまた黒へ反転する FSM です。BW Machine のデータ表現を定式化してください。

明らかに、私たちの問題の解は World プログラムです。

```racket
; FSM ->???
; match the keys pressed with the given FSM
(define (simulate an-fsm)
  (big-bang...
    [to-draw...]
    [on-key...]))
```

FSM を受け取ることになっていますが、プログラムが何を生成するかは手がかりがありません。プログラムを `simulate` と呼ぶのは、プレイヤーのキーストロークに応答して与えられた FSM のように振る舞うからです。

それでも World プログラムの設計レシピに従い、どこまで連れて行ってくれるかを見ましょう。それは「現実世界」で変わるものと変わらないものを区別するよう教えてくれます。`simulate` 関数は `FSM` のインスタンスを受け取りますが、この FSM は変わらないことも分かっています。変わるのは機械の現在状態です。

この分析は次のデータ定義を示唆します。

```racket
; A SimulationState.v1 is an FSM-State.
```

World プログラムの設計レシピによれば、このデータ定義がメイン関数を完成させます。

> **注意:** 定数 `empty-image` は「見えない」画像を表します。描画関数のヘッダを書き下すときの良いデフォルト値です。

```racket
(define (simulate.v1 fsm0)
  (big-bang initial-state
    [to-draw render-state.v1]
    [on-key find-next-state.v1]))
```

そして 2 項目の願望リストを含意します。

```racket
; SimulationState.v1 -> Image
; renders a world state as an image
(define (render-state.v1 s)
  empty-image)

; SimulationState.v1 KeyEvent -> SimulationState.v1
; finds the next state from ke and cs
(define (find-next-state.v1 cs ke)
   cs)
```

このスケッチは 2 つの疑問を提起します。第一に、ごく最初の `SimulationState.v1` がどう決まるかという問題があります。現在、選ばれた状態 `initial-state` は、問題を警告するために灰色で印が付けられています。第二に、願望リストの 2 番目の項目は当惑を引き起こすに違いありません。

> 現在状態とキーストロークだけが与えられているとき、`find-next-state` はどうやって次の状態を見つけられるのか？

この疑問はとくに尤もです。簡略化された問題文によれば、キーストロークの正確な性質は無関係だからです。どのキーが押されても FSM は次の状態へ遷移します。

この 2 番目の問題が露呈するのは、BSL+ の根本的な制限です。この制限を正しく理解するために、回避策から始めます。基本的に、分析は `find-next-state` 関数が現在状態だけでなく FSM も受け取り、遷移のリストを検索して次の状態を選べるようにすることを要求します。言い換えると、世界の状態は FSM の現在状態と FSM 自体の両方を含まなければなりません。

> **注意:** 最初の 2 人の計算機科学者であるアロンゾ・チャーチとアラン・チューリングは、1930 年代に、すべてのプログラミング言語がある種の数に対する関数を計算できることを証明しました。したがって、すべてのプログラミング言語は等しいと彼らは論じました。本書の第 1 著者は同意しません。彼は、プログラマーが解をどう表現できるかに応じて言語を区別します。

```racket
(define-struct fs [fsm current])
; A SimulationState.v2 is a structure:
;   (make-fs FSM FSM-State)
```

World 設計レシピによれば、この変更はキーイベントハンドラもこの組み合わせを返さなければならないことも意味します。

```racket
; SimulationState.v2 -> Image
; renders a world state as an image
(define (render-state.v2 s)
  empty-image)

; SimulationState.v2 KeyEvent -> SimulationState.v2
; finds the next state from ke and cs
(define (find-next-state.v2 cs ke)
   cs)
```

最後に、メイン関数は今や 2 つの引数を受け取らなければなりません。FSM とその最初の状態です。結局のところ、`simulate` が受け取るさまざまな FSM はあらゆる種類の状態を伴います。すべてが同じ初期状態を持つと仮定することはできません。改訂された関数ヘッダは次です。

```racket
; FSM FSM-State -> SimulationState.v2
; match the keys pressed with the given FSM
(define (simulate.v2 an-fsm s0)
  (big-bang (make-fs an-fsm s0)
    [to-draw state-as-colored-square]
    [on-key find-next-state]))
```

信号機 FSM の例に戻りましょう。この機械では、`simulate` を機械と `"red"` に適用するのが最良でしょう。

```racket
(simulate.v2 fsm-traffic "red")
```

> **注意:** 技術者は `"red"` を安全状態と呼びます。

*表現力に関する注意* 回避策が与えられたので、BSL の制限を今や説明できます。与えられた FSM はシミュレーションの過程で変わらないにもかかわらず、その記述は世界の状態の一部にならなければなりません。理想的には、プログラムは FSM の記述が定数のままであることを表現すべきですが、代わりにプログラムは FSM を常に変わる状態の一部として扱わなければなりません。プログラムの読者は、`big-bang` の最初の部分だけからはこの事実を演繹できません。

この時点で、願望リストに取りかかり、項目を 1 つずつ片付けられます。最初の項目、`state-as-colored-square` の設計は非常にまっすぐなので、完全な定義を単に提供します。

```racket
; SimulationState.v2 -> Image
; renders current world state as a colored square

(check-expect (state-as-colored-square
                (make-fs fsm-traffic "red"))
              (square 100 "solid" "red"))

(define (state-as-colored-square an-fsm)
  (square 100 "solid" (fs-current an-fsm)))
```

対照的に、キーイベントハンドラの設計はいくらか議論に値します。ヘッダ材料を思い出してください。

```racket
; SimulationState.v2 KeyEvent -> SimulationState.v2
; finds the next state from an-fsm and ke
(define (find-next-state an-fsm ke)
  an-fsm)
```

設計レシピによれば、ハンドラは世界の状態と `KeyEvent` を受け取り、世界の次の状態を生成しなければなりません。平易な言葉でのこのシグネチャの言い換えは、例の設計も導きます。最初の 2 つは次です。

```racket
(check-expect
  (find-next-state (make-fs fsm-traffic "red") "n")
  (make-fs fsm-traffic "green"))
(check-expect
  (find-next-state (make-fs fsm-traffic "red") "a")
  (make-fs fsm-traffic "green"))
```

例は、現在状態が `fsm-traffic` 機械とその `"red"` 状態を組み合わせるとき、結果は同じ FSM と `"green"` を組み合わせると言っています。プレイヤーがキーボードで `"n"` を打ったか `"a"` を打ったかにかかわらずです。もう 1 つ例を挙げます。

```racket
(check-expect
  (find-next-state (make-fs fsm-traffic "green") "q")
  (make-fs fsm-traffic "yellow"))
```

読み進める前にこの例を解釈してください。もう 1 つ思いつきますか？

関数は構造体を受け取るので、構造体処理のテンプレートを書き下します。

```racket
(define (find-next-state an-fsm ke)
  (... (fs-fsm an-fsm).. (fs-current an-fsm)...))
```

さらに、望ましい結果は `SimulationState.v2` なので、適切なコンストラクタの追加でテンプレートを洗練できます。

```racket
(define (find-next-state an-fsm ke)
  (make-fs
... (fs-fsm an-fsm)... (fs-current an-fsm)...))
```

例は、取り出された FSM が新しい `SimulationState.v2` の第 1 成分になり、関数が本当に計算する必要があるのは現在状態と、与えられた FSM を構成する `Transition` のリストからの次の状態だけであることを示唆します。後者は任意に長いので、願いを立てます—`(fs-current an-fsm)` が現在状態である `Transition` を探すためにリストを走査する `find` 関数—そして定義を完成させます。

```racket
(define (find-next-state an-fsm ke)
  (make-fs
    (fs-fsm an-fsm)
    (find (fs-fsm an-fsm) (fs-current an-fsm))))
```

新しい願いの定式化は次です。

```racket
; FSM FSM-State -> FSM-State
; finds the state representing current in transitions
; and retrieves the next field
(check-expect (find fsm-traffic "red") "green")
(check-expect (find fsm-traffic "green") "yellow")
(check-error (find fsm-traffic "black")
             "not found: black")
(define (find transitions current)
  current)
```

例は `find-next-state` の例から導かれています。

**練習問題230.** FSM についての次のデータ表現を考えてください。

```racket

(define-struct transition [current key next])
; An FSM.v2 is a structure:
;   (make-fsm FSM-State LOT FSM-State)
; A LOT is one of:
; – '()
; – (cons Transition.v3 LOT)
; A Transition.v3 is a structure:
;   (make-transition FSM-State KeyEvent FSM-State)
```

この文脈で練習問題109の FSM を表してください。

`FSM.v2` を受け取り、プレイヤーのキーストローク上でそれを動かす関数 `fsm-simulate` を設計してください。キーストローク列が `FSM.v2` を最終状態に達するよう強制したら、`fsm-simulate` は停止します。ヒント: この関数は、与えられた `fsm` 構造体の `initial` フィールドを使って現在状態を追跡します。

*反復的洗練に関する注意* これら最後の 2 つのプロジェクトは、「反復的洗練による設計」という概念を導入します。基本的な考えは、最初のプログラムは望ましい振る舞いの一部だけを実装し、次のものはもう少し、といった具合です。最終的に、望ましい振る舞いのすべて、あるいは少なくとも顧客を満足させるのに十分な振る舞いを示すプログラムにたどり着きます。詳細は「反復的洗練」を参照してください。
