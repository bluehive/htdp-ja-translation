# 9 自己参照データ定義を用いた設計（Designing with Self-Referential Data Definitions）

**原本:** `extracted/original_markdown_05_part_two.md`（第9章）

一見すると、自己参照データ定義は混合データ向けのものよりはるかに複雑に見える。しかし、`contains-flatt?` の例が示すように、設計レシピの6つのステップは引き続き機能する。それでも本節では、自己参照データ定義に対してよりよく働くよう、設計レシピを一般化する。新しい部分は、自己参照データ定義がいつ必要かを見極める過程、テンプレートの導出、そして関数本体の定義に関わる。

1. 問題文が任意の大きさの情報についてのものなら、それを表現するために自己参照データ定義が必要である。現時点で見たそのようなクラスは `List-of-names` だけである。図51の左側は、同じ仕方で `List-of-strings` を定義する方法を示す。他のアトミックデータのリストも同じように動く。

   自己参照データ定義が妥当であるためには、2つの条件を満たさなければならない。第一に、少なくとも2つの節を含むこと。第二に、少なくとも1つの節は、定義しているデータのクラスへ戻って参照してはならないことである。データ定義中の参照から定義されている語へ戻る矢印で、自己参照を明示的に識別するのがよい習慣である。そのような注記の例は図51を見よ。

   自己参照データ定義の妥当性は、データ例の作成で確認しなければならない。データ定義を参照しない節から始め、定義自体を参照する節では直前の例を使って続ける。図51のデータ定義からは、次のような3つのリストが得られる。

   - `'()` — 第1節による
   - `(cons "a" '())` — 第2節と直前の例による
   - `(cons "b" (cons "a" '()))` — 第2節と直前の例による

   データ定義から例を生成できないなら、その定義は無効である。例は生成できても、ますます大きな例をどう作るかが見えないなら、その定義は意図した解釈に達していないかもしれない。

   > **注:** 数も任意に大きく見える。非正確な数についてはこれは錯覚である。正確な整数については確かにそうである。したがって整数の扱いは本章の一部である。

2. ヘッダ材料——シグネチャ、目的文、ダミー定義——については何も変わらない。目的文を定式化するときは、関数が**何を**計算するかに焦点を当て、**どのように**それを行うか、とくに与えられたデータのインスタンスをどう辿るかには焦点を当てない。

   この設計レシピを具体化する例を示す。

    ; List-of-strings -> Number
    ; counts how many strings alos contains
    (define (how-many alos)
      0)

   この目的文は、関数が与えられた入力上の文字列を単に数えるだけだと明確に述べている。この考えを BSL 関数としてどう定式化するかを先回りして考える必要はない。

3. 機能例については、データ定義の自己参照節を何度も使った入力を必ず通して考えること。後で関数定義全体を覆うテストを定式化する最良の方法である。

   進行中の例では、目的文がデータ例から機能例をほとんど自分で生成する。

   | given | wanted |
   | :--- | :--- |
   | `'()` | `0` |
   | `(cons "a" '())` | `1` |
   | `(cons "b" (cons "a" '()))` | `2` |

   第1行は空リストについてであり、空リストには何も含まれない。第2行は1つの文字列のリストなので、望ましい答えは `1` である。最後の行は2つの文字列のリストについてである。

4. 核心では、自己参照データ定義は混合データのデータ定義のように見える。したがってテンプレートの開発は、「項目化と構造体」のレシピに従って進められる。具体的には、データ定義の節と同じ数の `cond` 節を持つ `cond` 式を立て、各認識条件をデータ定義の対応する節に合わせ、複合値を処理するすべての `cond` 行に適切なセレクタ式を書き下ろす。

   | 質問 | 回答 |
   | :--- | :--- |
   | データ定義はデータの異なる部分クラスを区別するか？ | テンプレートには、データ定義が区別する部分クラスと同じ数の `cond` 節が要る。 |
   | 部分クラスは互いにどう異なるか？ | その違いを使って、節ごとの条件を定式化する。 |
   | いずれかの節は構造化された値を扱うか？ | そうなら、その節に適切なセレクタ式を加える。 |
   | データ定義は自己参照を使うか？ | データ定義の自己参照を表す「自然再帰」をテンプレートに定式化する。 |
   | データ定義が他のデータ定義を参照するなら、その相互参照はどこか？ | 他のデータ定義用のテンプレートを特殊化し、それを参照する。「項目化を用いた設計、再び」の設計レシピのステップ4と5を見よ。 |

   **図52: データ定義をテンプレートへ翻訳する方法（How to translate a data definition into a template）**

   図52はこの考えを問答ゲームとして表す。左列は引数のデータ定義についての質問を述べ、右列はその答えがテンプレートの構築に何を意味するかを説明する。

   最後の行を無視し、最初の3つの質問を `List-of-strings` を消費する任意の関数に適用すると、次の形に至る。

    (define (fun-for-los alos)
      (cond
        [(empty? alos) ...]
        [else (... (first alos) ... (rest alos) ...)]))

   しかし思い出してほしい。テンプレートの目的は、データ定義を関数の配置として表現することである。つまりテンプレートは、入力のデータ定義が英語と BSL の混在として表現している内容を、コードとして表現する。したがってデータ定義の重要な断片はすべてテンプレートに対応物を見つけなければならず、この指針はデータ定義が自己参照的なとき——定義の内側から定義されている語への矢印を含むとき——にも成り立つ。とくに、データ定義が第 *i* 節で、そこに出てくる構造体の第 *k* フィールドにおいて自己参照するなら、テンプレートは第 *i* の `cond` 節で、第 *k* フィールドのセレクタ式において自己参照すべきである。そのようなセレクタ式ごとに、関数パラメータへ戻る矢印を加える。最終的に、テンプレートのデータ定義と同じ数の矢印を持たなければならない。

   図51は、この考えを `List-of-strings` を消費する関数のテンプレートとデータ定義を並べて示す。どちらも第2節——それぞれ `rest` フィールドとセレクタ——に起源を持ち、それぞれの定義の先頭を指す1本の矢印を含む。

   BSL とほとんどのプログラミング言語はテキスト指向なので、矢印の代わりに、適切なセレクタ式への関数の自己適用を使わなければならない。

    (define (fun-for-los alos)
      (cond
        [(empty? alos) ...]
        [else (... (first alos) ...
               ... (fun-for-los (rest alos)) ...)]))

   関数の自己使用を再帰と呼び、本書の最初の4部では自然再帰と呼ぶ。

5. 関数本体については、再帰的関数呼び出しのない `cond` 行——基本ケース——から始める。対応する答えは通常、定式化が容易か、すでに例として与えられている。

   次に自己参照のケースを扱う。テンプレート行の各式が何を計算するかを自分に思い起こすことから始める。自然再帰については、関数が目的文で指定された通りにすでに動くと仮定する。この最後のステップは信頼の飛躍だが、これから見るように、常にうまくいく。

   残りは、さまざまな値を組み合わせることである。

   | 質問 | 回答 |
   | :--- | :--- |
   | 非再帰的な `cond` 節の答えは何か？ | 例がここで必要な値を教えるはずである。そうでなければ適切な例とテストを定式化する。 |
   | 再帰節のセレクタ式は何を計算するか？ | データ定義がこれらの式がどんな種類のデータを取り出すかを教え、データ定義の解釈がこのデータが何を表すかを教える。 |
   | 自然再帰は何を計算するか？ | 関数の目的文を使い、再帰の値が何を**意味する**かを決める。どう計算するかではない。目的文が答えを教えてくれないなら、目的文を改善する。 |
   | 関数はこれらの値をどう組み合わせて望ましい答えを得るか？ | 値を組み合わせる BSL の関数を見つける。それがうまくいかなければ、補助関数を願望する。多くの関数ではこの最後のステップは素直である。目的、例、テンプレートが一緒になって、利用可能な値を適切な結果へ組み合わせる関数や式を教える。この関数や式を、既存用語を少し濫用して結合子と呼ぶ。 |

   **図53: テンプレートを関数定義へ変える方法（How to turn a template into a function definition）**

   | 質問 | 回答 |
   | :--- | :--- |
   | ここで行き詰まったら…… | 第3ステップの例を表に並べる。第1列に与えられた入力、最後の列に望ましい出力を置く。中間列にセレクタ式と自然再帰の値を入れる。結合子を示唆するパターンが見えるまで例を追加する。 |
   | テンプレートが他のテンプレートを参照するなら、補助関数は何を計算するか？ | その関数の目的文と例を参照して何を計算するかを決め、このヘルパーの設計が終わっていなくても結果を使ってよいと仮定する。 |

   **図54: テンプレートを関数へ変える表法（Turning a template into a function, the table method）**

   図53はこのステップの最初の4つの問答を定式化する。このゲームを使って `how-many` の定義を完成させよう。`fun-for-los` テンプレートを `how-many` に改名すると、ここまでになる。

    ; List-of-strings -> Number
    ; determines how many strings are on alos
    (define (how-many alos)
      (cond
        [(empty? alos) ...]
        [else (... (first alos) ...
               ... (how-many (rest alos)) ...)]))

   機能例がすでに示唆するように、基本ケースの答えは `0` である。第2節の2つの式は、先頭項目と `(rest alos)` 内の文字列の個数を計算する。`alos` 全体に何個の文字列があるかを計算するには、後者の式の値に `1` を足すだけでよい。

    (define (how-many alos)
      (cond
        [(empty? alos) 0]
        [else (+ (how-many (rest alos)) 1)]))

   値を望ましい答えへ組み合わせる正しい仕方を見つけるのは、常にこれほど簡単ではない。初心者はしばしばこのステップで行き詰まる。図54が示唆するように、機能例を、テンプレート内の式の値も書き出した表に並べるのがよい。図55は、`how-many` の例でこの表がどうなるかを示す。左端の列はサンプル入力を並べ、右端の列はそれらの入力に対する望ましい答えを含む。間の3列はテンプレート式の値を示す。`(first alos)`、`(rest alos)`、および自然再帰 `(how-many (rest alos))` である。この表を十分長く見つめると、結果列が常に自然再帰列の値より1大きいことに気づく。そこで次のように推測してよい。

    (+ (how-many (rest alos)) 1)

   これが望ましい結果を計算する式である。DrRacket はこの種の推測を速く検査できるので、組み込んで *RUN* をクリックする。例から作ったテストが通れば、その式がすべてのリストで動くと自分を納得させるまで考える。そうでなければ、別のアイデアが得られるまで表に例の行を追加する。

   表はまた、テンプレートの一部のセレクタ式が実際の定義には無関係かもしれないことも示す。ここでの `(first alos)` は最終答えの計算に要らない——これはテンプレートの両方の式を使う `contains-flatt?` と対照的である。

   本書の残りを進むあいだ、多くの場合、組み合わせのステップは BSL のプリミティブ、たとえば `+`、`and`、`cons` で表現できることを覚えておくこと。ただし場合によっては願望、すなわち補助関数の設計が要る。さらに入れ子の条件が要る場合もある。

   > **注:** 好奇心旺盛な読者のために記すと、任意に大きなデータ向けの設計レシピは、数学におけるいわゆる「帰納法による証明」に対応し、「信頼の飛躍」は、そのような証明の帰納ステップでの帰納仮説の使用を表す。論理学は帰納法の定理で、この証明技法の妥当性を示す。

   > **注:** Felix Klock が、結合子を推測するこの表に基づく方法を提案した。

**練習問題 139.** 次に、このデータ定義を見よ。

```racket

; – '()
```

このデータクラスの要素の一部は練習問題138の `sum` への適切な入力であり、一部はそうではない。

### 9.2 空でないリスト（Non-empty Lists）

これで `cons` を使い、リストのためのデータ定義を作る十分な知識が得られた。前節末の練習問題の（一部を）解いたなら、さまざまな種類の数のリスト、ブール値のリスト、画像のリストなどを扱える。本節では、リストとは何か、どう処理するかをさらに探る。

温度のリストの平均を計算するという、一見簡単な問題から始めよう。簡単のため、データ定義を与える。

```racket
; A List-of-temperatures is one of:
; – '()
; – (cons CTemperature List-of-temperatures)

(define ABSOLUTE0 -272)
; A CTemperature is a Number greater than ABSOLUTE0.
```

我々の意図では、温度を普通の数と考えてよいが、第2のデータ定義は、現実にはすべての数が温度ではないこと、そしてそれを念頭に置くべきことを思い起こさせる。

ヘッダ材料は素直である。

```racket
; List-of-temperatures -> Number
; computes the average temperature
(define (average alot) 0)
```

この問題の例作りも容易なので、テストを1つだけ定式化する。

```racket
(check-expect
  (average (cons 1 (cons 2 (cons 3 '())))) 2)
```

期待される結果はもちろん、温度の和を温度の個数で割ったものである。

少し考えると、`average` のテンプレートはこれまで見てきたものに似るはずだと分かる。

```racket
(define (average alot)
  (cond
    [(empty? alot)...]
    [(cons? alot)
     (... (first alot)...
... (average (rest alot))...)]))
```

2つの `cond` 節はデータ定義の2つの節を映し、質問は空リストと空でないリストを区別し、自然再帰はデータ定義の自己参照のために要る。

しかし、このテンプレートを関数定義へ変えるのはあまりにも難しい。第1の `cond` 節には、空の温度の集まりの平均を表す数が要るが、そのような数はない。同様に第2節は、1つの温度と残りの温度の平均を新しい平均へ組み合わせる関数を要求する。不可能ではないが、この仕方で平均を計算するのはきわめて不自然である。

温度の平均を計算するとき、それらの和を個数で割る。些細な例を定式化したときにもそう言った。この文は、`average` が3つの仕事——合計、計数、除算——の関数だと示唆する。「固定サイズのデータ」の指針は仕事ごとに1関数を書けと教え、そうすれば `average` の設計は明白である。

```racket
; List-of-temperatures -> Number
; computes the average temperature
(define (average alot)
  (/ (sum alot) (how-many alot)))

; List-of-temperatures -> Number
; adds up the temperatures on the given list
(define (sum alot) 0)

; List-of-temperatures -> Number
; counts the temperatures on the given list
(define (how-many alot) 0)
```

最後の2つの関数定義はもちろん願望であり、完全な定義を設計する必要がある。それは容易である。なぜなら、上の `how-many` は `List-of-strings` にも `List-of-temperatures` にも働き（なぜか？）、`sum` の設計も同じお決まりの手順に従うからである。

```racket
; List-of-temperatures -> Number
; adds up the temperatures on the given list
(define (sum alot)
  (cond
    [(empty? alot) 0]
    [else (+ (first alot) (sum (rest alot)))]))
```

ストップ！ `average` の例を使って `sum` の例を作り、テストが正しく走ることを確かめよ。その後 `average` のテストを実行せよ。

今この `average` の定義を読めば、学校で平均について学ぶことに直接対応するので明らかに正しい。それでもプログラムは我々のためだけでなく他者のためにも走る。とくに他者はシグネチャを読んで関数を使い、有益な答えを期待できるべきである。しかし我々の `average` の定義は空の温度リストでは動かない。

**練習問題 143.** `average` を空リストに適用したとき DrRacket でどう振る舞うかを確かめよ。次に、`'()` に適用されたとき有益なエラーメッセージを生成する関数 `checked-average` を設計せよ。

> **注:** 数学では、練習問題143は `average` が部分関数であることを示すと言う。`'()` に対してエラーを起こすからである。

代替の解決は、シグネチャを通じて将来の読者に、`average` が空リストでは動かないと知らせることである。そのためには `'()` を除外するリストのデータ表現が要る。たとえば次のようなものである。

```racket
; An NEList-of-temperatures is one of:
; –???
; – (cons CTemperature NEList-of-temperatures)
```

問題は、`'()` リストを除外しつつ他のすべての温度リストをなお構築できるよう、「???」を何で置き換えるかである。1つのヒントは、空リストが最短のリストである一方、1つの温度の任意のリストが次に短いリストだということである。これは、第1節が1つの温度のすべての可能なリストを記述すべきだと示唆する。

```racket
; An NEList-of-temperatures is one of:
; – (cons CTemperature '())
; – (cons CTemperature NEList-of-temperatures)
; interpretation non-empty lists of Celsius temperatures
```

この定義はこれまでのリスト定義とは異なるが、決定的な要素——自己参照と、自己参照を使わない節——を共有する。設計レシピへの厳密な遵守は、定義が意味をなすことを確かめるために `NEList-of-temperatures` の例をいくつか作るよう求める。いつものように基本節から始めるので、例は次のように見えるはずである。

```racket
(cons c '())
```

ここで `c` は `CTemperature` を表し、たとえば次のようになる。`(cons ABSOLUTE0 '())`。また、`List-of-temperatures` の空でない要素はすべて新しいデータクラスの要素でもあることは明らかである。`(cons 2 (cons 3 '()))` がそうなら `(cons 1 (cons 2 (cons 3 '())))` も適合し、`(cons 2 (cons 3 '()))` が `NEList-of-temperatures` に属するのは、すでに確認したように `(cons 3 '())` が `NEList-of-temperatures` の要素だからである。`NEList-of-temperatures` の大きさに限界がないことを自分で確かめよ。

では、誰もが空でないリスト専用だと分かるように `average` を設計する問題に戻ろう。`NEList-of-temperatures` の定義があれば、シグネチャで言いたいことを述べる手段がある。

> **注:** この代替の展開は、この場合 `average` の定義域を狭めて全域関数を作れることを説明する。

```racket
; NEList-of-temperatures -> Number
; computes the average temperature

(check-expect (average (cons 1 (cons 2 (cons 3 '()))))
              2)

(define (average ne-l)
  (/ (sum ne-l)
     (how-many ne-l)))
```

もちろん残り——目的文、例テスト、関数定義——は同じままである。結局、平均の計算という考え自体が空でない数の集まりを前提としており、それが議論全体の要点だった。

**練習問題 144.** `sum` と `how-many` は `List-of-temperatures` からの入力向けに設計されているのに、`NEList-of-temperatures` でも動くか？ 動かないと思うなら反例を示せ。動くと思うなら理由を説明せよ。

それでもこの定義は、今や `NEList-of-temperatures` のインスタンスを消費するので、`sum` と `how-many` をどう設計するかという問いも立てる。設計レシピの最初の3ステップの明白な結果は次の通りである。

```racket
; NEList-of-temperatures -> Number
; computes the sum of the given temperatures
(check-expect
  (sum (cons 1 (cons 2 (cons 3 '())))) 6)
(define (sum ne-l) 0)
```

例は `average` の例から適応した。ダミー定義は数を生成するが、与えられたテストに対しては誤った数である。

第4ステップが、`NEList-of-temperatures` 向け `sum` の設計で最も興味深い部分である。これまでの設計例はすべて、データ定義が適切な形を持つため、空リストと空でない、すなわち `cons` されたリストを区別するテンプレートを要求する。`NEList-of-temperatures` ではそうではない。ここでは両方の節が `cons` されたリストを加える。しかし2つの節はこれらのリストの `rest` フィールドで異なる。とくに第1節は常に `rest` フィールドに `'()` を使い、第2節は代わりに `cons` を使う。したがって第1種のデータと第2種を区別する適切な条件は、`rest` フィールドを取り出し `empty?` を使うことである。

```racket
; NEList-of-temperatures -> Number
(define (sum ne-l)
  (cond
    [(empty? (rest ne-l))...]
    [else...]))
```

ここで `else` が `(cons? (rest ne-l))` を置き換える。

次に両方の節を調べ、一方または両方で `ne-l` を構造体であるかのように扱うかを決める。もちろんそうであり、`ne-l` への無条件の `rest` の使用がそれを示す。言い換えれば、2つの節に適切なセレクタ式を加える。

```racket
(define (sum ne-l)
  (cond
    [(empty? (rest ne-l)) (... (first ne-l)...)]
    [else (... (first ne-l)... (rest ne-l)...)]))
```

読み進める前に、なぜ第1節にセレクタ式 `(rest ne-l)` が含まれないかを説明せよ。

テンプレート設計の最後の質問は、データ定義の自己参照に関する。知っている通り、`NEList-of-temperatures` には1つあり、したがって `sum` のテンプレートは1つの再帰的使用を要求する。

```racket
(define (sum ne-l)
  (cond
    [(empty? (rest ne-l)) (... (first ne-l)...)]
    [else
     (... (first ne-l)... (sum (rest ne-l))...)]))
```

具体的には、データ定義が対応する点で自己参照的なので、第2節で `(rest ne-l)` に対して `sum` が呼ばれる。

第5の設計ステップでは、すでにどれだけ持っているかを理解しよう。第1の `cond` 節は再帰呼び出しのある第2節よりかなり単純に見えるので、そちらから始めるべきである。この場合、条件は `sum` がちょうど1つの温度、`(first ne-l)` を持つリストに適用されると言う。明らかに、この1つの温度が与えられたリスト上のすべての温度の和である。

```racket
(define (sum ne-l)
  (cond
    [(empty? (rest ne-l)) (first ne-l)]
    [else
     (... (first ne-l)... (sum (rest ne-l))...)]))
```

第2節は、リストが1つの温度と少なくとももう1つからなると言う。`(first ne-l)` は先頭位置を取り出し、`(rest ne-l)` は残りを取り出す。さらにテンプレートは `(sum (rest ne-l))` の結果を使うことを示唆する。しかし `sum` はまさに定義中の関数であり、それが `(rest ne-l)` を**どう**使うかを知る由もない。知っているのは目的文の内容だけ、すなわち `sum` が与えられたリスト——これは `(rest ne-l)`——上のすべての温度を足すということである。この文が真なら、`(sum (rest ne-l))` は `ne-l` の数の1つを除くすべてを足す。合計を得るには、関数は最初の温度を足すだけでよい。

```racket
(define (sum ne-l)
  (cond
    [(empty? (rest ne-l)) (first ne-l)]
    [else (+ (first ne-l) (sum (rest ne-l)))]))
```

今この関数のテストを実行すれば、信頼の飛躍が正当化されることが分かる。実際、本書の範囲を超える理由により、この飛躍は常に正当化され、それゆえ設計レシピの固有の一部である。

### 9.3 自然数（Natural Numbers）

BSL プログラミング言語はリストを消費する多くの関数と、リストを生成する関数もいくつか供給する。その中に `make-list` がある。これは数 `n` と他の値 `v` を消費し、`v` を `n` 回含むリストを生成する。いくつかの例を示す。

```racket
> (make-list 2 "hello")
(cons "hello" (cons "hello" '()))
> (make-list 3 #true)
(cons #true (cons #true (cons #true '())))
> (make-list 0 17)
'()
```

要するに、この関数はアトミックデータを消費するにもかかわらず、任意に大きなデータを生成する。どうしてそれが可能なのか、が問いであるべきである。

答えは、`make-list` の入力が単なる数ではなく、特別な種類の数だということである。幼稚園ではこれらの数を「数えの数」と呼んだ。すなわち物を数えるのに使う数である。コンピュータサイエンスでは、これらの数は自然数と呼ばれる。通常の数と違い、自然数にはデータ定義が付く。

```racket
; An N is one of:
; – 0
; – (add1 N)
; interpretation represents the counting numbers
```

第1節は `0` が自然数だと言う。数える物がないことを言うのに使う。第2節は、`n` が自然数なら `n+1` もそうだと教える。`add1` は与えられた任意の数に `1` を足す関数だからである。この第2節を `(+ n 1)` と書くこともできるが、`add1` の使用はこの加算が特別だと合図するためのものである。

この `add1` の使用が特別なのは、通常の関数というより、何らかの構造体型定義のコンストラクタのように働くことである。そのため BSL には `add1` に対応する「セレクタ」である関数 `sub1` もある。`0` と等しくない任意の自然数 `m` が与えられたとき、`sub1` を使って `m` の構築に入った数を見つけられる。言い換えれば、`add1` は `cons` のようであり、`sub1` は `first` と `rest` のようである。

この時点で、`0` と `0` でない自然数を区別する述語は何かと思うかもしれない。リストと同様に2つある。ある与えられた数が `0` かどうかを判定する `zero?` と、ある数が `0` より大きいかを判定する `positive?` である。

これで自然数上の関数、たとえば `make-list` を自分で設計する位置にいる。データ定義はすでに利用可能なので、ヘッダ材料を加えよう。

```racket
; N String -> List-of-strings
; creates a list of n copies of s

(check-expect (copier 0 "hello") '())
(check-expect (copier 2 "hello")
              (cons "hello" (cons "hello" '())))

(define (copier n s)
  '())
```

テンプレートの開発が次のステップである。テンプレートの質問は、`copier` の本体が2節の `cond` 式——`0` 用と正の数用——だと示唆する。さらに `0` はアトミックとみなされ、正の数は構造化された値とみなされるので、テンプレートは第2節にセレクタ式を要す。最後に、`N` のデータ定義は第2節で自己参照的である。したがってテンプレートは第2節の対応するセレクタ式への再帰適用を要す。

```racket
(define (copier n s)
  (cond
    [(zero? n)...]
    [(positive? n) (... (copier (sub1 n) s)...)]))
```

```
+----------------------------------------------------+
| Figure 59: Creating a list of copies               |
|                                                    |
|; N String -> List-of-strings                      |
|; creates a list of n copies of s                  |
|                                                    |
| (check-expect (copier 0 "hello") '())              |
| (check-expect (copier 2 "hello")                   |
|               (cons "hello" (cons "hello" '())))   |
|                                                    |
| (define (copier n s)                               |
|   (cond                                            |
|     [(zero? n) '()]                                |
|     [(positive? n) (cons s (copier (sub1 n) s))])) |
+----------------------------------------------------+
```

図59は、テンプレートから得られた `copier` 関数の完全な定義を含む。このステップを注意深く再構成しよう。いつものように、再帰呼び出しのない `cond` 節から始める。ここでの条件は（重要な）入力が `0` だと教え、それは関数が `0` 項目、すなわち何もないリストを生成しなければならないことを意味する。もちろん第2の例を通して考えることでこのケースはすでに明らかである。次に他方の `cond` 節に向かい、その式が何を計算するかを思い起こす。

1. `(sub1 n)` は、`n`——`0` より大きいと分かっている——の構築に入った自然数を取り出す。
2. `(copier (sub1 n) s)` は、`copier` の目的文に従い、`(sub1 n)` 個の文字列 `s` のリストを生成する。

しかし関数には `n` が与えられているので、`n` 個の文字列 `s` のリストを生成しなければならない。文字列が1つ足りないリストが与えられたなら、関数は単に `(copier (sub1 n) s)` の結果へ `s` を1つ `cons` すればよい。そしてそれこそが第2節が指定することである。

この時点でテストを実行し、この関数が少なくとも2つの作成した例で動くことを確かめよ。加えて、いくつかの追加入力で関数を使ってみてもよい。

**練習問題 149.** `copier` を自然数とブール値や画像に適用しても適切に動くか？ それとも別の関数を設計しなければならないか？ 答えは「抽象化」を読め。

`copier` の代替定義は `else` を使うかもしれない。

```racket
(define (copier.v2 n s)
  (cond
    [(zero? n) '()]
    [else (cons s (copier.v2 (sub1 n) s))]))
```

### 9.4 ロシア人形（Russian Dolls）

Wikipedia はロシア人形を「大きさが小さくなっていく人形の一組で、一方が他方の中に置かれたもの」と定義し、次の写真で説明する。

> [image: russian-doll-from-wikipedia.png]

この写真では、鑑賞者がすべてを見られるよう人形が分解されている。

> **注:** この問題は抽象的、あるいは不条理にさえ感じるかもしれない。ロシア人形をなぜ表現したいのか、そのような表現で何をするのかがはっきりしない。今はただ付き合ってみてほしい。

さて、そのようなロシア人形を BSL データで表現する問題を考えよう。少し想像すれば、職人が任意の個数の人形からなるロシア人形の入れ子を作れることは容易に分かる。結局、与えられたロシア人形の周りにいつでも別の層を巻ける。一方で、内側の奥には中に何もない固い人形があることも分かる。

ロシア人形の各層について、多くのことに関心を持てる。大きさ（ただし入れ子の深さに関係する）、色、表面に描かれた画像、などである。ここでは1つだけ、すなわち人形の色を選び、文字列で表す。そうすると、ロシア人形の各層には2つの性質があることが分かる。その色と、内側にある人形である。2つの性質を持つ情報片を表すには、常に構造体型を定義する。

```racket
(define-struct layer [color doll])
```

そしてデータ定義を加える。

```racket
; An RD (short for Russian doll) is one of:
; – String
; – (make-layer String RD)
```

当然、このデータ定義の第1節は最も内側の人形、正確にはその色を表す。第2節は、与えられたロシア人形の周りに層を加えるためのものである。これを `layer` のインスタンスで表す。それには明らかに人形の色と、もう1つのフィールド、すなわちこの人形のすぐ内側に入れ子になった人形が含まれる。

次の人形を見よ。

> [image: pict_99.png]

これは3つの人形からなる。赤いものが最も内側、緑が中央、黄が現在の最も外側の包みである。この人形を `RD` の要素として表すには、どちらの端から始めてもよい。内側から外側へ進む。赤い人形を `RD` として表すのは簡単である。中に何もなく、赤なので、文字列 `"red"` で十分である。第2の層には次を使う。

```racket
(make-layer "green" "red")
```

これは、緑の（中空の）人形が赤い人形を含むと言う。最後に、外側を得るには、この最後の人形の周りにもう1層巻く。

```racket
(make-layer "yellow" (make-layer "green" "red"))
```

この過程で、任意の色付きロシア人形の組からデータ表現へどう進むかがよく分かるはずである。しかしプログラマは逆もできなければならない、すなわちデータ片から具体的な情報へ進むこともできなければならない。この精神で、次の `RD` の要素について略図のロシア人形を描け。

```racket
(make-layer "pink" (make-layer "black" "white"))
```

BSL で試してみてもよい。

データ定義があり、実際の人形をどう表すか、`RD` の要素を人形としてどう解釈するかを理解したので、`RD` を消費する関数を設計する準備ができた。具体的には、ロシア人形の組に何個の人形が含まれるかを数える関数を設計しよう。この文はよい目的文であり、シグネチャも決める。

```racket
; RD -> Number
; how many dolls are part of an-rd
```

データ例としては、`(make-layer "yellow" (make-layer "green" "red"))` から始めよう。上の画像は、期待される答えが `3` だと教える。赤、緑、黄の3つの人形があるからである。この1つの例を通して考えるだけで、入力が次の人形の表現であるとき

> [image: pict_100.png]

答えが `1` であることも分かる。

第4ステップはテンプレートの開発を要求する。このステップの標準的な質問を使うと、次のテンプレートが得られる。

```racket
; RD -> Number
; how many dolls are a part of an-rd
(define (depth an-rd)
  (cond
    [(string? an-rd)...]
    [(layer? an-rd)
     (... (layer-color an-rd)...
... (depth (layer-doll an-rd))...)]))
```

`cond` 節の数は `RD` の定義の節の数で決まる。各節はそれがどんな種類のデータについてのものかを具体的に述べ、それが `string?` と `layer?` の述語を使うよう教える。文字列は複合データではないが、`layer` のインスタンスは2つの値を含む。関数がこれらの値を必要とするなら、セレクタ式 `(layer-color an-rd)` と `(layer-doll an-rd)` を使う。最後に、データ定義の第2節は `layer` 構造体の `doll` フィールドから定義自体への自己参照を含む。したがって第2のセレクタ式に再帰的関数呼び出しが要る。

例とテンプレートが関数定義をほとんど指示する。非再帰の `cond` 節では答えは明らかに `1` である。再帰節では、テンプレートの式は次の結果を計算する。

- `(layer-color an-rd)` は現在の層の色を記述する文字列を取り出す。
- `(layer-doll an-rd)` は現在の層の中に含まれる人形を取り出す。
- `(depth (layer-doll an-rd))` は、`depth` の目的文に従い、`(layer-doll an-rd)` に何個の人形が含まれるかを決める。

この最後の数は望ましい答えにほぼ近いが、完全ではない。なぜなら `an-rd` と `(layer-doll an-rd)` の差は1層、すなわち1つ余分な人形だからである。言い換えれば、関数は実際の答えを得るために再帰結果に `1` を加えなければならない。

```racket
; RD -> Number
; how many dolls are a part of an-rd
(define (depth an-rd)
  (cond
    [(string? an-rd) 1]
    [else (+ (depth (layer-doll an-rd)) 1)]))
```

関数定義が第2節で `(layer-color an-rd)` を使わないことに注意せよ。ここでもまた、テンプレートはデータ定義について知っているすべてを整理するスキーマだが、実際の定義ではそのすべての断片が要るとは限らないことが分かる。

最後に例をテストへ翻訳しよう。

```racket
(check-expect (depth "red") 1)
(check-expect
  (depth
   (make-layer "yellow" (make-layer "green" "red")))
  3)
```

これらを DrRacket で実行すれば、その評価が `depth` の定義のすべての断片に触れることが分かる。


### 9.5 リストと World（Lists and World）

リスト、および一般に自己参照データ定義があれば、有限データよりもはるかに多くの興味深い World プログラムを設計し実行できる。「項目化と構造体」のスペースインベーダープログラムで、プレイヤーが戦車から望むだけ多くのショットを撃てる版を作れると想像してほしい。この問題の単純化した版から始めよう。

> **注:** しばらく World プログラムを設計していないなら、「World プログラムの設計」を読み返せ。

> **サンプル問題** ショットの発射をシミュレートする World プログラムを設計せよ。「プレイヤー」がスペースバーを押すたびに、プログラムはキャンバスの底にショットを加える。これらのショットはチックごとに1ピクセルの速さで垂直に上昇する。

World プログラムの設計は、情報を定数と、絶えず変わる世界状態の要素へ分けることから始まる。前者には物理定数と図形定数を導入し、後者には世界状態のデータ表現を開発する必要がある。サンプル問題は細部については比較的曖昧だが、ショットが垂直線に沿って描かれる長方形の風景を明確に想定する。明らかにショットの位置は毎チック変わるが、風景の大きさとショット列の x 座標は同じままである。

```racket
(define HEIGHT 80); distances in terms of pixels
(define WIDTH 100)
(define XSHOTS (/ WIDTH 2))

; graphical constants
(define BACKGROUND (empty-scene WIDTH HEIGHT))
(define SHOT (triangle 3 "solid" "red"))
```

問題文のどこにもこれらの特定の選択を要求するものはないが、簡単に変えられる——つまり1つの定義を編集するだけで変えられる——限り、目標は達成されている。

「世界」のうち変わる側面について、問題文は2つ述べる。第一に、スペースバーを押すとショットが加わる。第二に、すべてのショットはチックごとに1ピクセル真上へ動く。プレイヤーが何発「撃つ」かを予測できないので、リストで表す。

```racket
; A List-of-shots is one of:
; – '()
; – (cons Shot List-of-shots)
; interpretation the collection of shots fired
```

残る問いは、個々のショットをどう表すかである。すべてが同じ x 座標を持ち、この座標は終始変わらないことはすでに分かっている。さらに、すべてのショットは同じ見た目である。したがって y 座標だけが互いに異なる唯一の性質である。よって各ショットを数として表せば十分である。

```racket
; A Shot is a Number.
; interpretation represents the shot's y-coordinate
```

ショットの表現を `HEIGHT` 未満の数の区間に制限することもできる。すべてのショットはキャンバスの底から発射され、その後上へ動く、すなわち y 座標が連続的に減ることが分かっているからである。

この世界を表すのに、次のようなデータ定義も使える。

```racket
; A ShotWorld is List-of-numbers.
; interpretation each number on such a list
;   represents the y-coordinate of a shot
```

結局、上の2つの定義はすべての数のリストを記述する。数のリストの定義はすでにあり、名前 `ShotWorld` がこのデータクラスが何についてのものかを皆に伝える。

定数を定義し世界状態のデータ表現を開発したあとの主要な仕事は、どのイベントハンドラを使うかを選び、シグネチャを与えられた問題に合わせることである。進行中の例はクロックチックとスペースバーに言及し、それは3つの関数の願望リストに翻訳される。

- 世界状態を画像へ変える関数：

    ; ShotWorld -> Image
    ; adds the image of a shot for each y on w
    ; at (MID,y) to the background image
    (define (to-image w)
      BACKGROUND)

  問題が視覚的な描画を要求するからである。

- チックイベントを扱うもの：

    ; ShotWorld -> ShotWorld
    ; moves each shot on w up by one pixel
    (define (tock w)
      w)

- キーイベントを扱う関数：

    ; ShotWorld KeyEvent -> ShotWorld
    ; adds a shot to the world
    ; if the player presses the space bar
    (define (keyh w ke)
      w)

初期の願望リストに加え、実際に世界を設定しハンドラをインストールする `main` 関数も定義する必要があることを忘れるな。図61はこの1つの関数を含む。それは設計されるのではなく、標準スキーマの修正として定義される。

まず `to-image` の設計から始めよう。シグネチャ、目的文、ヘッダがあるので、次は例が要る。データ定義には2節があるので、少なくとも2つの例が要る。`'()` と、たとえば `(cons 9 '())` のような `cons` されたリストである。`'()` の期待結果は明らかに `BACKGROUND` である。y 座標がある場合は、関数は `MID` と指定された座標にショットの画像を置かなければならない。

```racket
(check-expect (to-image (cons 9 '()))
              (place-image SHOT XSHOTS 9 BACKGROUND))
```

読み進める前に、2つの `Shot` のリストに `to-image` を適用する例を通して考えよ。そうすると関数の動きが分かりやすくなる。

第4ステップはデータ定義をテンプレートへ翻訳することである。

```racket
; ShotWorld -> Image
(define (to-image w)
  (cond
    [(empty? w)...]
    [else
     (... (first w)... (to-image (rest w))...)]))
```

リストのデータ定義向けテンプレートは今や非常に馴染み深いので、多くの説明は要らない。疑わしいなら図52の質問を読み、自分でテンプレートを設計せよ。

ここから関数を定義するのは素直である。鍵は例とテンプレートを組み合わせ、図53の質問に答えることである。それに従い、空のショットリストという基本ケースから始め、例から期待答えが `BACKGROUND` だと分かる。次に第2の `cond` のテンプレート式が何を計算するかを定式化する。

- `(first w)` はリストから最初の座標を取り出す。
- `(rest w)` は残りの座標である。
- `(to-image (rest w))` は、`to-image` の目的文に従い、`(rest w)` 上の各ショットを背景画像に加える。

言い換えれば、`(to-image (rest w))` はリストの残りを画像として描画し、ほとんどすべての仕事をする。足りないのは最初のショット `(first w)` である。これら2つの式に目的文を適用すると、第2の `cond` 節の望ましい式が得られる。

```racket
(place-image SHOT XSHOTS (first w)
             (to-image (rest w)))
```

加えられるアイコンはショットの標準画像である。2つの座標は目的文に書かれている。そして `place-image` の最後の引数は、リストの残りから構築された画像である。

図61は `to-image` の完全な関数定義と、プログラムの残りも示す。`tock` の設計は `to-image` の設計と同じであり、自分で通して考えよ。一方、`keyh` ハンドラのシグネチャは1つの興味深い問いを立てる。ハンドラが非自明なデータ定義を持つ2つの入力を消費すると指定する。一方で `ShotWorld` は自己参照データ定義である。他方で `KeyEvent` の定義は大きな列挙である。今のところ、2つの引数のどちらがテンプレートの開発を駆動すべきかを「推測」してもらう。後でこのような場合を深く学ぶ。

```
+------------------------------------------------------+
| Figure 61: A list-based world program                |
|                                                      |
|; ShotWorld -> ShotWorld                             |
| (define (main w0)                                    |
|   (big-bang w0                                       |
|     [on-tick tock]                                   |
|     [on-key keyh]                                    |
|     [to-draw to-image]))                             |
|                                                      |
|; ShotWorld -> ShotWorld                             |
|; moves each shot up by one pixel                    |
| (define (tock w)                                     |
|   (cond                                              |
|     [(empty? w) '()]                                 |
|     [else (cons (sub1 (first w)) (tock (rest w)))])) |
|                                                      |
|; ShotWorld KeyEvent -> ShotWorld                    |
|; adds a shot to the world if the space bar is hit   |
| (define (keyh w ke)                                  |
|   (if (key=? ke " ") (cons HEIGHT w) w))             |
|                                                      |
|; ShotWorld -> Image                                 |
|; adds each shot y on w at (XSHOTS,y} to BACKGROUND  |
| (define (to-image w)                                 |
|   (cond                                              |
|     [(empty? w) BACKGROUND]                          |
|     [else (place-image SHOT XSHOTS (first w)         |
|                        (to-image (rest w)))]))       |
+------------------------------------------------------+
```

World プログラムに関する限り、`keyh` のようなキーハンドラは、それが消費するキーイベントについてのものである。したがってそれを主な引数とみなし、そのデータ定義からテンプレートを導く。具体的には、「列挙」の `KeyEvent` のデータ定義に従い、関数には次のような多数の節を持つ `cond` 式が要ると指示する。

```racket
(define (keyh w ke)
  (cond
    [(key=? ke "left")...]
    [(key=? ke "right")...]
...
    [(key=? ke " ")...]
...
    [(key=? ke "a")...]
...
    [(key=? ke "z")...]))
```

もちろん、可能なすべての BSL 値を消費する関数と同様に、キーハンドラは通常すべての可能なケースを検査する必要はない。進行中の問題については、キーハンドラがスペースバーにだけ反応し、他はすべて無視することを具体的に知っている。したがって `" "` の節を除き、すべての `cond` 節を `else` 節へ畳むのが自然である。

### 9.6 リストと集合についての注（A Note on Lists and Sets）

本書は、BSL 値の集まりとしての集合に対する、読者の直感的理解に頼る。「データの宇宙」は、データ定義が BSL 値の集合の名前を導入すると具体的に述べる。本書が集合について一貫して問う問いが1つあり、それはある要素が与えられた集合に含まれるかどうかである。たとえば `4` は `Number` に含まれるが、`"four"` は含まれない。本書はまた、ある値が名前付き集合のメンバーかどうかを確認するためにデータ定義を使う方法や、一部のデータ定義を使って集合のサンプル要素を生成する方法も示すが、これら2つの手続きはデータ定義についてのものであり、集合そのものについてではない。

同時に、リストは値の集まりを表す。したがってリストと集合の違いは何か、あるいはこれは不要な区別なのかと疑問に思うかもしれない。そうなら、本節はあなたのためのものである。

今のところ集合とリストの主な違いは、前者がコード設計のステップを議論するために使う概念であり、後者が選んだプログラミング言語 BSL の多くのデータ形式の1つだということである。2つの考えは会話のかなり異なる水準に住む。しかし、データ定義が BSL 内部で実際の情報のデータ表現を導入すること、集合が情報の集まりであることを踏まえると、集合が BSL 内部でデータとしてどう表されるかを今や自問してよい。

> **注:** ほとんどの本格的言語は、リストと集合の両方のデータ表現を直接サポートする。

リストは BSL で特別な地位を持つが、集合はそうではない。しかし同時に、集合はどこかリストに似る。主な違いは、プログラムが通常どちらのデータ形式と一緒に使う関数の種類である。BSL はリスト向けのいくつかの基本定数と関数——たとえば `empty`、`empty?`、`cons`、`cons?`、`first`、`rest`——と、自分で定義できるいくつかの関数——たとえば `member?`、`length`、`remove`、`reverse` など——を提供する。次は自分で定義できるが BSL には付属しない関数の例である。

```racket
; List-of-string String -> N
; determines how often s occurs in los
(define (count los s)
  0)
```

ストップ！ この関数の設計を完成させよ。

素直で、おそらく素朴な仕方で進め、集合は基本的にリストだとしよう。さらに簡単にするため、本節では数のリストに焦点を当てる。ある数が集合の一部かどうかが重要だと認めるなら、リストを2通りの仕方で集合を表すのに使えることがほぼすぐに明らかになる。

```
+--------------------------------------------------------------------------------+
| Figure 62: Two data representations for sets                                   |
|                                                                                |
| +--------------------------------------+--+----------------------------------- |
| |; A Son.L is one of: ⏎; – empty ⏎ … | |; A Son.R is one of: ⏎; – empty ⏎  |
| … |                                                                            |
| +--------------------------------------+--+----------------------------------- |
| |;ASon.L is one of: | | |                                                     |
| |;–empty | | |                                                                |
| |; – (cons Number Son.L) | | |                                                |
| |; | | |                                                                      |
| |;Son is used when it | | |                                                   |
| |;applies toSon.L andSon.R | | |                                              |
| |;ASon.R is one of: | | |                                                     |
| |;–empty | | |                                                                |
| |; – (cons Number Son.R) | | |                                                |
| |; | | |                                                                      |
| |;Constraint Ifs is aSon.R, | | |                                             |
| |;no number occurs twice ins | | |                                            |
| +--------------------------------------+--+----------------------------------- |
+--------------------------------------------------------------------------------+
```

図62は2つのデータ定義を示す。どちらも基本的に、集合は数のリストとして表されると述べる。違いは、右側の定義にはリスト上で同じ数が2回以上現れてはならないという制約が付くことである。結局、集合について問う主要な問いはある数が集合に含まれるかどうかであり、1回、2回、3回含まれるかは違いを生まない。

どちらの定義を選んでも、すでに2つの重要な概念を定義できる。

```racket
; Son
(define es '())

; Number Son -> Boolean
; is x in s
(define (in? x s)
  (member? x s))
```

最初のものは空集合であり、どちらの場合も空リストで表される。2番目は所属判定である。

より大きな集合を作る1つの方法は、`cons` と上の定義を使うことである。たとえば `1`、`2`、`3` を含む集合の表現を作りたいとしよう。そのような表現の1つは次である。

```racket
(cons 1 (cons 2 (cons 3 '())))
```

そしてそれは両方のデータ表現で動く。しかし、

```racket
(cons 2 (cons 1 (cons 3 '())))
```

は本当に同じ集合の表現ではないのか？ あるいは次はどうか。

```racket
(cons 1 (cons 2 (cons 1 (cons 3 '()))))
```

主な関心が数が集合に含まれるかどうかである限り、答えは肯定でなければならない。それでも、`cons` の順序は問題になり得ないが、右側のデータ定義の制約は、最後のリストを `Son.R` として排除する。`1` を2回含むからである。

```
+--------------------------------------------------------------------------------+
| Figure 63: Functions for the two data representations of sets                  |
|                                                                                |
| +--------------------------------------+--+----------------------------------- |
| |; Number Son.L -> Son.L ⏎; removes… | |; Number Son.R -> Son.R ⏎;         |
| removes… |                                                                     |
| +--------------------------------------+--+----------------------------------- |
| |;NumberSon.L ->Son.L | | |                                                   |
| |;removesx froms | | |                                                        |
| | (define s1.L | | |                                                           |
| | (cons 1 (cons 1 '()))) | | |                                                 |
| | (check-expect | | |                                                          |
| | (set-.L 1 s1.L) es) | | |                                                    |
| | (define (set-.L x s) | | |                                                   |
| | (remove-all x s)) | | |                                                      |
| |;NumberSon.R ->Son.R | | |                                                   |
| |;removesx froms | | |                                                        |
| | (define s1.R | | |                                                           |
| | (cons 1 '())) | | |                                                          |
| | (check-expect | | |                                                          |
| | (set-.R 1 s1.R) es) | | |                                                    |
| | (define (set-.R x s) | | |                                                   |

```

2つのデータ定義の違いは、関数を設計するときに現れる。たとえば集合から数を除く関数が欲しいとしよう。両方の表現に適用される願望リスト項目は次の通りである。

```racket
; Number Son -> Son
; subtracts x from s
(define (set- x s)
  s)
```

目的文が「subtract（差し引く）」という言葉を使うのは、論理学者や数学者が集合を扱うときに使う語だからである。

図63は結果を示す。2つの列は2点で異なる。

1. 左側のテストは `1` を2回含むリストを使い、右側のテストは同じ集合を1つの `cons` で表す。
2. これらの違いのため、左側の `set-` は `remove-all` を使わなければならず、右側は `remove` で済む。

ストップ！ コードを DrRacket の定義領域にコピーし、テストが通ることを確かめよ。その後読み進め、コードで実験せよ。

図63の好ましくない側面は、テストが期待結果としてプレーンなリスト `es` を使うことである。この問題は一見些細に見えるかもしれない。しかし次の例を考えよ。

```racket
(set- 1 set123)
```

ここで `set123` は、`1`、`2`、`3` を含む集合を次の2通りのいずれかで表す。

```racket
(define set123-version1
  (cons 1 (cons 2 (cons 3 '()))))

(define set123-version2
  (cons 1 (cons 3 (cons 2 '()))))
```

どちらの表現を選んでも、`(set- 1 set123)` は次の2つのリストの一方へ評価される。

```racket
(define set23-version1
  (cons 2 (cons 3 '())))

(define set23-version2
  (cons 3 (cons 2 '())))
```

しかし `set-` がどちらを生成するかを予測することはできない。

2つの代替という単純なケースでは、次のように `check-member-of` テスト機能を使える。

```racket
(check-member-of (set-.v1 1 set123.v1)
                 set23-version1
                 set23-version2)
```

期待される集合が3要素を含むなら、左側のデータ定義が許す重複表現を除いても、6通りの可能な変異がある。

この問題を直すには2つの考えの組み合わせが要る。第一に、`set-` は本当は与えられた要素が結果に現れないことを保証することについてのものだと思い出せ。それは例をテストへ直す我々の仕方では伝わらない考えである。第二に、BSL の `check-satisfied` テスト機能を使えば、この考えを正確に述べられる。

「インターメッツォ1：Beginning Student Language」は `check-satisfied` に簡単に触れるが、要するにこの機能はある式が特定の性質を満たすかどうかを判定する。性質とは値からブール値への関数である。我々の具体的なケースでは、`1` が集合のメンバーでないことを述べたい。

```racket
; Son -> Boolean
; #true if 1 is not a member of s;  #false otherwise
(define (not-member-1? s)
  (not (in? 1 s)))
```

`not-member-1?` を使い、テストケースを次のように定式化できる。

```racket
(check-satisfied (set- 1 set123) not-member-1?)
```

そしてこの変種は、関数が何を達成すべきかを明確に述べる。さらに良いことに、この定式化は入力や出力の集合がどう表されているかにまったく依存しない。

要約すると、リストと集合はどちらも値の集まりについてのものである点で関連するが、強く異なる。

| 性質 | リスト | 集合 |
| :--- | :--- | :--- |
| 所属（membership） | 多くのうちの1つ | 決定的 |
| 順序（ordering） | 決定的 | 無関係 |
| 出現回数（# of occurrences） | 意味がある | 無関係 |
| 大きさ（size） | 有限だが任意 | 有限または無限 |

表の最後の行は新しい考えを提示するが、明白でもある。本書で言及される集合の多くは無限に大きい。たとえば `Number`、`String`、そして `List-of-strings` もである。対照的に、リストは任意に多くの項目を含み得るが、常に有限である。

要約すると、本節は集合とリストの本質的な違いと、有限リストで有限集合を2通りの仕方で表す方法を説明した。BSL は無限集合を表すのに十分な表現力を持たない。練習問題299は無限集合にも対処できるまったく異なる集合表現を導入する。しかし実際のプログラミング言語が集合をどう表すかという問いは、本書の範囲を超える。
