# 9 自己参照データ定義を用いた設計 (Designing with Self-Referential Data Definitions)

一見すると、自己参照データ定義は、混合データに対するものよりもはるかに複雑に見えるかもしれません。しかし、`contains-flatt?` の例が示しているように、設計レシピの6つのステップは引き続き有効に機能します。それにもかかわらず、このセクションでは、自己参照データ定義に対して設計レシピをさらに効果的に適用できるように一般化します。新しい部分は、自己参照データ定義が必要とされる時期の発見、テンプレートの導出、および関数本体の定義プロセスに関するものです。

1. **問題分析**: 問題文が「任意のサイズの情報」に関するものである場合、それを表現するために自己参照データ定義が必要です。現時点では、そのようなクラスとして `List-of-names` のみを見てきました。図51の左側は、同様の方法で `List-of-strings` を定義する方法を示しています。他のアトミックなデータのリストも同様に機能します。
   （※補足：数値も任意に大きくなるように見えます。不正確な数（浮動小数点数など）の場合、これは錯覚です。しかし、正確な整数（多倍長整数など）の場合には、実際に任意に大きくなります。したがって、整数を扱うこともこの章の一部です。）

   自己参照データ定義が有効であるためには、2つの条件を満たす必要があります。第一に、少なくとも2つの節（clause）を含んでいる必要があります。第二に、少なくとも1つの節は、定義されているデータクラス自体を参照し返してはなりません。データ定義における自己参照から、定義されている用語へ戻る矢印を明示的に識別することが良い習慣です。このような注記の例については、図51を参照してください。

   データ例の作成を通じて、自己参照データ定義の妥当性を確認する必要があります。まず、データ定義を参照しない節から始めます。次に、その最初の例を使用して、自身を参照する節を使って次の例を作成します。図51のデータ定義については、結果として以下のような3つのリストが得られます。

   * `'()` : 第1節による。
   * `(cons "a" '())` : 第2節と、前述の例による。
   * `(cons "b" (cons "a" '()))` : 第2節と、前述の例による。

   もしデータ定義から例を生成することが不可能であるなら、その定義は無効です。例を生成することはできても、ますます大きな例を生成する方法が見出せない場合、その定義は意図した解釈に沿っていない可能性があります。

2. **ヘッダー**: シグネチャ、目的ステートメント、およびダミー定義（スタブ）に関しては何も変わりません。目的ステートメントを定式化するときは、関数が「何を（what）」計算するのかに焦点を当て、「どのように（how）」それを処理するのか（特に、与えられたデータのインスタンスをどのように辿るか）には焦点を当てないでください。

   この設計レシピを具体化するための例を示します。

   ```racket
   ;; List-of-strings -> Number
   ;; alos に含まれる文字列の個数を数える
   (define (how-many alos)
     0)
   ```

   この目的ステートメントは、関数が与えられた入力内の文字列を単に数えるだけであることを明確に述べています。このアイデアをBSLの関数としてどのように定式化するかを前もって思い悩む必要はありません。

3. **例**: 機能的な例を作成する際には、データ定義の自己参照節を複数回使用した入力について必ず検討してください。これは、後で関数定義全体をカバーするテストを定式化するための最良の方法です。

   私たちの進行中の例については、目的ステートメント自体がデータ例からほぼ自動的に機能例を生成します。

   * `(how-many '())` に対して期待される結果は `0`。空リストには何も含まれていないためです。
   * `(how-many (cons "a" '()))` に対して期待される結果は `1`。1つの文字列 of リストだからです。
   * `(how-many (cons "b" (cons "a" '())))` に対して期待される結果は `2`。2つの文字列 of リストについてです。

4. **テンプレート**: 本質的に、自己参照データ定義は混合データのデータ定義と似ています。したがって、テンプレートの開発は、第I部「項目化と構造体」のレシピに従って進めることができます。具体的には、データ定義の節と同じ数の節を持つ `cond` 式を定式化し、各識別条件をデータ定義の対応する節に一致させ、複合データを処理するすべての `cond` 行に適切なセレクタ式を書き留めます。

   図52はこの考え方を問答形式で表現したものです。

   **図52: 自己参照データに対する関数の設計テンプレート質問**
   * **質問**: データ定義はデータの異なるサブクラスを区別しているか？
     * **回答**: テンプレートには、データ定義が区別するサブクラスと同じ数の `cond` 節が必要です。
   * **質問**: サブクラスは互いにどのように異なっているか？
     * **回答**: その違いを利用して、節ごとに条件（テスト）を定式化します。
   * **質問**: 構造化された値を扱う節はあるか？
     * **回答**: もしあるなら、その節に適切なセレクタ式を追加します。
   * **質問**: データ定義は自己参照を使用しているか？
     * **回答**: テンプレートがデータ定義の自己参照を表すように、「自然な再帰」を定式化します。
   * **質問**: データ定義が他のデータ定義を参照している場合、この相互参照はどこにあるか？
     * **回答**: もう一つのデータ定義のためにテンプレートを特殊化し、そのテンプレートを参照します（第I部「項目化の設計、再び」のステップ4・5を参照）。

   最後の行を無視して、最初の3つの質問を `List-of-strings` を受け取る任意の関数に適用すると、次のような形状になります。

   ```racket
   (define (fun-for-los alos)
     (cond
       [(empty? alos) ...]
       [else
        (... (first alos) ... (rest alos) ...)]))
   ```

   しかし、テンプレートの目的は、データ定義を関数のレイアウトとして表現することであるのを思い出してください。つまり、テンプレートは、入力のデータ定義が英語と BSL の混在で表現している内容をコードとして表現します。したがって、データ定義のすべての重要な部分はテンプレートに対応するものを見つけなければならず、このガイドラインはデータ定義が自己参照的である（定義の内部から定義される用語への矢印が含まれる）場合にも適用されるべきです。特に、データ定義の第 $i$ 節で言及されている構造体の第 $k$ フィールドで自己参照している場合、テンプレートは第 $i$ 番目の `cond` 節の第 $k$ フィールドのセレクタ式において自己参照するべきです。そのようなセレクタ式ごとに、関数のパラメータに戻る矢印を追加します。最終的に、テンプレートにはデータ定義と同じ数の矢印が存在しなければなりません。

   図51は、この考え方を `List-of-strings` を受け取る関数のテンプレートとデータ定義を並めて示しています。両方とも、第2節（それぞれ `rest` フィールドと `rest` セレクタ）から始まり、それぞれの定義のトップを指し示す1つの矢印を含んでいます。

   BSLやほとんどのプログラミング言語はテキスト指向であるため、矢印の代替手段、すなわち**適切なセレクタ式に対する関数の自己適用**を使用する必要があります。

   ```racket
   (define (fun-for-los alos)
     (cond
       [(empty? alos) ...]
       [else
        (... (first alos) ... (fun-for-los (rest alos)) ...)]))
   ```

   関数の自己利用を**再帰（recursion）**と呼び、この本の最初の4つの部では**自然な再帰（natural recursion）**と呼びます。

5. **定義**: 関数本体については、再帰的な関数呼び出しを含まない `cond` 行、すなわち**基本ケース（base cases）**から始めます。対応する答えは通常、定式化が容易であるか、すでに例として与えられています。

   次に、自己参照のケースを処理します。まず、テンプレート行の各式が何を計算するかを自分自身に思い出させることから始めます。自然な再帰については、関数が目的ステートメントで指定された通りにすでに動作していると仮定します。この最後のステップは「信頼の飛躍（leap of faith）」ですが、見ての通り、常に機能します。

   **残る作業は、さまざまな値を組み合わせることです。**

   **図53: テンプレートを関数定義に変換する手順**
   * **質問**: 非再帰的な `cond` 節に対する答えは何か？
     * **回答**: 例がここで必要な値を示しているはずです。そうでなければ、適切な例とテストを定式化してください。
   * **質問**: 再帰節のセレクタ式は何を計算するか？
     * **回答**: データ定義がこれらの式がどのような種類のデータを抽出するかを教えてくれ、データ定義の解釈がこのデータが何を表すかを教えてくれます。
   * **質問**: 自然な再帰は何を計算するか？
     * **回答**: 関数の目的ステートメントを使用して、再帰の値が何を意味するかを決定します（どのようにこの答えを計算するかではありません）。目的ステートメントが答えを教えてくれない場合は、目的ステートメントを改善してください。
   * **質問**: 関数は望ましい答えを得るために、これらの値をどのように組み合わせることができるか？
     * **回答**: 値を組み合わせるBSLの関数を見つけます。それが機能しない場合は、補助関数を「願望（wish）」として設計します。多くの関数にとって、この最後のステップは簡単です。目的、例、およびテンプレートが組み合わさって、利用可能な値を適切な結果に組み合わせる（combine）関数や式を教えてくれます。この関数や式を、既存の用語を少し拡張して「結合子（combinator）」と呼びます。

   **図54: テンプレートから関数への変換、テーブル法**
   * **質問**: ここで行き詰まった場合はどうすればよいか？
     * **回答**: 第3のステップの例をテーブルに整理します。左端の列に与えられた入力を配置し、右端の列に望ましい出力を配置します。中間の列に、セレクタ式と自然な再帰の値を入力します。結合子を示唆するパターンが現れるまで、例の行を追加してください。
   * **質問**: テンプレートが他のテンプレートを参照している場合、補助関数は何を計算するか？
     * **回答**: その補助関数の目的ステートメントと例を参照して何を計算するかを判断し、そのヘルパー関数の設計が完了していなくても、その結果を利用できると仮定します。

   図53はこのステップに関する最初の4つの質問と回答を定式化したものです。このゲームを使って `how-many` の定義を完成させましょう。`fun-for-los` テンプレートを `how-many` に名前変更すると、ここまでのものが得られます。

   ```racket
   ;; List-of-strings -> Number
   ;; alos に含まれる文字列の個数を判定する
   (define (how-many alos)
     (cond
       [(empty? alos) ...]
       [else
        (... (first alos) ... (how-many (rest alos)) ...)]))
   ```

   機能的な例がすでに示唆しているように、基本ケースに対する答えは `0` です。2番目の節の2つの式は、最初の項目と `(rest alos)` 内の文字列の個数を計算します。`alos` 全体に何個の文字列があるかを計算するには、関数は後者の式の値に `1` を加算するだけです。

   ```racket
   (define (how-many alos)
     (cond
       [(empty? alos) 0]
       [else
        (+ (how-many (rest alos)) 1)]))
   ```

   値を望ましい答えに組み合わせる正しい方法を見つけることは、常にこれほど簡単であるとは限りません。初心者のプログラマは、このステップでよく行き詰まります。図54が示唆するように、機能的な例を、テンプレート内の式の値も明記したテーブルに整理することをお勧めします。図55は、`how-many` の例におけるこのテーブルの様子を示しています。左端の列にはサンプル入力をリストし、右端の列にはこれらの入力に対する望ましい答えを含めます。その間の3つの列は、テンプレート式の値を示しています：`(first alos)`、`(rest alos)`、そして自然な再帰である `(how-many (rest alos))` です。このテーブルを十分に長く見つめれば、結果の列が常に自然な再帰の列の値よりも `1` だけ大きいことに気づくでしょう。したがって、あなたは以下のように推測するかもしれません。

   ```racket
   (+ (how-many (rest alos)) 1)
   ```

   これが望ましい結果を計算する式です。DrRacketはこの種の推測を素早くチェックできるため、それをプラグインして「RUN」をクリックします。例がテストに合格した場合は、その式がすべてのリストで機能することを確信するために思考を巡らせます。そうでない場合は、別のアイデアが得られるまでテーブルにさらに多くの例の行を追加します。

   **図55: 引数、中間値、および結果の集計 (Tabulating arguments, intermediate values, and results)**
   | `alos` | `(first alos)` | `(rest alos)` | `(how-many (rest alos))` | `(how-many alos)` |
   | :--- | :--- | :--- | :--- | :--- |
   | `(cons "a" '())` | `"a"` | `'()` | `0` | `1` |
   | `(cons "b" (cons "a" '()))` | `"b"` | `(cons "a" '())` | `1` | `2` |
   | `(cons "x" (cons "b" (cons "a" '())))` | `"x"` | `(cons "b" (cons "a" '()))` | `2` | `3` |

   このテーブルは、テンプレート内のいくつかのセレクタ式が実際の定義においては無関係である可能性もあることを指摘しています。ここでは、最終的な答えを計算するために `(first alos)` は必要ありません。これは、テンプレートの画一的な両方の式を使用する `contains-flatt?` とは対照的です。

   この本の残りの部分を進めるにあたり、多くの場合、組み合わせステップは BSL のプリミティブ（例えば `+`、`and`、`cons`）で表現できることを覚えておいてください。しかし、場合によっては、補助関数を設計（願望）しなければならないこともあります。また、入れ子になった条件が必要な場合もあります。

6. **テスト**: 最後に、すべての例をテストに変換し、これらのテストが合格し、それらを実行することが関数のすべての部分をカバーしていることを確認してください。

   `how-many` の例をテストに変換したものは以下の通りです。

   ```racket
   (check-expect (how-many '()) 0)
   (check-expect (how-many (cons "a" '())) 1)
   (check-expect (how-many (cons "b" (cons "a" '()))) 2)
   ```

   例を直接テストとして定式化するのが最善であり、BSLはこれを可能にします。そうすることは、前ステップのテーブルベースの推測アプローチに頼る必要がある場合にも役立ちます。

   図56は、このセクションの設計レシピを要約したものです。この図は、この章で用いるような自己参照リストの定義に合わせて調整されています。

   **図56: 自己参照データに対する関数の設計 (Designing a function for self-referential data)**
   | ステップ (steps) | 成果 (outcome) | 活動 (activity) |
   | :--- | :--- | :--- |
   | 問題分析 (problem analysis) | データ定義 (data definition) | 情報のためのデータ表現を構築する。具体的な情報について例を作成し、データを情報として解釈する。自己参照を識別する。 |
   | ヘッダー (header) | シグネチャ; 目的; ダミー定義 | 定義された名前を使ってシグネチャを書き下ろす。簡潔な目的ステートメントを定式化する。指定された値の範囲から定数値を生成するダミー関数を作成する。 |
   | 例 (examples) | 例とテスト | いくつかの例を作成する。データ定義の各節につき、少なくとも1つは作成する。 |
   | テンプレート (template) | 関数テンプレート | データ定義をテンプレートに翻訳する：データ節ごとに1つの `cond` 節；条件が構造体を識別する場所にはセレクタ；自己参照ごとに1つの自然な再帰。 |

**Exercise 141.** 文字列のリストを受け取り、それらすべてを結合して1つの長い文字列にする `cat` 関数の設計を求められた場合、以下のような不完全な定義に必ず辿り着くことになります。

```racket
;; List-of-string -> String
;; l に含まれるすべての文字列を1つの長い文字列に結合する

(check-expect (cat '()) "")
(check-expect (cat (cons "a" (cons "b" '()))) "ab")
(check-expect (cat (cons "ab" (cons "cd" (cons "ef" '())))) "abcdef")

(define (cat l)
  (cond
    [(empty? l) ""]
    [else (... (first l) ... (cat (rest l)) ...)]))
```

この定義を完成させなさい。残された作業は、`(first l)` と `(cat (rest l))` の値を組み合わせて望ましい答えを得るための結合子を見つけることです。ヒント：文字列に関する第1部（「アトミックデータ」）のガイドを参照してください。

## 9.2 空でないリスト (Non-empty Lists)

これで、`cons` を使用してリストのデータ定義を作成する方法を十分に理解できます。前のセクションの最後にある練習問題のいくつか（あるいはすべて）を解いたならば、さまざまな種類の数値のリスト、Boolean値のリスト、画像のリストなどを扱うことができるようになっています。このセクションでは、リストとは何か、そしてそれをどのように処理するかについてさらに探求を続けます。

まずは、温度のリストの平均値を計算するという、一見簡単そうな問題から始めましょう。簡単にするために、以下のデータ定義を提供します。

```racket
(define ABSOLUTE0 -272)
;; A CTemperature is a Number greater than ABSOLUTE0.

;; A List-of-temperatures is one of:
;; – '()
;; – (cons CTemperature List-of-temperatures)
```

私たちの目的においては、温度をプレーンな数値と考えて差し支えありませんが、2番目のデータ定義は、実際にはすべての数値が温度であるわけではないことを思い出させます。この点を頭に留めておいてください。

ヘッダーに関する記述は素直なものです。

```racket
;; List-of-temperatures -> Number
;; 平均温度を計算する
(define (average alot)
  0)
```

この問題の例を作成するのも簡単ですので、テストを1つだけ定式化します。

```racket
(check-expect (average (cons 1 (cons 2 (cons 3 '())))) 2)
```

期待される結果はもちろん、温度の合計を温度の個数で割ったものです。

少し考えると、`average` のテンプレートはこれまで見てきたものと同様になるはずだとわかります。

```racket
(define (average alot)
  (cond
    [(empty? alot) ...]
    [(cons? alot)
     (... (first alot) ... (average (rest alot)) ...)]))
```

2つの `cond` 節はデータ定義の2つの節を反映しており、条件式は空リストと空でないリストを区別します。そして、データ定義の自己参照のために、自然な再帰が必要となります。

しかし、このテンプレートを関数定義に変換するのはあまりにも困難です。最初の `cond` 節には、空の温度コレクションの平均を表す数値が必要ですが、そのような数値は存在しません。同様に、2番目の節は、1つの温度と、残りの温度の平均値を組み合わせて新しい平均値を計算する関数を要求します。そのような計算方法も不可能ではありませんが、極めて不自然です。

温度の平均を計算するとき、私たちはそれらの合計を個数で割ります。この簡単な例を定式化したときにもそう言いました。この文は、`average` が「合計する」「数える」「割る」という3つのタスクからなる関数であることを示唆しています。「固定サイズデータ」のガイドラインは、1つのタスクにつき1つの関数を書くように教えており、そうすれば `average` の設計は明白になります。

```racket
;; List-of-temperatures -> Number
;; 平均温度を計算する
(define (average alot)
  (/ (sum alot) (how-many alot)))

;; List-of-temperatures -> Number
;; 与えられたリスト内の温度を合計する
(define (sum alot)
  0)

;; List-of-temperatures -> Number
;; 与えられたリスト内の温度を数える
(define (how-many alot)
  0)
```

もちろん、後ろの2つの関数定義は「願望（wishes）」であり、これらについて完全な定義を設計する必要があります。これを行うのは簡単です。なぜなら、前述の `how-many` は `List-of-strings` にも `List-of-temperatures` にも機能し（なぜでしょう？）、`sum` の設計もこれまでと同じお決まりの手順に従うからです。

```racket
;; List-of-temperatures -> Number
;; 与えられたリスト内の温度を合計する
(define (sum alot)
  (cond
    [(empty? alot) 0]
    [else
     (+ (first alot) (sum (rest alot)))]))
```

ちょっと待ってください！`average` の例を参考にして `sum` の例を作成し、そのテストが適切に動作することを確認しなさい。その後、`average` のテストを実行しなさい。

この `average` の定義を読めば、学校で平均について学ぶ内容に直接対応しているため、明らかに正しいことがわかります。それでも、プログラムは私たちのためだけでなく、他人のためにも実行されます。特に、他の開発者はシグネチャを読んで関数を利用し、有益な回答を期待できるべきです。しかし、私たちの `average` の定義は、空の温度リストに対しては機能しません。

**Exercise 143.** `average` を空リストに適用した際、DrRacketでどのような挙動になるか確認しなさい。その後、`'()` に適用されたときに有益なエラーメッセージを生成する関数 `checked-average` を設計しなさい。

数学的には、Exercise 143 は `average` が「部分関数（partial function）」であることを示しています。なぜなら、`'()` に対してエラーを発生させるからです。

代替の解決策は、`average` が空リストに対しては機能しないことをシグネチャを通じて将来の読者に知らせることです。そのためには、`'()` を除外するようなリストのデータ表現が必要になります。例えば次のようになります。

```racket
;; An NEList-of-temperatures is one of:
;; – ???
;; – (cons CTemperature NEList-of-temperatures)
```

問題は、`'()` リストを除外しつつ、他のすべての温度リストを構築できるようにするために、"???" を何に置き換えるべきかということです。1つのヒントは、空リストが最も短いリストであるのに対し、1つの温度のリストが次に短いリストであるということです。このことから、第1節は「1つの温度のすべての可能なリスト」を記述すべきだということが示唆されます。

```racket
;; An NEList-of-temperatures is one of:
;; – (cons CTemperature '())
;; – (cons CTemperature NEList-of-temperatures)
;; interpretation: Celsius temperatures の空でないリスト
```

この定義は以前のリスト定義とは異なりますが、「自己参照」と「自己参照を**用いない**節」という決定的な要素を共有しています。設計レシピを厳格に遵守するためには、定義が意味をなすことを確認するために `NEList-of-temperatures` の例をいくつか作成する必要があります。いつものように基本節から始めるため、例は次のようになります。

```racket
(cons c '())
```

ここで `c` は `CTemperature` を表し、例えば `(cons ABSOLUTE0 '())` のようになります。また、`List-of-temperatures` のすべての空でない要素が、この新しいデータクラスの要素でもあることは明らかです。もし `(cons 2 (cons 3 '()))` がそうであれば、`(cons 1 (cons 2 (cons 3 '())))` も適合します。そして、`(cons 2 (cons 3 '()))` が `NEList-of-temperatures` に属するのは、すでに確認したように `(cons 3 '())` が `NEList-of-temperatures` の要素だからです。`NEList-of-temperatures` のサイズには制限がないことを自分で確認しなさい。

では、誰もが「空でないリスト専用である」とわかるように `average` を設計する問題に戻りましょう。`NEList-of-temperatures` の定義があれば、シグネチャでやりたいことを表現する手段が得られます。
（※この代替的な開発は、この場合、`average` の定義域を狭めて「全域関数（total function）」を作成できることを説明しています。）

```racket
;; NEList-of-temperatures -> Number
;; 平均温度を計算する

(check-expect (average (cons 1 (cons 2 (cons 3 '())))) 2)

(define (average ne-l)
  (/ (sum ne-l) (how-many ne-l)))
```

当然、残りの部分（目的ステートメント、例とテスト、関数定義）は同じままです。結局のところ、平均の計算というアイデア自体が、空でない数値のコレクションを前提としており、それが私たちの議論の要点でした。

**Exercise 144.** `sum` と `how-many` は、`List-of-temperatures` からの入力を想定して設計されているにもかかわらず、`NEList-of-temperatures` に対しても機能するでしょうか？機能しないと思う場合は反例を示しなさい。機能すると思う場合はその理由を説明しなさい。

それにもかかわらず、この定義は、`sum` と `how-many` が `NEList-of-temperatures` のインスタンスを受け取るようになったため、これらをどのように設計するかという疑問も生じさせます。設計レシピの最初の3ステップの明らかな結果は以下の通りです。

```racket
;; NEList-of-temperatures -> Number
;; 与えられた温度の合計を計算する

  0)
```

この例は `average` の例を適応させたものです。ダミー定義は数値を生成しますが、与えられたテストに対しては誤った数値です。

4番目のステップは、`NEList-of-temperatures` に対する `sum` の設計において最も興味深い部分です。これまでの設計例はすべて、データ定義の形状に基づいて、空リストと空でない（つまり `cons` された）リストを区別するテンプレートを要求していました。しかし、`NEList-of-temperatures` にはこれが当てはまりません。ここでは、両方の節が `cons` されたリストを追加します。ただし、これら2つの節は、リストの `rest` フィールドにおいて異なります。特に、第1節は常に `rest` フィールドに `'()` を使用し、第2節は代わりに `cons` を使用します。したがって、第1のデータと第2のデータを区別するための適切な条件は、`rest` フィールドを取り出して `empty?` を使用することです。

```racket
;; NEList-of-temperatures -> Number
(define (sum ne-l)
  (cond
    [(empty? (rest ne-l)) ...]
    [else ...]))
```

ここでは、`else` が `(cons? (rest ne-l))` に置き換わっています。

次に、両方の節を検査し、それらが `ne-l` を構造体であるかのように扱うかどうかを判断する必要があります。もちろんその通りであり、`ne-l` に対して無条件に `rest` を使用していることがそれを示しています。言い換えれば、2つの節に適切なセレクタ式を追加します。

```racket
(define (sum ne-l)
  (cond
    [(empty? (rest ne-l)) (... (first ne-l) ...)]
    [else (... (first ne-l) ... (rest ne-l) ...)]))
```

読み進める前に、なぜ最初の節にはセレクタ式 `(rest ne-l)` が含まれていないのか説明しなさい。

テンプレート設計 of 最後の質問は、データ定義における自己参照に関するものです。ご存知のように、`NEList-of-temperatures` には自己参照が1つ含まれているため、`sum` のテンプレートは1つの再帰的な使用を要求します。

```racket
(define (sum ne-l)
  (cond
    [(empty? (rest ne-l)) (... (first ne-l) ...)]
    [else (... (first ne-l) ... (sum (rest ne-l)) ...)]))
```

具体的には、データ定義の対応する箇所が自己参照的であるため、2番目の節で `(rest ne-l)` に対して `sum` が呼び出されます。

5番目の設計ステップでは、私たちがすでにどれだけのものを手に入れているかを理解しましょう。最初の `cond` 節は、再帰的な関数呼び出しを持つ2番目の節よりも大幅にシンプルに見えるため、そちらから始めるべきです。この特定のケースでは、条件は `sum` が正確に1つの温度 `(first ne-l)` を持つリストに適用されることを示しています。明らかに、この1つの温度が、与えられたリスト内のすべての温度の合計です。

```racket
(define (sum ne-l)
  (cond
    [(empty? (rest ne-l)) (first ne-l)]
    [else (... (first ne-l) ... (sum (rest ne-l)) ...)]))
```

2番目の節は、リストが1つの温度と少なくとももう1つの温度から構成されていることを示しています。`(first ne-l)` は最初の位置を抽出し、`(rest ne-l)` は残りの位置を抽出します。さらに、テンプレートは `(sum (rest ne-l))` の結果を使用することを示唆しています。しかし、`sum` はあなたが定義している最中の関数であり、それが `(rest ne-l)` を**どのように**使用するかを知る由もありません。あなたが知っているのは目的ステートメントの内容だけです。すなわち、`sum` は与えられたリスト（ここでは `(rest ne-l)`）内のすべての温度を合計するということです。このステートメントが真であるならば、`(sum (rest ne-l))` は `ne-l` の1つを除くすべての数値を合計します。合計を得るために、関数は最初の温度を加算するだけです。

```racket
(define (sum ne-l)
  (cond
    [(empty? (rest ne-l)) (first ne-l)]
    [else (+ (first ne-l) (sum (rest ne-l)))]))
```

ここでこの関数のテストを実行すると、信頼の飛躍が正当化されることがわかります。実際、本書の範囲を超える理由により、この飛躍は**常に**正当化されます。これが、信頼の飛躍が設計レシピの不可欠な一部である理由です。

## 9.3 自然数 (Natural Numbers)

BSLプログラミング言語は、リストを受け取る多くの関数を提供しており、リストを生成する関数もいくつか提供しています。その中には `make-list` があります。これは、数値 `n` と他の任意の値 `v` を受け取り、`v` を `n` 個含むリストを生成します。以下にいくつかの例を示します。

```racket
> (make-list 2 "hello")
(cons "hello" (cons "hello" '()))
> (make-list 3 #true)
(cons #true (cons #true (cons #true '())))
> (make-list 0 17)
'()
```

要するに、この関数はアトミックなデータを受け取るにもかかわらず、任意に大きなデータを生成します。疑問に思うべきは、これがどのようにして可能なのかということです。

私たちの答えは、`make-list` の入力が単なる数値ではなく、特別な種類の数値であるということです。幼稚園で、これらの数値はオブジェクトを数えるために使用されるため、「数える数」と呼びました。コンピュータサイエンスでは、これらの数値は**自然数（natural numbers）**と呼ばれます。通常の数値とは異なり、自然数にはデータ定義が存在します。

```racket
;; An N is one of:
;; – 0
;; – (add1 N)
;; interpretation: 物の個数を数える数を表す
```

最初の節は `0` が自然数であることを述べています。これは、数えるべきオブジェクトがないことを示すために使用されます。2番目の節は、`n` が自然数であれば `n+1` も自然数であることを示しています。なぜなら、`add1` は与えられた任意の数値に `1` を加算する関数だからです。この2番目の節を `(+ n 1)` と書くこともできますが、`add1` の使用は、この加算が特別であることを示すためのものです。

この `add1` の使用が特別である理由は、それが通常の関数というよりも、構造体型の定義におけるコンストラクタのように機能することです。そのため、BSLには `add1` に対応する「セレクタ」である `sub1` 関数も用意されています。`0` と等しくない任意の自然数 `m` が与えられたとき、`sub1` を使用して `m` の構築に使われた数値を見つけることができます。言い換えれば、`add1` は `cons` のようなものであり、`sub1` は `first` や `rest` のようなものです。

この時点で、`0` と、`0` ではない自然数を区別する述語は何だろうかと思うかもしれません。リストと同様に、これには2つの述語があります。与えられた数値が `0` であるかを判定する `zero?` と、数値が `0` より大きいかを判定する `positive?` です。

これで、自然数に対する関数（例えば `make-list` など）を自分で設計する準備が整いました。データ定義はすでに利用可能ですので、ヘッダーの要素を追加しましょう。

```racket
;; N String -> List-of-strings
;; n 個の s のコピーからなるリストを作成する

(check-expect (copier 0 "hello") '())
(check-expect (copier 2 "hello") (cons "hello" (cons "hello" '())))

(define (copier n s)
  '())
```

テンプレートの開発が次のステップです。テンプレートに関する質問は、`copier` の本体が、`0` に対する節と正の数に対する節の2つの節を持つ `cond` 式であることを示唆しています。さらに、`0` はアトミックとみなされ、正の数は構造化された値とみなされるため、テンプレートは2番目の節にセレクタ式を必要とします。最後に重要な点として、`N` のデータ定義は2番目の節において自己参照的です。したがって、テンプレートは2番目の節の対応するセレクタ式に対して再帰的な適用を必要とします。

```racket
(define (copier n s)
  (cond
    [(zero? n) ...]
    [(positive? n)
     (... (copier (sub1 n) s) ...)]))
```

図59は、テンプレートから得られた `copier` 関数の完全な定義を示しています。このステップを注意深く再現してみましょう。いつものように、再帰呼び出しのない `cond` 節から始めます。ここでは、条件は（重要な）入力が `0` であることを示しており、それは関数が `0` 個の項目、すなわち**何もない**リストを生成しなければならないことを意味します。もちろん、2番目の例に取り組むことで、このケースはすでに明らかになっています。次に、もう一方の `cond` 節に目を向け、その式が何を計算するかを思い出します。

1. `(sub1 n)` は、`n`（`0` より大きいことがわかっています）の構築に使われた自然数を抽出します。
2. `(copier (sub1 n) s)` は、`copier` の目的ステートメントに従って、`(sub1 n)` 個の文字列 `s` のリストを生成します。

しかし、関数には `n` が与えられているため、`n` 個の文字列 `s` のリストを生成しなければなりません。文字列が1つ少ないリストが与えられた場合、関数は単に `(copier (sub1 n) s)` の結果の先頭に `s` を `cons` するだけでよいことは容易に理解できます。And それこそが2番目の節が指定していることです。

**図59: コピーのリストの作成 (Creating a list of copies)**
```racket
;; N String -> List-of-strings
;; n 個の s のコピーからなるリストを作成する

(check-expect (copier 0 "hello") '())
(check-expect (copier 2 "hello")
              (cons "hello" (cons "hello" '())))

(define (copier n s)
  (cond
    [(zero? n) '()]
    [(positive? n) (cons s (copier (sub1 n) s))]))
```

## 9.4 ロシア人形 (Russian Dolls)

Wikipediaはロシア人形（マトリョーシカ）を「サイズが小さくなっていく人形で、一方が他方の内部に入れ子状に配置された一組の人形」と定義し、次の写真で説明しています。

[Wikipediaのロシア人形の写真]

この写真では、鑑賞者がすべての人形を見ることができるように、人形が分解されています。
（※この問題は抽象的、あるいは不条理にさえ思えるかもしれません。ロシア人形を表現したい理由や、そのような表現で何をするのかが明確ではないからです。今はただ、この例に付き合ってみてください。）

ここで、そのようなロシア人形を BSL のデータで表現する問題を考えてみましょう。少しの想像力があれば、アーティストが任意の数の人形からなるロシア人形の入れ子を作成できることは容易に理解できます。結局のところ、与えられたロシア人形の周囲にいつでも別のレイヤーを巻き付けることが可能です。一方で、中には何もない頑丈な人形が深くに入っていることもご知でしょう。

ロシア人形の各レイヤーについて、私たちは多くの異なる事柄に関心を持つことができます。そのサイズ（ただし、これは入れ子の深さに関連しています）、その色、表面に描かれている絵、などです。ここでは、人形の**色**だけを取り上げ、それを文字列で表現します。それを前提とすると、ロシア人形の各レイヤーには、その色と、その内側にある人形という2つのプロパティがあることがわかります。2つのプロパティを持つ情報を表現するために、私たちは常に構造体型を定義します。

```racket
(define-struct layer [color doll])
```

And そして、データ定義を追加します。

```racket
;; An RD (short for Russian doll) is one of:
;; – String
;; – (make-layer String RD)
```

当然、このデータ定義の最初の節は、最も内側の人形、正確に言うとその色を表しています。2番目の節は、与えられたロシア人形の周囲にレイヤーを追加するためのものです。これを `layer` のインスタンスで表現します。これには明らかに人形の色と、もう1つのフィールド、すなわちこの人形のすぐ内側に入れ子になっている人形が含まれています。

この人形を見てみましょう。

[3つのレイヤーからなる人形の画像]

これは3つの人形で構成されています。赤が最も内側にあり、緑が真ん中にあり、黄が現在の最も外側のラッパーです。この人形を `RD` の要素として表現するには、どちらの端から始めても構いません。ここでは内側から外側へと進めます。赤い人形を `RD` として表現するのは簡単です。中には何もなく、赤色なので、文字列 `"red"` で十分です。2番目のレイヤーには、以下を使用します。

```racket
(make-layer "green" "red")
```

これは、緑色の（中空の）人形が赤い人形を含んでいることを示しています。最後に、最も外側を得るために、この最後の人形の周囲に別のレイヤーを巻き付けます。

```racket
(make-layer "yellow" (make-layer "green" "red"))
```

このプロセスにより、色付きのロシア人形の任意のセットからデータ表現へと移行する方法がよくわかるはずです。しかし、プログラマは逆のこともできなければならない、すなわち、データから具体的な情報へと移行できなければならないということを頭に留めておいてください。この精神で、次の `RD` の要素について略図を描きなさい。

```racket
(make-layer "pink" (make-layer "black" "white"))
```

これを BSL で試してみるのもよいでしょう。

データ定義が手に入り、実際の人形を表現する方法と `RD` の要素を人形として解釈する方法を理解したところで、`RD` を受け取る関数を設計する準備が整いました。具体的には、ロシア人形のセットに何個の人形が含まれているかを数える関数を設計しましょう。この文は優れた目的ステートメントであり、シグネチャも決定します。

```racket
;; RD -> Number
;; an-rd に含まれる人形の個数を数える
```

データ例については、`(make-layer "yellow" (make-layer "green" "red"))` から始めましょう。上記の画像から、赤、緑、黄の3つの人形があるため、期待される答えは `3` であることがわかります。この1つの例に取り組むだけでも、入力が最も内側の人形を表している場合、答えは `1` になることがわかります。

ステップ4ではテンプレートの開発が要求されます。このステップの標準的な質問を使用すると、次のテンプレートが生成されます。

```racket
;; RD -> Number
(define (depth an-rd)
  (cond
    [(string? an-rd) ...]
    [(layer? an-rd)
     (... (layer-color an-rd) ...
      ... (depth (layer-doll an-rd)) ...)]))
```

`cond` 節の数は、`RD` の定義における節の数によって決定されます。各節は、それがどのようなデータに関するものであるかを具体的に示しており、そこから `string?` および `layer?` 述語を使用するよう指示されます。文字列は複合データではありませんが、`layer` のインスタンスは2つの値を保持します。関数がこれらの値を必要とする場合は、セレクタ式 `(layer-color an-rd)` と `(layer-doll an-rd)` を使用します。最後に、データ定義の2番目の節には、`layer` 構造体の `doll` フィールドから定義自体への自己参照が含まれています。したがって、2番目のセレクタ式に対して再帰的な関数呼び出しが必要になります。

例とテンプレートが、関数定義をほぼ決定づけます。再帰のない `cond` 節については、答えは明らかに `1` です。再帰のある節については、テンプレートの式は以下の結果を計算します。

- `(layer-color an-rd)` は、現在のレイヤーの色を表す文字列を抽出します。
- `(layer-doll an-rd)` は、現在のレイヤーの内部に含まれる人形を抽出します。
- `(depth (layer-doll an-rd))` は、`depth` の目的ステートメントに従って、`(layer-doll an-rd)` に何個の人形が含まれているかを判定します。

この最後の数値は望ましい答えに極めて近いですが、完全に一致しているわけではありません。なぜなら、`an-rd` と `(layer-doll an-rd)` の差は1つのレイヤー、すなわち追加の1つの人形だからです。言い換えれば、関数は実際の答えを得るために、再帰的な結果に `1` を加算しなければなりません。

```racket
;; RD -> Number
;; an-rd に含まれる人形の個数を数える

(check-expect (depth "red") 1)
(check-expect (depth (make-layer "yellow" (make-layer "green" "red"))) 3)

(define (depth an-rd)
  (cond
    [(string? an-rd) 1]
    [else (+ (depth (layer-doll an-rd)) 1)]))
```

この関数定義が、2番目の節で `(layer-color an-rd)` を使用していないことに注目してください。繰り返しますが、テンプレートはデータ定義について私たちが知っているすべての情報を整理するためのスキーマですが、実際の定義においてそれらすべてのパーツを必要とするわけではないことがわかります。

最後に、例をテストに変換しましょう。

```racket
(check-expect (depth "red") 1)
(check-expect (depth (make-layer "yellow" (make-layer "green" "red"))) 3)
```

これらを DrRacket で実行すると、それらの評価が `depth` の定義のすべての部分に触れることがわかります。

**Exercise 154.** `colors` 関数を設計しなさい。これはロシア人形を受け取り、すべての色をカンマとスペースで区切った文字列を生成します。したがって、私たちの例では次のように生成されます。

`"yellow, green, red"`

## 9.5 リストと World (Lists and World)

リスト、および自己参照データ定義全般を使用すると、有限データを使用する場合よりも、はるかに多くの興味深い World プログラムを設計して実行することができます。「項目化と構造体」のスペースインベーダープログラムで、プレイヤーが戦車から好きなだけ多くのショットを発射できるバージョンを作成することを想像してみてください。この問題の単純化されたバージョンから始めましょう。
（※しばらく World プログラムを設計していない場合は、「World プログラムの設計」を読み返してください。）

> **Sample Problem** ショットの発射をシミュレートする World プログラムを設計しなさい。「プレイヤー」がスペースキーを押すたびに、プログラムはキャンバスの底にショットを追加します。これらのショットは、クロックの1チックごとに1ピクセルの割合で垂直に上昇します。

World プログラムの設計は、情報を定数と、絶えず変化する世界の状態の要素へと分離することから始まります。前者のために物理定数と画像定数を導入し、後者のために世界の状態の**データ表現**を開発する必要があります。サンプル問題は具体的な仕様については比較的曖昧ですが、ショットが垂直線に沿って描画される長方形の風景を明確に想定しています。明らかにショットの位置はクロックのチックごとに変化しますが、風景のサイズとショットの列の x 座標は同じままです。

```racket
(define HEIGHT 80) ; distance in terms of pixels
(define WIDTH 100)
(define XSHOTS (/ WIDTH 2))

;; graphical constants
(define BACKGROUND (empty-scene WIDTH HEIGHT))
(define SHOT (triangle 3 "solid" "red"))
```

問題文のどこにもこれらの具体的な選択を要求するものはありませんが、これらが簡単に変更できる（つまり、1つの定義を編集するだけで変更できる）限り、私たちの目標は達成されています。

「世界」の変化する側面について、問題文は2つ言及しています。第一に、スペースキーを押すとショットが追加されます。第二に、すべてのショットはクロックのチックごとに1ピクセルずつ真上に移動します。プレイヤーが何発のショットを「発射」するかを予測することはできないため、これらを表現するためにリストを使用します。

```racket
;; A Shot is a Number.
;; interpretation: ショットの y 座標を表す

;; A List-of-shots is one of:
;; – '()
;; – (cons Shot List-of-shots)
;; interpretation: 発射されたショットのコレクション
```

すべてのショットが同じ x 座標を持ち、この座標が終始同じままであることはすでにわかっています。さらに、すべてのショットは同じ見た目をしています。したがって、それらの y 座標だけが、互いに異なる唯一のプロパティです。したがって、各ショットを数値として表現するだけで十分です。

ショットの表現を `HEIGHT` 未満の数値の区間に制限することもできます。なぜなら、すべてのショットはキャンバスの底から発射され、その後上へと移動するため、それらの y 座標は連続的に減少することがわかっているからです。

この世界を表現するために、以下のようなデータ定義を使用することもできます。

```racket
;; A ShotWorld is List-of-numbers.
;; interpretation: このようなリストの各数値はショットの y 座標を表す
```

結局のところ、上記の2つの定義はすべての数値のリストを記述しています。私たちはすでに数値のリストの定義を持っており、`ShotWorld` という名前は、このデータクラスが何に関するものであるかを皆に伝えています。

定数を定義し、世界の状態を表すデータ表現を開発した後の主要なタスクは、どのイベントハンドラを採用するかを選択し、それらのシグネチャを特定の問題に適応させることです。この進行中の例ではクロックのチックとスペースキーについて言及されており、これは3つの関数のウィッシュリスト（wish list）に翻訳されます。

- 世界の状態を画像に変換する関数：
  ```racket
  ;; ShotWorld -> Image
  ;; w に含まれる各 y に対して、(XSHOTS, y) の位置にショットの画像を追加して BACKGROUND に描画する
  (define (to-image w)
    BACKGROUND)
  ```
  これは問題が視覚的な描画を要求しているためです。

- チックイベントを処理するための関数：
  ```racket
  ;; ShotWorld -> ShotWorld
  ;; w に含まれる各ショットを1ピクセル上に移動する
  (define (tock w)
    w)
  ```

- キーイベントを処理するための関数：
  ```racket
  ;; ShotWorld KeyEvent -> ShotWorld
  ;; プレイヤーがスペースキーを押した場合に世界にショットを追加する
  (define (keyh w ke)
    w)
  ```

初期のウィッシュリストに加えて、実際に世界を設定してハンドラをインストールする `main` 関数を定義する必要があることも忘れないでください。図61は、設計されるのではなく標準的なスキーマの修正として定義される、この1つの関数を含んでいます。

まず `to-image` の設計から始めましょう。シグネチャ、目的ステートメント、およびヘッダーが揃っているため、次に例が必要です。データ定義には2つの節があるため、少なくとも2つの例が必要です：`'()` と、例えば `(cons 9 '())` のような `cons` されたリストです。`'()` に対する期待される結果は明らかに `BACKGROUND` です。しかし、y 座標がある場合、関数は `XSHOTS` と指定された座標にショットの画像を配置しなければなりません。

```racket
(check-expect (to-image '()) BACKGROUND)
(check-expect (to-image (cons 9 '()))
              (place-image SHOT XSHOTS 9 BACKGROUND))
```

読み進める前に、2つの `Shot` のリストに `to-image` を適用する例に取り組んでみてください。そうすることで、関数がどのように動作するかが理解しやすくなります。

第4のステップは、データ定義をテンプレートに翻訳することです。

```racket
(define (to-image w)
  (cond
    [(empty? w) ...]
    [else
     (... (first w) ... (to-image (rest w)) ...)]))
```

リストのデータ定義に対するテンプレートは今や非常に馴染み深いものであるため、多くの説明は不要でしょう。疑念がある場合は、図52の質問を読み返し、自分でテンプレートを設計してください。

ここから関数を定義するのは簡単です。鍵となるのは、例とテンプレートを組み合わせ、図53の質問に答えることです。それらに従って、空のショットリストという基本ケースから始め、例から、期待される答えが `BACKGROUND` であることがわかります。次に、2番目の `cond` 節のテンプレート式が何を計算するかを定式化します。

- `(first w)` はリストから最初の座標を抽出します。
- `(rest w)` は残りの座標です。
- `(to-image (rest w))` は、`to-image` の目的ステートメントに従って、`(rest w)` 内の各ショットを背景画像に追加します。

言い換えれば、`(to-image (rest w))` はリストの残りの部分を画像として描画するため、ほとんどすべての作業を実行します。不足しているのは最初のショット `(first w)` です。これら2つの式に目的ステートメントを適用すると、2番目の `cond` 節の望ましい式が得られます。

```racket
(place-image SHOT XSHOTS (first w) (to-image (rest w)))
```

追加されるアイコンはショットの標準画像です。2つの座標は目的ステートメントで詳しく説明されています。そして、`place-image` の最後の引数は、リストの残りの部分から構築された画像です。

図61は、`to-image` の完全な関数定義と、プログラムの残りの部分を示しています。`tock` の設計は `to-image` の設計とまったく同じであり、自分で取り組んでみてください。ただし、`keyh` ハンドラのシグネチャは1つの興味深い疑問を投げかけます。これは、ハンドラが非自明なデータ定義を持つ2つの入力を受け取ることを指定しています。一方では、`ShotWorld` は自己参照的なデータ定義です。他方では、`KeyEvent` の定義は大きな列挙です。ここでは、2つの引数のどちらがテンプレートの開発を駆動すべきかを「推測」してもらい、このようなケースについては後で詳しく学習することにします。

**図61: リストに基づく World プログラム (A list-based world program)**
```racket
;; ShotWorld -> ShotWorld
(define (main w0)
  (big-bang w0
    [on-tick tock]
    [on-key keyh]
    [to-draw to-image]))

;; ShotWorld -> ShotWorld
;; 各ショットを1ピクセル上に移動する
(define (tock w)
  (cond
    [(empty? w) '()]
    [else (cons (sub1 (first w)) (tock (rest w)))]))

;; ShotWorld KeyEvent -> ShotWorld
;; スペースキーが押された場合に世界にショットを追加する
(define (keyh w ke)
  (if (key=? ke " ")
      (cons HEIGHT w)
      w))

;; ShotWorld -> Image
;; w に含まれる各ショット y を (XSHOTS, y) に配置して BACKGROUND に追加する
(define (to-image w)
  (cond
    [(empty? w) BACKGROUND]
    [else (place-image SHOT XSHOTS (first w)
                       (to-image (rest w)))]))
```

(define (keyh w ke)

## 9.6 リストと集合についての注意 (A Note on Lists and Sets)

本書は、BSLの値のコレクションとしての**集合（sets）**に関する、あなたの直感的な理解に依存しています。「データの宇宙」では、データ定義が BSL の値の集合の名前を導入すると明確に述べています。本書が集合に関して一貫して問いかける質問が1つあり、それは「ある要素が与えられた集合に含まれているかどうか」です。例えば、`4` は `Number` に含まれますが、`"four"` は含まれません。本書では、ある値が名前付きの集合のメンバーであるかどうかを確認するためにデータ定義を使用する方法や、いくつかのデータ定義を使用して集合のサンプル要素を生成する方法も示していますが、これら2つの手順はデータ定義に関するものであり、集合そのものに関するものではありません。

同時に、リストは値のコレクションを表します。そのため、リストと集合の違いは何であるのか、あるいはこれは不要な区別であるのかと疑問に思うかもしれません。そうであるなら、このセクションはあなたのためのものです。

現時点での集合とリストの主な違いは、前者はコードの設計手順を議論するために使用する概念であり、後者は選定したプログラミング言語である BSL の多くのデータ形式の1つであるということです。この2つのアイデアは、私たちの会話においてかなり異なるレベルに位置しています。しかし、データ定義が BSL の内部で実際の情報のデータ表現を導入すること、および集合が情報のコレクションであることを踏まえると、集合が BSL の内部でデータとしてどのように表現されるのかを自問するかもしれません。

ほとんどの本格的な言語は、リストと集合の両方のデータ表現を直接サポートしています。

リストは BSL において特別な地位を持っていますが、集合にはそれがありません。しかし同時に、集合はどこかリストに似ています。主な違いは、プログラムが通常どちらのデータ形式で使用する関数の種類にあります。BSL は、リストに関するいくつかの基本的な定数と関数（例えば `empty`、`empty?`、`cons`、`cons?`、`first`、`rest`）や、自分で定義できるいくつかの関数（例えば `member?`、`length`、`remove`、`reverse` など）を提供しています。以下は、自分で定義することはできますが、BSL には付属していない関数の例です。

```racket
;; List-of-string String -> N
;; los 内に s が出現する回数を判定する
(define (count los s)
  0)
```

ちょっと待ってください！この関数の設計を完成させなさい。

簡単で、おそらく素朴な方法で進め、集合は基本的にはリストであるとしましょう。And、さらに簡単にするために、このセクションでは数値のリストに焦点を当てます。ある数値が集合に含まれるかどうかが重要であると認めるならば、リストを使用して集合を表現する方法には**2つ**の異なるアプローチがあることが、ほぼ即座に明らかになります。

**図62: 集合の2つのデータ表現 (Two data representations for sets)**
| `Son.L` (左側の表現) | `Son.R` (右側の表現) |
| :--- | :--- |
| `;; A Son.L is one of:` | `;; A Son.R is one of:` |
| `;; – empty` | `;; – empty` |
| `;; – (cons Number Son.L)` | `;; – (cons Number Son.R)` |
| | `;; Constraint: If s is a Son.R,` |
| `;; Son is used when it applies` | `;; no number occurs twice in s` |
| `;; to Son.L and Son.R` | |

図62は、これら2つのデータ定義を示しています。どちらも基本的には、集合が数値のリストとして表現されることを述べています。違いは、右側の定義には、リスト上で同じ数値が2回以上出現してはならないという制約（Constraint）が伴っている点です。結局のところ、集合に関して私たちが問いかける主要な質問は「ある数値が集合に含まれているかどうか」であり、それが集合の中に1回、2回、あるいは3回含まれているかどうかは違いを生じさせないからです。

どちらの定義を選択するかにかかわらず、2つの重要概念をすでに定義できます。

```racket
;; Son
(define es '())

;; Number Son -> Boolean
;; x は s に含まれるか
(define (in? x s)
  (member? x s))
```

最初のものは**空集合 (empty set)**であり、どちらのケースでも空リストによって表現されます。2番目のものはメンバーシップテスト（所属判定）です。

より大きな集合を構築する1つの方法は、`cons` と上記の定義を使用することです。たとえば、`1`、`2`、`3` を含む集合の表現を構築したいとします。そのような表現の1つは次の通りです。

```racket
(cons 1 (cons 2 (cons 3 '())))
```

And それは両方のデータ表現で機能します。But、

```racket
(cons 2 (cons 1 (cons 3 '())))
```

は本当に同じ集合の表現ではないのでしょうか？あるいは、

```racket
(cons 1 (cons 2 (cons 1 (cons 3 '()))))
```

はどうでしょうか？

主要な関心が「数値が集合に含まれているかどうか」である限り、その答えは肯定的でなければなりません。それでも、`cons` の順序は重要になり得ませんが、右側のデータ定義における制約は、最後のリストを `Son.R` とすることを排除します。なぜなら、それが `1` を2回含んでいるからです。

**図63: 集合の2つのデータ表現のための関数 (Functions for the two data representations of sets)**
| `Son.L` のための関数とテスト | `Son.R` のための関数とテスト |
| :--- | :--- |
| `;; Son.L` | `;; Son.R` |
| `(define s1.L (cons 1 (cons 1 '())))` | `(define s1.R (cons 1 '()))` |
| | |
| `;; Number Son.L -> Son.L` | `;; Number Son.R -> Son.R` |
| `;; s から x を削除する` | `;; s から x を削除する` |
| `(check-expect (set-.L 1 s1.L) es)` | `(check-expect (set-.R 1 s1.R) es)` |
| | |
| `(define (set-.L x s)` | `(define (set-.R x s)` |
| `  (remove-all x s))` | `  (remove x s))` |

2つのデータ定義の違いは、関数を設計する際に現れます。集合から数値を削除する関数が必要だとします。以下は、両方の表現に適用されるウィッシュリストの項目です。

```racket
;; Number Son -> Son
;; s から x を差し引く (subtracts)
(define (set- x s)
  s)
```

目的ステートメントで "subtract"（差し引く）という言葉が使用されているのは、論理学者や数学者が集合を扱う際にこの言葉を使用するからです。

図63はその結果を示しています。2つの列は2つの点で異なります。

1. 左側のテストは `1` を2回含むリストを使用していますが、右側のテストは同じ集合を1つの `cons` で表現しています。
2. これらの違いにより、左側の `set-` は `remove-all` を使用しなければなりませんが、右側のものは `remove` で済みます。

ちょっと待ってください！コードを DrRacket の定義領域にコピーし、テストが合格することを確認しなさい。その後、読み進めながらコードを使って実験しなさい。

図63の好ましくない側面は、テストが期待される結果としてプレーンなリストである `es` を使用していることです。この問題は一見すると些細なことのように思えるかもしれません。しかし、次の例を考えてみましょう。

```racket
(set- 1 set123)
```

ここで `set123` は、`1`、`2`、`3` を含む集合を、次の2つの方法のいずれかで表現しています。

```racket
(define set123-version1
  (cons 1 (cons 2 (cons 3 '()))))

(define set123-version2
  (cons 1 (cons 3 (cons 2 '()))))
```

Regardless of どちらの表現を選択するかにかかわらず、`(set- 1 set123)` は次の2つのリストのいずれかに評価されます。

```racket
(define set23-version1
  (cons 2 (cons 3 '())))

(define set23-version2
  (cons 3 (cons 2 '())))
```

But 私たちはこれら2つのうちどちらを `set-` が生成するかを予測することはできません。

2つの代替案という単純なケースについては、次のように `check-member-of` テスト機能を使用することが可能です。

```racket
(check-member-of (set-.v1 1 set123.v1)
                 set23-version1
                 set23-version2)
```

もし期待される集合が3つの要素を含んでいる場合、左側のデータ定義が許容する重複のある表現を除外したとしても、6つの可能なバリエーションが存在します。

この問題を解決するには、2つのアイデアを組み合わせる必要があります。第一に、`set-` が本質的に「与えられた要素が結果に出現しないことを保証する」ことに関するものであることを思い出してください。これは、例をテストに変換する私たちの方法では伝わらないアイデアです。第二に、BSL の `check-satisfied` テスト機能を使用すれば、このアイデアを正確に述べることができます。

「インターメッツォ1：Beginning Student Language」で `check-satisfied` について簡単に言及されていますが、簡単に言うと、この機能はある式が特定のプロパティを満たしているかどうかを判定します。プロパティとは、値から `Boolean` への関数です。私たちの具体的なケースでは、`1` が集合のメンバーではないことを正確に述べたいと考えます。

```racket
;; Son -> Boolean
;; 1 が s のメンバーでなければ #true、そうでなければ #false
(define (not-member-1? s)
  (not (in? 1 s)))
```

`not-member-1?` を使用すると、テストケースを次のように定式化できます。

```racket
(check-satisfied (set- 1 set123) not-member-1?)
```

and このバリアントは関数が何を達成すべきかを明確に述べています。さらに良いことに、この定式化は入力または出力の集合がどのように表現されているかにまったく依存しません。

要約すると、リストと集合はどちらも値のコレクションに関するものであるという点で関連していますが、大きく異なります。

| 特性 (property) | リスト (lists) | 集合 (sets) |
| :--- | :--- | :--- |
| メンバーシップ (membership) | 多くのうちの1つ | 決定的 (critical) |
| 順序 (ordering) | 決定的 (critical) | 無関係 (irrelevant) |
| 出現回数 (# of occurrences) | 意味を持つ (sensible) | 無関係 (irrelevant) |
| サイズ (size) | 有限だが任意 | 有限または無限 |

The 最後の行は、明白なことではありますが、新しいアイデアを提示しています。本書で言及されている集合の多く（例えば `Number`、`String`、さらには `List-of-strings`）は無限に大きいです。対照的に、リストは任意の数の項目を含むことができますが、**常に**有限です。

要約すると、このセクションでは、集合とリストの決定的な違いと、有限リストを用いて有限集合を2つの異なる方法で表現する方法について説明しました。BSL は無限集合を表現するのに十分な表現力を持っていません。練習問題299では、無限集合にも対処できる、まったく異なる集合の表現を導入します。ただし、実際のプログラミング言語がどのように集合を表現するかという問題は、本書の範囲を超えています。
